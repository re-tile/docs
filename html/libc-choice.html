Choosing Which Libc to Use

This application note discusses considerations for application developers in choosing whether to use glibc or newlib for their application.

Why to Choose a Particular Libc

There are two C libraries shipped with the MDE.

  • glibc is the standard fully Linux-conforming library, that also includes other libraries (libcrypt, libutil, librt, libresolv) and full support for all Linux syscalls and library functions. It is LGPL licensed, so our customers have to either link to it dynamically, or else provide a linkable object file with their application so their customers could relink to a different libc.
  • newlib is a much more restricted C library that has no GPL or LGPL components (see the Tilera MDE License note for more licensing info). However, it does not currently support the full range of syscall or library functions, and its headers are noticeably different from standard Linux headers, making for more difficulty in porting Linux applications to it. However, customers can link with it statically and are under no obligations to release any source code with it.

The Tilera-specific code (e.g. our optimized memcpy and the like) are present in both libraries, and licensed in each case the same way as the containing library.

How to Choose a Libc

By default, glibc is the library that is used by the compiler, and its headers are installed in tile/usr/include. However, customers can pass the "-mnewlib" argument to tile-gcc to switch to using the newlib headers and linking against the newlib libraries instead.
