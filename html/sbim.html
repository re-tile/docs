Description

sbim manages the contents of a bootable SPI ROM (SROM). It allows installation and management of multiple bootable images within one ROM. The bootable images may be the Tilera standalone boot loader, mboot; one of the standard Linux images distributed by Tilera; or a custom bootable image constructed to meet specific customer requirements.

Once an appropriate bootable image has been installed in an SROM, a Tile Processor may then be booted from it. (Systems must generally be reconfigured via DIP switch settings to boot from their SPI ROMs; for instance see the TILExpress-64 Card User's Guide for details.)

Generally Tilera recommends using sbim on the target Tile Processor. This provides the most reliable operation since sbim can automatically sense SROM characteristics, and can modify only the necessary areas of the SROM for each operation. However, this is not always possible. For instance, you might want to prepare a simulated SROM to use with tile-sim(1), or you might need to prepare a prototype image which would be transferred to many target SROMs as part of a manufacturing process. In this case, the tile-sbim command, which runs on the host system, can be used.

Actions

-o , --info

The --info action prints information on the currently installed SROM booter and bootable images within an SROM. This action is the default if no other action is specified. The output produced varies with configuration, but will look something like this: 

SROM booter: Present, 4528 bytes.
   Time installed: Fri May 16 05:46:26 2008
Image in slot 0: Present, generation 20, 3472152 bytes.
   Time installed: Fri May 16 06:04:01 2008
   Comment: Latest software version
Image in slot 1: Present, generation 19, 3472148 bytes.
   Time installed: Fri May 16 06:03:18 2008
   Comment: Backup image

-i , --install input-image-file

The --install action installs a bootable image within the SROM, potentially replacing a preexisting image. An image is built with the tile-mkboot(1) command, and customarily has the extension .bootrom. Since space within the ROM is normally at a premium, it is advisable to give the --compress option to tile-mkboot.

Unless the --oldest option is specified, the newly added image will be given a generation number larger than those assigned to any existing image. In most cases this will mean that the new image will be the one which will be loaded at the next boot (see Boot Policy). If any image slots are empty, one of them will be overwritten with the new image. Otherwise, by default, the existing unlocked image with the lowest generation number will be overwritten. The image to be overwritten may be explicitly specified with the --location option; this image must be unlocked.

The time required to install a bootable image varies with the size of the image and the previous contents of the SROM, but is generally less than a minute.

-e , --extract output-image-file

The --extract action reads an image which is currently installed within the SROM, and writes it to an output file. Assuming no corruption has occurred in the SROM, the output file will be identical to the input file originally used with the --install option. By default, the exiting image with the highest generation number will be extracted. The image to be extracted may be explicitly specified with the --location option.

-n , --invalidate image-location

The --invalidate action marks a specified image as invalid. Once an image has been invalidated, it cannot be loaded at boot, and it may no longer be extracted. The invalidation operation is not reversible by sbim. Note, however, that the image is not completely erased, merely marked as invalid, so invalidation is not an absolute protection against later retrieval of image data. See Locations for the format of image-location. The image to be invalidated must be unlocked.

If there are no valid images, the system will not be bootable from the SROM. Because of this, if an attempt is made to invalidate the last valid image within an SROM, sbim will first issue a warning on standard output, and prompt for confirmation. This prompt may be avoided by specifying the --yes option.

-p , --promote image-location

The --promote action marks a specified image as being the most recently added image (the one with the highest generation number). This will in most cases make it the image which will be loaded at the next boot. See Locations for the format of image-location.

-L , --lock image-location

The --lock action marks a specified image as being locked. Locked images cannot be invalidated, and a new image cannot be installed in a locked image's slot. Images are initially unlocked. See Locations for the format of image-location.

-U , --unlock image-location

The --unlock action marks a specified image as being unlocked. Once unlocked, an image can be invalidated, and a new image can be installed in its slot. See Locations for the format of image-location.

-d , --dump

The --dump action dumps out a list of key-value pairs which describe the SROM, as well as the currently installed SROM booter and bootable images. This may be useful for certain post-processing operations. The output produced varies with configuration, but will look something like this: 

srom_valid: 1
srom_RomHeader: 0
srom_RomSegmentHeader: 0x30020
srom_sector_size: 65536
srom_page_size: 256
srom_srom_size: 7995392
srom_num_images: 2
srom_offset: 132
srom_booter_words: 1140
srom_booter_crc: 0x866445f3
srom_comment: This is the booter.
srom_timestamp: Fri May 16 15:33:21 2008
srom_header_crc: 0x4e3785be
image0_valid: 0
image1_valid: 1
image1_generation: 1
image1_offset: 120
image1_l0_boot_words: 691
image1_l0_boot_crc: 0x6f3efe9c
image1_total_words: 345687
image1_total_crc: 0x63372645
image1_comment: howdy
image1_timestamp: Fri May 16 15:33:21 2008
image1_header_crc: 0x80cfb8be

--install-booter input-booter-file

The --install-booter action installs an SROM header and SROM booter within the SROM, replacing any preexisting booter or prebooter. Typically the input-booter-file specified is the standard SROM booter in ${TILERA_ROOT}/tile/boot/sromboot.bin, whose behavior is described in this manual page. In the future there may be additional booters which provide different behaviors.

Since there is no redundancy for the SROM booter, a power failure or other system fault during its installation could result in a system which was not bootable from the SROM. Because of this, when --install-booter is used on an SROM which already contains an SROM booter or prebooter, sbim will first issue a warning on standard output, and prompt for confirmation. This prompt may be avoided by specifying the --yes option.

--install-prebooter input-prebooter-file

The --install-prebooter action installs an SROM header and prebooter within the SROM, replacing any preexisting booter or prebooter. One example of a prebooter is the PCIe prebooter, supplied in ${TILERA_ROOT}/tile/boot/pci_preboot.bin. This file enables more robust PCIe link initialization in some system configurations, and is not necessary for most users.

Since there is no redundancy for the SROM booter, a power failure or other system fault during its installation could result in a system which was not bootable from the SROM. Because of this, when --install-prebooter is used on an SROM which already contains an SROM booter or prebooter, sbim will first issue a warning on standard output, and prompt for confirmation. This prompt may be avoided by specifying the --yes option.

--extract-booter output-booter-file

The --extract-booter action reads the SROM booter or prebooter currently installed within the SROM, and writes it to an output file. Assuming no corruption has occurred in the SROM, the output file will be identical to the input file originally used with the --install-booter or --install-prebooter option.

--invalidate-booter

The --invalidate--booter action marks the SROM booter or prebooter as invalid. Once the booter has been invalidated, it may no longer be extracted. The invalidation operation is not reversible by sbim. Note, however, that the booter is not completely erased, merely marked as invalid, so invalidation is not an absolute protection against later retrieval of booter data.

Since there is no redundancy for the SROM booter, invalidating it will result in a system which was not bootable from the SROM. Because of this, when --invalidate-booter is used, sbim will first issue a warning on standard output, and prompt for confirmation. This prompt may be avoided by specifying the --yes option.

--sanitize

The --sanitize action writes a constant data pattern to the SROM, destroying any data previously written there. This may be useful if sensitive data has been written to a system which must be used for a different purpose, or returned for repair. In addition to erasing /dev/srom/bootimage, which is used to store the SROM booter and any bootable images, this action also erases /dev/srom/bootparam, which is used by the Tilera standalone boot loader, mboot, and /dev/srom/userparam, which is made available for use by applications programs. Note that these file names may not be modified; the --file option may not be specified with the --sanitize action.

After the sanitize operation has completed, the target system will no longer be bootable from the SROM. Because of this, when --sanitize is used, sbim will first issue a warning on standard output, and prompt for confirmation. This prompt may be avoided by specifying the --yes option.

Options

-c , --comment comment-string

The --comment option may be specified with the --install, --install-booter, and --install-prebooter actions. The specified string is saved in the SROM with the booter, or the corresponding image, and may be retrieved via the --info action. The comment is truncated to no more than 80 bytes in length.

-f , --file srom-file

The --file option specifies the SROM file which will be acted upon by sbim. If this option is not specified, sbim uses /dev/srom/bootimage as the default; this is the recommended mode of operation, since in that case the SROM parameters are automatically determined based on the actual SROM chip in the system. tile-sbim does not have a default value for this option, so when running on the host, the SROM file must always be specified explicitly.

Note that tile-sbim will not automatically expand a file specified with the --file option. When creating a new image in a regular file, you may wish to use the dd and tr commands in conjuction with /dev/zero to create a file with the proper length and with all of its bits set to 1. (Initializing to all 1's avoids unnecessary page write operations if the file is then written to a blank SROM.) For instance, to create an empty 8 megabyte SROM image: 

$ dd if=/dev/zero bs=1M count=8 | tr '\0' '\377' >my.srom

-l , --location image-location

The --location option is used with the --install and --extract actions to specify a target image other than the default. See Locations for the format of image-location.

-m , --num-images num-images

The --num-images option is used with the --install-booter action to modify the number of images supported by the SROM. By default, 2 images are supported. Once the number of images has been set, it cannot be changed without invalidating and re-installing the SROM booter. The available SROM space is split more-or-less evenly among the supported images, so specifying fewer images will allow larger images to be contained within the SROM, potentially at the cost of decreased robustness.

It is legal to specify --num-images 0; in this case, all of the available SROM space may be used by the SROM booter, which may be a .bootrom file. This configuration sacrifices the robustness benefits normally achieved by using a separate SROM booter and boot images, and is not recommended in most cases. When --install-prebooter is used, the number of images is always set to zero; in this case, the boot image is assumed to be coming from another interface, such as PCIe.

-g , --page-size page-size

The --page-size option is used with the --install-booter action to set the page size supported by the SROM, in bytes. This option is normally only used on the host with tile-sbim, since in that case the page size cannot be automatically determined. Once the page size has been set, it cannot be changed without invalidating and re-installing the SROM booter.

-s , --sector-size sector-size

The --sector-size option is used with the --install-booter action to set the sector size supported by the SROM, in bytes. This option is normally only used on the host with tile-sbim, since in that case the sector size cannot be automatically determined. Once the sector size has been set, it cannot be changed without invalidating and re-installing the SROM booter.

-Z , --sectors num-sectors

The --num-sectors option is used with the --install-booter action to set the number of sectors supported by the SROM. If not specified, the number of sectors is set automatically based on the size of the SROM file and its sector size. Once the number of sectors has been set, it cannot be changed without invalidating and re-installing the SROM booter. You cannot specify both the --num-sectors and --srom-size options.

It is legal to specify a number of sectors that is smaller, but not larger, than the actual number of sectors present in the SROM; this may be necessary when creating a boot image within a fully-sized SROM file on a host (see SROM Boot Image Region Size for details). Using a smaller number of sectors may also be useful to verify that a particular set of images will fit in a smaller SROM.

-z , --srom-size srom-size

The --srom-size option is used with the --install-booter action to set the total size of the SROM, in bytes. If not specified, the size is set automatically based on the size of the SROM file. Once the SROM size has been set, it cannot be changed without invalidating and re-installing the SROM booter. You cannot specify both the --num-sectors and --srom-size options.

It is legal to specify a size that is smaller, but not larger, than the actual size of the SROM; this may be necessary when creating a boot image within a fully-sized SROM file on a host (see SROM Boot Image Region Size for details). Using a smaller size may also be useful to verify that a particular set of images will fit in a smaller SROM.

--pcie0-bar1-size bar-size , --pcie1-bar1-size bar-size

The --pcie0-bar1-size and --pcie1-bar1-size options may be specified with the --install-booter and --install-prebooter actions, on TILEPro processors only. When used, data is added to the SROM which modifies registers in the specified PCI Express shim. The register settings configure the base address register for memory region number 1 to request the specified amount of address space.

-V , --verbose

The --verbose option causes some actions to display additional output.

-v , --verify

The --verify option may be used with the --info, --install, and --install-booter actions. When used with --info, it causes the checksums of the installed images to be verified; the result of the verification is displayed along with the normal --info output. When used with --install or --install-booter, it causes the data written to the SROM to be read back and compared against the source; if errors are detected, they are reported.

--old-format

Boot images and SROM booters built by MDE 2.0 and later contain additional embedded error-checking information which allows sbim to verify that such files have not been corrupted. Attempting to install a file built with a previous MDE version via --install or --install-booter will result in an error message. The --old-format option suppresses this message and allows such files to be installed. This option is only available on TILEPro processors.

--yes

The --yes option suppresses confirmation requests which might be made by the --invalidate, --install-booter, --invalidate-booter, and --sanitize actions.

--oldest

When used with the --install action, the --oldest option causes the newly installed image to be given a generation number smaller than those assigned to any existing image. Generation numbers are non-negative, so an installation action with this option will fail if an image with generation number zero exists and is not being replaced.

--pattern sanitize-pattern

The --pattern option may be specified with the --sanitize action. The argument is a 64-bit hexadecimal value, which will be used as the data pattern written to the SROM during the sanitization operation. If this option is omitted, a 64-bit word of all 1's will be used.

Numeric Values

The numeric arguments supplied to the --sector-size, --page-size, --srom-size, --sectors, and --num-images options may be given in kilo- (1024) or mega- (1048576) units by appending a 'k' or 'm', respectively, to the value.

SROM Boot Image Region Size

While the SROM is primarily used to hold boot images, some parts of it are dedicated to other uses (for example, booter parameters and application parameters). Currently the hypervisor reserves 6 sectors at the end of the SROM for these other uses. Thus, /dev/srom/bootimage, which is the region of the SROM used by sbim, will be 6 sectors shorter than the entire SROM.

When using sbim on a Tile Processor, this is not a concern; the size of the boot image region is automatically taken into account when initializing it and installing mages. However, when using tile-sbim on a host system, this is not true. In that case, you need to do one of two things:

  1. Set the size of the disk file containing the SROM image to the size of the boot image region on the target Tile Processor, rather than the size of the entire SROM.

  2. Use the --num-sectors or --srom-size options to set the proper size of the boot image region within a larger disk file.

The best way to ascertain the size of the boot image partition on a particular host is to look at the SROM device properties there: 

$ cat /sys/devices/platform/srom/0/total_size
15204352

However, if this is not possible, using a value of 6 reserved sectors is a reasonable approach. Failure to set the proper size for the boot image region when creating an SROM image on the host may lead to later failures when using sbim to update images on the Tile Processor, or may even lead to image corruption.

Locations

Once a booter has been installed in an SROM, it consists of a number of image slots, each of which can hold one bootable image. Image slots are numbered from zero to one less than the total number of images. Commands which act on a particular image slot have a location as a required or optional argument; a location specifies a slot number, either directly, or indirectly. The valid formats for a location are as follows:

@n

A location argument of the format @n specifies an absolute image slot number.

0

A location argument of the format 0 specifies the image with the largest generation number.

-n

A location argument of the format -n specifies the nth most recently installed image. Thus, -1 specifies the image installed just before the most recently installed one; -2 specifies the image installed just before that; and so on.

Boot Policy

When a TILExpress card is booted from its SROM, the first code executed is the SROM booter. The SROM booter is responsible for verifying the remaining content of the SROM, choosing one of the set of valid bootable images, and then actually loading that image into the processor to complete the boot process.

The policy used by the default SROM booter provided with the MDE is to pick the image within the SROM which was most recently installed or promoted. This determination is done by examining each image's generation number; the image with the largest number is the most recent. An image with a lower generation number may be used if the one with the largest number is found to be corrupt.

Examples

Online Example

In order to create a new boot image my.bootrom using a customized hypervisor configuration file my.hvc, and then install it in a previously-unused SROM on a TILExpress card, one could proceed as follows. (Note that the image is installed twice; for robustness, it is recommended that you have at least two valid bootable images in your SROM whenever possible.) 

$ ${TILERA_ROOT}/bin/tile-mkboot \
  --compress --output my.bootrom --hvc my.hvc \
  vmlinux=${TILERA_ROOT}/tile/boot/vmlinux
$ ${TILERA_ROOT}/bin/tile-monitor --pci \
  --upload-tile /boot/sromboot.bin --upload my.bootrom /my.bootrom
Command: run /usr/sbin/sbim --install-booter /boot/sromboot.bin --yes
Command: run /usr/sbin/sbim --install /my.bootrom --comment "First bootrom"
Command: run /usr/sbin/sbim --install /my.bootrom
Command: run /usr/sbin/sbim --info
SROM booter: Present, 4544 bytes.
   Time installed: Wed May 21 16:26:30 2008
Image in slot 0: Present, generation 2, 3472872 bytes.
   Time installed: Wed May 21 16:26:46 2008
Image in slot 1: Present, generation 1, 3472872 bytes.
   Time installed: Wed May 21 16:26:39 2008
   Comment: First bootrom

Offline Example

In some cases, it may not be possible to run sbim on the system you want to configure; for instance, you may need to create a master SROM image to be used as part of a manufacturing process. The following example demonstrates how that might be done with tile-sbim. We assume we already have a boot image in my.bootrom, prepared as in the example above. In this example, the SROM is 8 megabytes in size, its page size is 256 bytes, and its sector size is 65536 bytes; these values are normally taken from the hardware data sheet for the particular SROM chip being used. To get the correct size of the /bootimage partition, we reduce the number of sectors from the default 128 as described in SROM Boot Image Region Size. 

$ dd if=/dev/zero bs=1M count=8 | tr '\0' '\377' >my.srom
$ ${TILERA_ROOT}/bin/tile-sbim --file my.srom \
  --page-size 256 --sector-size 64k --num-sectors 122 --yes \
  --install-booter ${TILERA_ROOT}/tile/boot/sromboot.bin
$ ${TILERA_ROOT}/bin/tile-sbim --file my.srom \
  --install my.bootrom
$ ${TILERA_ROOT}/bin/tile-sbim --file my.srom \
  --install my.bootrom
$ ${TILERA_ROOT}/bin/tile-sbim --file my.srom \
  --info
SROM booter: Present, 4544 bytes.
   Time installed: Wed May 21 16:26:30 2008
Image in slot 0: Present, generation 2, 3472872 bytes.
   Time installed: Wed May 21 16:26:46 2008
Image in slot 1: Present, generation 1, 3472872 bytes.
   Time installed: Wed May 21 16:26:39 2008

At this point, the file my.srom can be used to initialize an SROM chip which, once connected to a Tile Processor, will enable it to boot.
