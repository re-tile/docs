Profiling applications from the command line with OProfile

Introduction

For profiling applications, the Tilera MDE provides a customized version of the OProfile system profiler, a utility for collecting whole system performance information. Tilera's OProfile implementation is very efficient and non-intrusive; it uses the Tile architecture's hardware performance counters to collect performance data.

You can find documentation on the standard OProfile commands and tools at the OProfile website. These are supported by the Tilera MDE versions of these commands and tools. Note that in keeping with standard naming conventions for cross-platform tools, the Tilera versions of the OProfile commands have a tile- prefix. For example the Tilera MDE version of the opreport command is named tile-opreport.

The Tilera MDE also supports generating an XML file from the raw OProfile performance data and viewing this profile.xml file in the Profile View of the Tilera IDE, tile-eclipse. This can be more convenient than using the command-line OProfile tools, since the Profile View provides filtering and selection options, as well as providing a Tilera-centric view of the profiling data.

This document's focus is on generation and viewing of OProfile data from the command line. For more information on the IDE's Profile View, see the online documentation within tile-eclipse, or the MDE User's Guide.

The OProfile Profiler

OProfile captures performance information from all running applications, as well as from the Linux OS. Any application can be profiled as-is; no special rebuild or libraries are required. Profiling may be started and stopped at any time, allowing profiling of long-running applications without having to stop and restart them. OProfile takes a "global view" of a program's execution, and hence captured profiling data may include information about background or OS processes unrelated to your application. However it is easy to filter these processes out of the final results, so it is generally not worth trying to exclude them from a profiling run.

OProfile's reporting tools can report time spent in each single application, in each invocation of an application (that is, per-process), or in any given thread or set of threads of a particular process. It can show flat profiles in the style of prof or call graph profiles in the style of gprof. It can also be used to produce profile data readable by the Tilera MDE's implementation of gprof, tile-gprof.

OProfile can also produce source code annotated with profiling information, allowing the user to find out which individual source lines need attention. It can produce annotated assembly listings to provide the most low level and fine grained possible information.

NOTE: OProfile collects data via kernel- and user-space processes within Linux, so it has no direct visibility to the Hypervisor. Any performance counter events triggered while the Hypervisor is running will be attributed to the code that executes when the Hypervisor code returns. For instance a Linux system call that invokes the Hypervisor, or a user-level library such as NetIO that directly invokes the Hypervisor, or any code that was executing while a Hypervisor TLB miss handler was invoked by the hardware, will be attributed to the code that is executed when control returns from the Hypervisor.

Hardware Events

OProfile gathers its profiling data by monitoring hardware profiling events. The Tilera hardware supports a number of hardware event counters, each of which can be set to track any one of a number of event types. When an event counter reaches a specified profiling interval (number of events seen) it triggers an interrupt. This is handled by OProfile, which notes the current program location and optionally the calling function(s) on the current stack. This is known as a "sample" in OProfile parlance, and OProfile keeps track of the number of samples seen for each of the hardware events it has been set to monitor.

One event type can be specified for each event counter supported by the hardware, in the form --event=NAME:INTERVAL, where NAME is the name of the event (see below), and INTERVAL is the sampling interval, that is, the number of times the event occurs before each sampling interrupt is taken.

The number of available hardware event counters depends on the chip architecture. For example on the Gx chip architecture a maximum of four events can be profiled simultaneously.

A few of the most useful profiling events are:

  • ONE -- occurs once every cycle, and hence can be used to measure the relative amount of time spent in various areas of your code.
  • INSTRUCTION_BUNDLE -- occurs once per instruction bundle processed.
  • LOAD_STALL -- occurs whenever there is a data cache stall.
  • INSTRUCTION_STALL -- occurs whenever there is an instruction stall.

There is also a ZERO event, which never occurs. This can be used to disable an event counter that is not needed for a given profiling run.

For a list of the available hardware counter event names, you can run the command tile-ophelp.

It's important to note that OProfile counts samples, not unique events. This has a couple of important consequences:

  1. OProfile sampling is statistical, not exact. It does not count, for example, how many times a certain function has been called. Instead it counts how often your code happened to be in that function when a sample was taken. This means the sampling interval can be important to getting good profiling data: if the sampling interval is too high, or a function is invoked too rarely or too quickly, you may not see any samples taken in that function at all.
  2. OProfile sampling is handled via interrupts, which can influence the timing and behavior of your code. In particular, if you set the sampling interval for an event too low (or the event type is one that normally occurs very often), then the large number of resulting interrupts can significantly affect the overall performance of your code, and can in extreme cases cause the OS to thrash or hang.

In short, choosing a good sampling interval, neither too high nor too low, is important. In general, you'll need to experiment a little with profiling your code to find event intervals appropriate for the events you want to track.

The appropriate sampling interval to use will differ from event to event, and can also depend on your application's behavior. For example, an event that ticks slowly needs a shorter interval, one that ticks close to every cycle needs a longer interval, and one that depends on application behavior (memory cache misses, etc.) may require either large or small values based on what the application is doing.

A reasonable starting point is an event interval that results in around 100 samples per second, and you can adjust the interval up or down as needed. For example, on a Tile processor running at 1200 MHz, a ONE event interval of 12000000 (12 million) gives a sampling rate of 100 samples per second. (If your processor is running at a different rate, you may want to adjust the interval accordingly, but your starting point for interval will still be somewere in the same order of magnitude.)

Don't make the sampling interval too small, i.e. less than 10,000-100,000 depending on the counter, or you may hang or crash the Linux OS due to excessive profiling interrupt calls. (The OS will attempt to detect this condition and disable profiling, but it's better simply to not cause it in the first place.)

The OProfile Daemon

OProfile collects and processes profiling events via a profiler daemon, which is configured when it starts up, and retains that configuration until it is shut down (and/or restarted).

Hence, commands that specify OProfile properties, such as profile-events and profile-flags (discussed below) take effect only when the OProfile daemon is started (or re-started). If invoked while the daemon is running, they only take effect the next time the daemon is started.

This is particularly important on "persistent" platforms such as TILEmpower, etc., where the OProfile daemon may already be running as a result of someone else doing a prior profiling run on the same platform.

You can use the command profile-shutdown to completely shutdown the OProfile daemon, and profile-start to restart it.

In general, it's best to get in the habit of using profile-reset at the start of a profiling run, to clean out any old state, and profile-shutdown at the end of the profiling run, to be sure you're starting and stopping the OProfile daemon in a way that allows you to easily change profiling options for each run.

Interactive Profiling Example

Here is a sample interactive profiling session for an application running on Tilera hardware, specifically on a device connected to the current host via a USB connection.

  1. First, log onto the machine, and cd to your application directory.

  2. Start up tile-monitor in interactive mode, targeting (and booting) the USB device. Wait for the "Command:" prompt to appear: 

    $ tile-monitor --usb
Command:

  3. Upload the application executable(s) you want to run, as well as any libraries, etc. they depend on that are not already present in the OS. For example: 

    Command: upload your_program             /bin/your_program
Command: upload your_shared_lib.so.1.0.0 /lib/your_shared_lib.so.1.0.0
... etc. ...

  4. tile-monitor provides a number of commands for configuring and invoking the OProfile profiling tools. For example, you'll first want to upload the basic profiling support files, by typing: 

    Command: profile-init

  5. Optionally, if you want more detail on the operations of the OS kernel, you can upload the kernel profiling support files, and request kernel profiling: 

    Command: profile-kernel

    Note that the commands above only need to be invoked once for each profiling session.

  6. If you're running in a "persistent" environment, such as a TILEmpower box, or a USB device or PCI card booted with the --resume option, there may be old profiling state left around from the last profiling run. You can clear out any such prior profiling state on the TILE hardware with this command: 

    Command: profile-reset

  7. You can optionally specify one or more event types to profile for, using the profile-events command to specify the --event argument(s) to pass to the OProfile daemon when it starts up. If you don't explicitly specify any events, tile-monitor will default to using a selection of the most common and useful event names for the chip architecture.

    (Note: commands that specify OProfile options take effect only when the OProfile daemon is started (or re-started). For more information, see the section above on the OProfile Daemon.)

    On TILE Gx, there are four performance counters, and the default profile-events event/interval specification is: 

    Command: profile-events '--event=ONE:12000000 --event=INSTRUCTION_BUNDLE:12000000 --event=LOAD_STALL:12000000 --event=INSTRUCTION_STALL:12000000'

    This finds time spent in code, instructions executed, and relative time stalled for data and instruction cache operations.

    Note that you'll want to compare (specifically take the ratio) of a given event's sample count with the sample count for the ONE event, which occurs every cycle. This gives you a "per-cycle" value that is relatively independent of the event interval you've selected, and hence can be compared from run to run even where the sampling interval is different.

    To only collect statistics on data cache stall time, you can specify: 

    Command: profile-events '--event=ONE:12000000 --event=LOAD_STALL:12000000'

    And compare the LOAD_STALL sample counts with the ONE sample count.

    To only collect instruction cache stall time, use: 

    Command: profile-events "--event=ONE:12000000 --event=INSTRUCTION_STALL:12000000"

    To find estimated TLB miss time (assuming an average of 100 cycles to service a TLB miss), use: 

    Command: profile-events "--event=ONE:12000000 --event=TLB_EXCEPTION:40000"

    These are only a few of the most commonly used events and their uses. For more information about the hardware events and their interpretation, see the MDE Optimization Guide.

  8. You can optionally use the profile-flags command to specify other OProfile configuration flags, for example, to specify the callgraph depth, (in other words, the number of functions to backtrace on the stack each time OProfile handles a sample interrupt), type: 

    Command: profile-flags '--callgraph=20'

    (Note: commands that specify OProfile options take effect only when the OProfile daemon is started (or re-started). For more information, see the section above on the OProfile Daemon.)

    The default callgraph depth is 5, which is a reasonable compromise between coverage and the overhead of capturing the stack at each sampling interrupt. You can specify higher callgraph values to get more coverage of the caller/callee relationships in your program, or a lower callgraph to minimize the overhead if you're only interested in relative time spent in individual functions. Note: OProfile may lose samples if the callgraph depth is large and too many samples are collected in a short period of time.

    If you want to disable callgraph profiling completely (and you are NOT intending to use the profile-analyze to generate a profile.xml file), you can specify the following: 

    Command: profile-flags ''

  9. To start the OProfile daemon and begin profiling data collection, type: 

    Command: profile-start

    This starts the OProfile daemon, if it was not already running, with the currently specified events and options, and also enables profiling data capture.

  10. Run any applications you want to profile, and wait for them to exit. For example, assuming the upload command above, you could do: 

    Command: run your_program [arguments...]

  11. Stop profiling data collection: 

    Command: profile-stop

    NOTE: This command leaves the OProfile daemon running, in case you might want to use profile-start to start data collection again.

  12. Ensure OProfile has written all data, logs, etc. to its TILE-side directory: 

    Command: profile-dump

  13. Now "capture" the profiling data you've collected on the Tilera platform, by copying it to a directory on the host that tile-monitor is running on: 

    Command: profile-capture samples

    The pathname (samples) is the pathname of a directory on the host, relative to the current directory in which you invoked tile-monitor. The above command, for example, creates a samples subdirectory in your application directory and copies the OProfile data into it.

    Note: We'll refer to this host copy of the OProfile data as the profile "capture" directory in discussions below.

  14. If you want to be able to view the profiling data using the tile-eclipse IDE's Profile View, the run the following command on the capture directory: 

    Command: profile-analyze samples

    This creates a profile.xml file in the samples directory, which can then be loaded into the IDE. (For more information, see the tile-eclipse IDE's online help on profiling.)

    If you're doing multiple profiling runs, you can use the profile-analyze command to process the profile output in one of two ways, based on the kind of profiling you're doing.

    First, if each run is independent of the others (for example, if you're repeatedly profiling a program and editing the program in between runs), you can simply profile-capture each profile run's data to a separate directory, and run profile-analyze on each sample directory to generate a separate profile.xml file for that run.

    Alternatively, if you want to combine the output of multiple runs (for example, if you're profiling the same program repeatedly, but using different hardware profiling events for each run), you'll want to capture the output from each profiling run into a subdirectory, and then run profile-analyze on the parent directory of all the runs. This will generate a single profile.xml file covering all the runs in the various subdirectories.

    For example: 

    Command: profile-capture samples/run001
  ...
Command: profile-capture samples/run002
  ...
Command: profile-capture samples/run003
  ...
Command: profile-analyze samples

    If you want to add additional runs to the profile.xml file, just capture more runs into additional subdirectories, then re-run the profile-analyze command on the parent directory to add more sample data to the generated profile file: 

    Command: profile-capture samples/run004
  ...
Command: profile-capture samples/run005
  ...
Command: profile-analyze samples

  15. Optional: If you want to do more profiling in the same tile-monitor session, you can reset the profiling counters by typing: 

    Command: profile-reset

    Then repeat the profiling steps above starting with profile-start, to perform another profiling session.

  16. Finally, when you're done with profiling your application, shut down the OProfile daemon: 

    Command: profile-shutdown

    (Note: commands that specify OProfile options take effect only when the OProfile daemon is shutdown and re-started. For more information, see the section above on the OProfile Daemon.)

  17. And quit out of tile-monitor: 

    Command: quit

    You can now create reports from the captured profiling data, as described below under Generating Profile Reports.

Batch Profiling Example

Note that all of the profiling commands described above can be specified on the command line. This allows you to construct a single tile-monitor command that describes the entire process of configuring the profiler, profiling your application, and capturing/analyzing the data.

For example, all of the steps of the previous interactive example can be invoked from a single tile-monitor command line as shown below: 

tile-monitor --dev usb/0 --resume \
    --upload your_program /bin/your_program \
    ... any other upload commands ...
    --profile-init --profile-kernel \
    --profile-reset \
    --profile-events '--event=ONE:12000000 --event=INSTRUCTION_BUNDLE:12000000 --event=LOAD_STALL:12000000 --event=INSTRUCTION_STALL:12000000' \
    --profile-flags '--callgraph=20' \
    --profile-start \
    --run -+- your_program [arguments...] -+- \
    --profile-stop \
    --profile-dump \
    --profile-capture samples \
    --profile-analyze samples \
    --profile-shutdown \
    --quit

You can now create reports from the captured profiling data, as described below under Generating Profile Reports.

Generating Profile Reports

Capturing the profiling data is only half the task. Now you need to filter and generate a report from the captured data.

This section discusses the OProfile command-line tools for report generation: tile-opreport and tile-opannotate.

In the examples below, we'll assume that you've worked through the Interactive Profiling Example section above, and as a result already have a samples profile capture subdirectory to work with. Also, we assume that you're running the sample commands in the application directory, so the unstripped binaries for the profiled application can be found in the current directory.

NOTE: For many users, particularly those who just want to quickly browse the profile results in a Tilera-centric format, it may be quicker to just run the profile-analyze command in tile-monitor and generate an XML file to be viewed in the tile-eclipse IDE.

Profile Summary Reports (tile-opreport)

To generate a report from profiling data, you can run the tile-opreport command on the profile capture directory you created when profiling your application.

Before we go on, it's only fair to warn you that tile-opreport is a very powerful report generating tool, and with this power comes complexity. The tool can often seem unforgiving and even stubbornly cryptic to new users, given the number of ways one can filter profiling data, and the many ways filtering options can interact and even conflict with one another. It takes patience and some experimentation to determine the correct set of options to construct the report that you want, and it really pays to read the OProfile documentation and understand the effect of tile-opreport's filtering options, rather than simply adding options to the command line at random to see what they do.

In particular, it can be surprisingly easy to "overspecify" filtering options, and thus exclude all of your sample data. You can tell you've done this if tile-opreport prints an error message like the following: 

error: no sample files found: profile specification too strict ?

This error doesn't really indicate there were no sample files to display. Usually it just indicates that you've specified too many filtering options, or have specified two options that conflict with one another, thus excluding all sample files from the set to display.

Often tile-opreport will display an error resulting from an ambiguity in what you've requested; in such cases you need to "merge", or aggregate, the data differently to eliminate the ambiguity. For example, if you omit the "--merge" option altogether, often OProfile will complain about duplicate reporting of one or more events. Specifying "--merge=all" is a simple means to avoid this for reports where you don't have any special filtering requirements.

Another downside of tile-opreport is that the reports it generates are generic and treat the Tilera hardware simply as a collection of SMP cpus. In particular, the processing "tiles" of the Tilera chip are referred to by their Linux OS cpu numbers, and there's no means of referring to them or filtering them by their grid coordinates.

All this is not meant to dissude you from using tile-opreport if you prefer command-line tools. Just be aware that tile-opreport is really a "power user" tool, and should be treated accordingly.

Below we present some quick examples of tile-opreport usage, in particular some combinations of command-line options that generally produce useful reports and serve as good starting points. For a more detailed discussion of how to configure tile-opreport, see the Extended Example section below, as well as the OProfile manual at the OProfile website.

The first report-generating command you'll want to run for just about any sample directory is the following: 

$ tile-opreport archive:samples --merge=all -x -f

INSTRUCTION_BU...|INSTRUCTION_ST...|LOAD_STALL:100...|      ONE:1000000|
  samples|      %|  samples|      %|  samples|      %|  samples|      %|
------------------------------------------------------------------------
      259 93.8406       215 98.1735       186 98.9362       497 95.2107 /hello_world
       15  5.4348         2  0.9132         2  1.0638        21  4.0230 /bin/busybox
        2  0.7246         0       0         0       0         0       0 /usr/bin/ophelp
        0       0         2  0.9132         0       0         4  0.7663 /bin/shepherd

This displays a high-level summary report of the profile data from the profile capture directory samples, merging the results from all processes/threads on all cpus, and displaying summary statistics for each of the executables invoked while your program was running. (For clarity, we've omitted the progress messages displayed by tile-opreport prior to generating the report output.)

The -x option excludes calls between executables and shared libraries or the Linux OS, restricting the output to just code within the named executables and/or libraries. This simplifies the output and gives you a useful "top level" process-based view of the profiling data.

The -f option displays the full pathname for each executable as it was invoked on the remote host.

You can specify a specific event to see (and sort by) just that column: 

$ tile-opreport archive:samples --merge=all -x -f event:ONE

      ONE:1000000|
  samples|      %|
------------------
      497 95.2107 /hello_world
       21  4.0230 /bin/busybox
        4  0.7663 /bin/shepherd

This displays the executables sorted by the ONE event, and thus shows where the most time was spent during the profiling run.

You can narrow the list to a specific executable, such as your application: 

$ tile-opreport archive:samples --merge=all -x -f event:ONE /hello_world

      ONE:1000000|
  samples|      %|
------------------
      497 100.000 /hello_world

Note that here we've specified the absolute remote path of the application, as it was executed on the TILE processor. Unless you're using the -p to supply a local search path for executable files, you should use the full pathname of the executable as shown above, or OProfile may report that it could not locate any data for the specified executable.

One can add per-function summaries to the results, but for this we need a local, unstripped copy of the executable to get symbol information from. Below, we use the -p option to tell tile-opreport to look in the current directory (.) for application binaries. This allows us to use the relative path of the local, unstripped copy of your_program, which can be found in the current directory. 

$ tile-opreport archive:samples --merge=all -lgw -x -f event:ONE -p . hello_world

vma      samples  %        linenr info                 symbol name
00010870 496      99.7988  /u/bswanson/dev/profile/gx/hello_world_fork/hello_world.c:9 main
000315a0 1         0.2012  /home/scratch/bswanson/main_gx_build/tools/glibc/source/malloc/arena.c:467 ptmalloc_init

The -g option adds file and line number columns to the output, and the -w displays the virtual memory address for each function.

You can also get per-function results without restricting the output to a single executable, but in this case we need to add the ${TILERA_ROOT}/tile directory to the search path so Tilera binaries and libraries can be located: 

$ tile-opreport archive:samples --merge=all -l -x -f event:ONE -p .,${TILERA_ROOT}/tile

496      95.0192  /u/bswanson/dev/profile/gx/hello_world_fork/hello_world main

Each section of the callgraph data (separated by the dashed lines) displays sample data for one function, plus its immediate caller and callees. The function of interest in each section is shown outdented, with the caller (if any) indented above it, and the callees (if any) indented below. Lines marked with [self] indicate time spent within the function's own code.

Note: the -c flag to request callgraph information is not compatible with --merge=all, specifically with --merge=lib. A callgraph report will be generated, but it will not contain full callgraph information. This is why we use --merge=tgid,cpu above. It's also an example of why it's never a good idea to add options to the tile-opreport command line without fully understanding their effects.

To view a load-balancing report for a given executable across the cpus in use, merge the results only by process id (tgid) and libraries: 

$ tile-opreport archive:samples --merge=tgid,lib -f event:ONE -p . hello_world

            cpu:0|            cpu:1|            cpu:2|            cpu:3|
  samples|      %|  samples|      %|  samples|      %|  samples|      %|
------------------------------------------------------------------------
      126 100.000       118 100.000       125 100.000       128 100.000 /hello_world

This produces very wide output (one column per tile/CPU), so if you're running on many CPUs we recommend you capture the output in a file and view it in a web browser or an editor that allows horizontal scrolling.

Alternatively, you can restruct the CPU columns displayed with a filter option: 

$ tile-opreport archive:samples --merge=tgid,lib -f event:ONE cpu:0,2 -p . hello_world

            cpu:0|            cpu:2|
  samples|      %|  samples|      %|
------------------------------------
      126 100.000       125 100.000 /hello_world

There are many other combinations of configuration options possible for tile-opreport. We'll cover them in more detail in the extended example below, and you can find full details of the various options in the OProfile manual at the OProfile website.

Annotated Source Reports (tile-opannotate)

To annotate sources with sample counts, use the tile-opannotate command. For example: 

$ tile-opannotate --source archive:samples --merge=all -d . \
                  -p . hello_world > annotated.txt

Programmatic Profiler Control

Profiling can be started and stopped by programs running on the Tile chip. Note that this requires that profiling has been configured with tile-monitor's profile-kernel command: 

  Command: profile-kernel

A program can then invoke (e.g. via a system() call) the OProfile control program, /var/lib/oprofile-support/bin/opcontrol-kernel. For example, this command starts profiling (breaking out profile samples into kernel vs. user space, and also breaking them out via task ID): 

  system("opcontrol-kernel --start --separate=kernel,thread");

The following command stops profiling: 

  system("opcontrol-kernel --stop");

In fact, opcontrol-kernel may be used to perform all the more advanced functions of opcontrol. It merely provides the kernel profiling information as a convenience.

An Extended Example

This section presents a more in-depth example of the usage of tile-opreport and tile-opannotate, using output based on a real-world example application, the open-source network security program Snort.

In the examples below, we'll assume the Snort application was built in the directory /home/yourusername/apps/snort, which is the current working directory, and that the unstripped binary (snort) can be found in that directory.

  1. First, you need to generate profiling data for the application.

    Start tile-monitor, and at the Command: prompt, type: 

      Command: profile-init
  Command: profile-kernel
  Command: profile-reset
  Command: profile-events '--event=ONE:12000000'
  Command: profile-start

    Note that we're restricting event collection to just processor cycles (the ONE event). This is mostly to simplify the output of the examples below, but also because for these examples we're going to focus on time spent in various functions.

    You can omit profile-kernel if you really don't want kernel information, but it doesn't add much overhead. If you don't want the kernel-level sampling data it collects you can simply exclude it when generating a report.

    Adding profile-reset ensures there is no old profiling state before the profiling run starts.

  2. Run the application (in this example, Snort).
  3. Stop profiling, and capture the profiling data to a subdirectory named "samples": 
      Command: profile-stop
  Command: profile-dump
  Command: profile-capture samples
  Command: profile-analyze samples
  Command: profile-shutdown

    The profile-shutdown command ensures that the OProfile daemon is fully shut down. This is particularly important on "persistent" platforms such as TILEmpower, and PCI cards booted with the --resume option. If you don't shut down the daemon, then your next profiling run on the same platform will use the exact same profiling events and options.

    Now you are ready to look at the profile reports.

  4. tile-opreport archive:samples --merge=tgid,cpu -f /snort 
            ONE:1200000|
    samples|      %|
  ------------------
    1443822 100.000 /snort
                ONE:1200000|
            samples|      %|
          ------------------
            1437011 99.5283 /snort
               6811  0.4717 /var/lib/oprofile-support/vmlinux

    This report rolls up all cpus and all processes/threads for /snort and shows some of the program's time was spent in the Linux OS (/var/lib/oprofile-support/vmlinux).

    Notice that we're referencing the /snort executable by the absolute remote path (that is, the path used to invoke it on the TILE processor), and using the -f to see the full remote paths in the output. This is sufficient at this stage since we're only looking at overall sample counts, not per-function or file/line debugging information.

  5. tile-opreport archive:samples --merge=cpu -x /snort | grep "thread ID" 
      Processes with a thread ID of 717
  Processes with a thread ID of 718
  Processes with a thread ID of 720
  Processes with a thread ID of 721
  Processes with a thread ID of 723
  Processes with a thread ID of 725
  ...

    This command shows the process/thread IDs that the Snort binary is running on. (We've used -x to exclude libraries and the Linux OS.) There are a large number of them, because we're running the application on many tiles.

    Similarly, you can see the CPU ids that these threads are running on:

    tile-opreport archive:samples --merge=tgid -x /snort | grep "Samples on CPU"

  Samples on CPU 1
  Samples on CPU 2
  Samples on CPU 3
  Samples on CPU 4
  Samples on CPU 5
  Samples on CPU 6
  Samples on CPU 7
  Samples on CPU 8
  Samples on CPU 9
  Samples on CPU 10
  Samples on CPU 11
  Samples on CPU 12

  6. By using the cpu:nnn, tgid:nnn, and tid:nnn options, you can filter the output by selecting the CPUs and/or processes/threads to view. For example:

    tile-opreport archive:samples cpu:0 /snort 
      Processes with a thread ID of 717
  Processes with a thread ID of 718
            tid:717|          tid:718|
    samples|      %|  samples|      %|
  ------------------------------------
          9 100.000     25598 100.000 snort
                    tid:717|          tid:718|
            samples|      %|  samples|      %|
          ------------------------------------
                  7 77.7778       109  0.4258 vmlinux
                  2 22.2222     25489 99.5742 snort

    This shows us two threads are running on CPU 0.

    There's a wrinkle when we look at threads, however. Processes and threads can be rescheduled between tiles, so sometimes a given process or thread ID may correspond to multiple CPUs.

    This can make it tricky to use tile-opreport to generate the specific results you want, because the CPU and/or tile IDs you specify may overlap, which can result in tile-opreport complaining that you've excluded all the data files, or that some of the filters you've specified (explicitly or implicitly) are in conflict. For example:

    tile-opreport archive:samples tid:712 /snort 
      error: no sample files found: profile specification too strict ?
    tile-opreport archive:samples tid:717,718 /snort 
      tile-opreport error: Already displaying results for parameter tid with values:
  tid:717,tid:718,
  which conflicts with parameter cpu.
  Suggestion: specify tid: or --merge tid

    To avoid these issues, we need a way to determine which tile(s) a given thread was scheduled on. When a thread is scheduled on a single tile, tile-opreport may omit the CPU number from the regular output. With a little careful grepping of the verbose output, however, we can obtain the CPU(s) for any given thread:

    tile-opreport -V sfile archive:samples tid:717 /snort | egrep "^cpu:" 
      cpu: 0
  cpu: 3
  cpu: 36

    This allows us to choose process/thread and CPU filters that are compatible with each other:

    tile-opreport archive:samples tid:717 cpu:0,3,36 /snort 
      CPU: TILE-Gx, speed 1200 MHz (estimated)
  Counted ONE events (Always one - an event occurs every cycle) with a unit mask of 0x00 (No unit mask) count 700000
  Samples on CPU 0
  Samples on CPU 3
  Samples on CPU 36
              cpu:0|            cpu:3|           cpu:36|
    samples|      %|  samples|      %|  samples|      %|
  ------------------------------------------------------
          9 100.000         4 100.000         6 100.000 snort
                      cpu:0|            cpu:3|           cpu:36|
            samples|      %|  samples|      %|  samples|      %|
          ------------------------------------------------------
                  7 77.7778         4 100.000         6 100.000 vmlinux
                  2 22.2222         0       0         0       0 snort

    Or one can simply avoid the issue by merging all the CPUs for a given process or thread:

    tile-opreport archive:samples --merge=cpu tid:717 /snort 
      CPU: TILE-Gx, speed 1200 MHz (estimated)
  Counted ONE events (Always one - an event occurs every cycle) with a unit mask of 0x00 (No unit mask) count 700000
        ONE:1200000|
    samples|      %|
  ------------------
         19 100.000 snort
                ONE:1200000|
            samples|      %|
          ------------------
                 17 89.4737 vmlinux
                  2 10.5263 snort

      tile-opreport archive:samples cpu:0 tid:717,718 -l snort \
     -p .,${TILERA_ROOT}/tile

    We've specified the thread IDs explicitly here, to show how it's done. We could have omitted them, however, since these happen to be the only Snort-related threads on the tile.

  11. We can also look at call graph profiles which present application time hierarchically. This allows us to see not only which functions account for the time in an application but also attribute the time to the various functions' call sites. We can also see how much time particular functions spend in the functions they call versus executing directly. 

      tile-opreport archive:samples --merge=cpu,tgid -c snort \
    -p .,${TILERA_ROOT}/tile

    Here -c specifies call graph profiling. --merge=cpu,tgid means to sum all the data from all the separate threads (ilib processes) and CPUs. Of course, we could also omit --merge=tgid and specify a particular thread instead with tid:XXX where XXX is the thread id of a particular process or thread we want we want to profile. Similarly we could specify CPU:XX to specify a particular tile.

    What exactly does this mean? The first entry means that 710 samples were collected while executing directly in the Snort function fpEvalHeaderSW_noncontent.   12. Those who are more comfortable with gprof can also see the data collected by OProfile in the standard gprof format. To do this you first use tile-opgprof to generate the gprof-formatted data:

    tile-opgprof -o gmon.out archive:samples snort -p .,${TILERA_ROOT}/tile

    This writes gmon.out in the current directory after which you can run tile-gprof with the pathname of your application:

    tile-gprof snort

    Notice that the number of "calls" reported is the same as the number of samples. OProfile does not collect call counts, so this is just an arbitrary substitution for the correct information. Hence, you can ignore call counts in these reports.
