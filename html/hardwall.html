Name

hardwall — /dev/hardwall/* special files for per-cpu resource management

Synopsis

#include <asm/hardwall.h>

Description

The /dev/hardwall/* special files allows user processes to request access to per-cpu resources such as the user dynamic network (UDN) that permits low-latency communication between processors on the Tile chip (/dev/hardwall/udn) or the IPI mechanism that allows tiles to request user-space interrupts be delivered to particular tiles by I/O shims (/dev/hardwall/ipi).

User processes can open the /dev/hardwall/udn file, then use an ioctl to create a reserved rectangle of the chip, e.g. 20 tiles in a 5x4 grid with the upper left corner at tile coordinate (2,3). Tasks (either processes or threads) can then use affinity to "lock" themselves to a single processor within that rectangle, then, via another ioctl() on the still-open hardwall file descriptor, associate themselves with the rectangle and gain access to the UDN. The /dev/hardwall/ipi file works similarly, though it does not require rectangular set of tiles.

Tasks that have access to the UDN can send and receive UDN messages freely with other tasks that have associated themselves with the same hardwall file descriptor. When the file descriptor has been closed by all processes that had it open, the rectangle is deallocated, the kernel performs some cleanup to make sure no UDN messages are left in the network, and that region of the chip can be used by a new application. If any task tries to send a message outside the rectangle, the message will be trapped by the network hardware and the kernel notified, and the kernel will send a SIGABRT to all the tasks that have activated that hardwall file (much like an illegal instruction will kill a process and all the threads in it).

The /proc/tile/hardwall directory contains the current set of hardwalls, one per file.

Ioctls

HARDWALL_CREATE(size)

Create a new hardwall. These are kept unique across the chip by the kernel and must not share cpus with each other. Normally this operation is performed at the start of an application, before any other threads or processes are created. A pointer to a cpu_set_t is passed as the argument to the ioctl; sizeof(cpu_set_t) must be passed to the HARDWALL_CREATE macro, so that the kernel is independent of the cpu_set_t size chosen by user space.

For UDN hardwalls, the cpu_set_t argument must contain a geometrically contiguous rectangular set of cpus. All the cpus in the rectangle must be set in the cpu_set_t mask or an error is returned. (Note that the tmc_cpus_grid_add_rect() is useful for creating such a cpumask.) The set can contain cpus that are not available for Linux to use, such as dedicated hypervisor driver cpus.

HARDWALL_ACTIVATE

Activate the specified hardwall file for access by the calling task. A process or thread that has a /dev/hardwall/ file descriptor that has been associated with a hardwall via HARDWALL_CREATE may use this ioctl to associate itself with that hardwall. The task must have used sched_setaffinity() to bind itself to a single processor (or some equivalent wrapper mechanism, such as the tmc_cpus_set_my_cpu() function or the taskset command). On successful return from the ioctl, the task can then access the resource guarded by the hardwall (e.g. the UDN network, or the IPI functionality of the iorpc driver). For the UDN network, the task is then subject to being killed if it or any other task associated with the hardwall tries to send a UDN message outside the rectangle. If the task subsequently changes its affinity, it will cease to be activated for the hardwall and lose access to the resource guarded by the hardwall.

HARDWALL_DEACTIVATE

Deactivate the specified hardwall file for access by the calling task. This just undoes the effect of a previous HARDWALL_ACTIVATE ioctl.

Usage

Typically, the HARDWALL_CREATE ioctl is used at application startup to associate an opened /dev/hardwall/* file descriptor with a group of tiles. The file thus opened is then the "token" that all other threads and processes must "activate" to be able to access the UDN.

For threaded applications, this is very simple, since all threads share the same table of file descriptors. Each thread, as it starts up, should bind itself to a cpu within the hardwall with a suitable affinity-setting technique, then use the HARDWALL_ACTIVATE ioctl to gain access to the resource on that tile. Typically the hardwall file descriptor would be kept in a global variable or passed as an argument to the new thread.

For applications that start up as a single process and then fork multiple other processes, there is still a simple model to use. The first process creates the hardwall, and then after each fork(), the child process should bind itself to a cpu via affinity, then use the HARDWALL_ACTIVATE ioctl to gain access to the resource on that tile, just like a thread would. If the application uses exec() to start other programs that want UDN access on the tile, either the child can activate the file descriptor, or the new program can activate the file descriptor, as long as the actual file descriptor number can be conveyed to the new program somehow (e.g. in an environment variable). The dup() function is also useful for creating multiple file descriptors referencing the same underlying file, or dup2() to make a file descriptor with a fixed number. Note that normally file descriptors survive an exec(), unless the file descriptor has been marked as "close on exec" via fcntl()'s CLOEXEC flag.

For applications that consist of multiple processes started separately (with no common ancestor to perform the HARDWALL_CREATE) but that wish to share the resource, a more complex method of sharing a single open file must be used. Generally speaking the only way to do this is via sendmsg() and recvmsg() across a Unix socket, using SCM_RIGHTS to transfer a file descriptor in such a way that the receiving process receives a true reference to the underlying open file.

There is no way for multiple processes to access the same hardwall simply by opening the /dev/hardwall/* file multiple times; this is by design, since otherwise it would be possible for some other process to enter the hardwall unexpectedly, and, for example, violate the boundaries of a UDN hardwall rectangle with a bad UDN message and cause all the processes to receive a SIGABRT.

If a user process with an active UDN hardwall tries to send a UDN packet that crosses out of the hardwall region, all the processes that are currently bound to that hardwall will receive a SIGILL signal, with the signal code (si_code) set to ILL_HARDWALL. This is by extension of the fact that if you neglect to set up a hardwall in the first place and then try to use the UDN, your process will receive a SIGILL signal, with the signal code (si_code) set to ILL_PRVREG.

Errors

HARDWALL_CREATE errors:

EFAULT

Pointer to cpu_set_t is invalid.

EALREADY

File descriptor is already associated with a hardwall.

EINVAL

Set contains cpus outside the rectangular dimensions of the chip, or (for UDN hardwall) does not contain a geometrically contiguous rectangle of cpus.

EBUSY

Requested cpu set intersects an existing cpu set.

ENOSYS

The kernel does not have access to the resource due to a boot argument (the UDN due to the "noudn" boot argument, or IPI resources due to the "noipi" boot argument).

Note that, while not an error condition, it is not recommended to set up a UDN hardwall rectangle that includes hypervisor dedicated tiles. This is due to missing functionality in the dedicated hypervisor drivers that prevents them from participating in the UDN cleanup process when the last reference to a kernel hardwall rectangle is closed. The result is that there may still be UDN message traffic stuck in the network when the kernel creates the next rectangle for a new application, which could then cause the new application to hang. This situation is believed to be uncommon unless traffic is directed to the dedicated tile itself by mistake.

HARDWALL_ACTIVATE errors:

ENODATA

This file descriptor does not have a hardwall associated with it, i.e. HARDWALL_CREATE has not already been called on it.

EINVAL

The hardwall is currently being torn down by the kernel due to a hardwall violation, or the calling task is not bound to a cpu within the hardwall rectangle.

EPERM

The calling task is not bound to a single tile.

HARDWALL_DEACTIVATE errors:

EINVAL

This file descriptor does not have a hardwall associated with it, or this task is activated on a different hardwall (or none at all).
