Tilera Board Information Block Format for TILE-Gx Processors (AN040)

Last revised for MDE version 4.3.2, 12 September 2014

Introduction

The Tilera runtime software stack uses a set of system-specific information to govern the configuration and initialization of the Tile Processor™ and the devices it connects to. This data is contained in a structure called the Board Information Block (BIB).

Much of the data in the BIB is unique to a particular hardware design, but is the same on all boards of that design; for instance, a description of the connectivity between the on-chip memory controllers and the memory on the board. Some of the data might be different on every board; for instance, the board's serial number can be recorded in the BIB. Note that a board that contains multiple Tile Processors would need a separate BIB for each Tile Processor.

Tilera's board products are provided with a BIB during manufacturing, and that data is not expected to be modified by the end user. This document is instead aimed at customers designing and producing their own board products. It explains the format of the BIB, and provides the semantics of the various data items within it. Creating a BIB is an essential and early part of bringing up a new board design, since the Tilera Board Test Kit (BTK) derives virtually all of its board-specific knowledge from the BIB, either read from the board under test, or assumed as a result of configuring the BTK to work with a specific board type.

Differences from previous BIB versions

The version of the BIB used with TILE-Gx processors is culturally compatible with, but not at all identical to, the version used on previous Tilera products. Some of the notable differences:

  • Some things are now properties of the chip, rather than the board, and so are no longer part of the BIB. For instance, the frequency-to-voltage tables present in the TILEPro BIB are now burned into the TILE-Gx processor during manufacturing.

  • Some things which could be automatically determined on TILEPro can no longer be so treated on TILE-Gx, and are thus now part of the BIB. For instance, on TILEPro, there was a separate MDIO interface for every Ethernet port, so it was reasonable to just search for a port's PHY by probing all possible addresses on that port's MDIO bus. On TILE-Gx, there are only two MDIO buses for up to 16 Ethernet ports, so the BIB needs to contain the MDIO address of each port's PHY.

  • TILEPro was able to run in a limited mode without a BIB if the system configuration followed a very restrictive set of rules, mostly related to memory configuration. This is not possible on TILE-Gx, mainly because of an explosion of configuration options; without a BIB, the hypervisor will not boot.

  • In several cases, different BIB item types which all applied to the same piece of hardware, but which had been added at different times, were combined into a new unified item type.

  • The BIB creation and manipulation tools provided with TILE-Gx are a substantial upgrade over the tools provided with previous chips.

Creation of the BIB

BIB creation via the BTK

Most users will want to use the BIB support in the BTK to create their BIB. This feature allows you to specify your BIB using a sequence of Python objects, which automatically check all item values for legality, scale some values for easier presentation, and allow many values to be specified using symbolic constants. Once created, you can dump out the contents of the BIB in human-readable form or binary form, and write the BIB to an I2C EEPROM or SPI ROM on your board. You can also read the BIB back from such a ROM (the BTK does this automatically in most cases to learn about the board under test), and then dump it in human-readable or binary form, or even turn it back into Python code which, when run, recreates the BIB.

If you will be using the BTK to create your BIB (which we highly recommend), you can skip the parts of this document which describe the BIB header and trailer, and other low-level formatting details, since the BTK will handle that for you automatically. You will still need to understand the contents of the items within the BIB; the names used here are the same as those used by the Python objects supplied as part of the BTK. This document does not define the generic BIB operations; for that, see the BTK documentation.

BIB creation via custom code

Users who, for whatever reason, cannot use the BTK to create their boards' BIBs may wish to implement their own BIB-creation code. To assist in this process, the MDE includes a C header file with definitions for the BIB header, trailer, and all of the items and structures contained in the body of the BIB. This file is shipped as part of the hypervisor source distribution, and can be found at
$TILERA_ROOT/src/sys/common_hvbme/tilegx/include/hvbme/board_info.h. Customers writing their own code should pay special attention to the sections in this document which describe the proper values for header fields, including the CRC, and are advised to cross-check their output BIB against one generated by the BTK to check the correctness of their implementation.

Location of the BIB

Onboard

An onboard BIB can be located within an I2C EEPROM that is connected to the master I2C interface of the Tile Processor, or within an SPI ROM that is connected to the Tile Processor's SPI interface. An I2C EEPROM or SPI ROM can contain only one BIB, so a separate EEPROM or SPI ROM is required for each Tile Processor.

If the BIB is within an I2C EEPROM, that device must use 16-bit addressing; this means it will most likely be at least 4 Kbit in size. The software that locates the BIB will probe all of the standard addresses for I2C EEPROMs (7-bit addresses 0x50 through 0x57, or 8-bit addresses 0xA0 through 0xAE), on I2C bus zero, and will examine each EEPROM found for a BIB. When a BIB in the appropriate format with a good checksum is found, the search terminates.

If the BIB is within an SPI ROM, there are no special device requirements. Because only one SPI ROM can be connected to a Tile Processor, only one probe operation is performed. If a valid BIB is present in both an I2C EEPROM and the SPI ROM, the I2C version will be used.

Within either type of ROM, the BIB is contained in the set of bytes with the highest-possible addresses; in other words, at the very end of the ROM. For example, if a 32-word BIB (128 or 0x80 bytes) were contained in a 512 Kbit EEPROM (64K or 0x10000 bytes), it would start at byte address 0xFF80 and end at byte address 0xFFFF. This placement, combined with the BIB trailer format, allows the block to be found without prior knowledge of the ROM's size. (It could also be placed at the very start of the ROM, but that space is reserved for bootable code or register settings.)

Compiled-in

In some situations, it is not convenient to include the BIB in onboard nonvolatile memory. For instance, if the I2C or SPI buses were nonfunctional on the first version of a prototype board, you might want to be able to boot the software stack on that board before the I2C or SPI problems were fixed. Another example might be a board where the Tile Processor is a slave coprocessor, controlled by another master processor which supplies its boot image; if the master processor knows all of the board characteristics already, via some other mechanism, it might be duplicative to supply a BIB for the coprocessor.

In cases like these, it is possible to compile the BIB directly into the hypervisor. To do so, follow these steps:

  1. Create the BIB using the BTK. In most cases you will have already implemented a create_bib() method as part of your board class, so you could just do: 

      btk> b = board.create_bib()

  2. Use the BTK to convert the Python BIB object to a C source file. 

      btk> outfile = open("my_bib.c", "w")
  btk> outfile.write(b.get_c())
  btk> outfile.close()

  3. Do a custom hypervisor build, adding that C source file to the hypervisor and the booter. This can be done by specifying the newly created C source file in the EXTRA_SHARED_SOURCES variable: 

      $ cp my_bib.c $TILERA_ROOT/src/sys/hv
  $ cd $TILERA_ROOT/src
  $ make -C sys EXTRA_SHARED_SOURCES="my_bib.c"

    See the System Programmer's Guide for more information on compiling a custom hypervisor, and on using it once you've compiled it.

Notes on Data Formats

In the descriptions below, we use C language conventions for numeric literals and expressions. Numerical values without a prefix are decimal; those with a 0x prefix are in hexadecimal.

Multi-byte quantities (16-bit halfwords or 32-bit words) are arranged within byte-addressed storage in little-endian format, since that is the native format of the Tile Processor. This means that the least significant bits of the quantity appear in the lowest-addressed byte, while the most significant bits of the quantity appear in the highest-addressed byte. So, if a word containing 0x12345678 hexadecimal (305419896 decimal) were stored starting at byte address zero, byte 0 would contain 0x78; byte 1 would contain 0x56; byte 2 would contain 0x34; and byte 3 would contain 0x12.

Format Overview

A BIB consists of a header, zero or more item descriptors, and a trailer. The total length of the BIB can be no more than 1024 4-byte words (4096 bytes), although it is generally much smaller.

Both the header and trailer contain specific values (magic numbers) used to identify the block, as well as the block's length. The trailer also contains a checksum (a Cyclic Redundancy Check, CRC-32), which validates that the block has not become corrupt.

Each item descriptor is variable-length, and contains a type code, an instance number (used to associate multiple items of the same type with a specific piece of hardware), and some type-dependent data.

Header Format

The header consists of three 4-byte words:

  • The first word contains the header magic number. For a BIB, the magic number is 0x30426942 (BiB0 if interpreted as ASCII text).

  • The second word contains the version number of the BIB format. Currently, this is 1 (it was 0 for TILE64 and TILEPro). The version defines the overall structure of the block (for instance, the components of the header or of an item descriptor); it will not change just because new item types are added.

  • The third word contains the number of 4-byte words in the block, not counting the header or trailer. So, for a block which contained no items, this word would contain zero.

BIB Body

The body of the BIB is made of of a sequence of items, each with an item descriptor and zero or more members. Members may be booleans; bitfields or integers of various sizes; character strings; structures, which are containers for members; fixed- or variable-length arrays of those types; or unions, which contain different members, depending on a selector field. Members, even structures, do not stand alone; they only appear as part of an item.

Overall Item Format

Each item consists of a 4-byte item descriptor, followed by zero or more members; the item's length in bytes is always an integral multiple of four. If the last member of an item is a character string, and the last character in such a string is not aligned with the end of the item, then any remaining filler bytes are zero, but if it is, the string will not be null-terminated.

The descriptor itself consists of three subfields:

  • The low 8 bits of the 4-byte word contain the number of 4-byte words in the data payload; this does not count the descriptor itself, so in a descriptor without a data payload, this field is zero.

  • The next 8 bits of the 4-byte word (bits [15:8]) contain the instance number of the descriptor. The instance number is used to distinguish between multiple descriptors with the same type, and its interpretation depends upon the particular descriptor type is it used with. Often an instance number is associated with a particular hardware entity; for instance, a descriptor that contains a particular property of memory shim n might have instance number n. In some cases the instance number is a structure.

  • The top 16 bits of the 4-byte word contain a type field. Valid types are detailed in the next section.

Since the length is included as part of the descriptor, items of unknown type may be skipped by code that reads the block.

Specific Item/Structure Descriptions

The name of the item or structure appears in the first line of its description. This is used to identify the item when using the BIB-building support in the BTK (where it will appear in lower-case) or when using the C structures in the supplied header files (where it may appear in upper- or lower-case, perhaps with a prefix, depending on the context). Items also have a code, which is included as part of the item's descriptor.

In the tables following:

  • Pos is the position of a member within an item, in the format w.b. w is the 4-byte word offset of the first bit of the item, while b is the bit offset within the word of that first bit; both are numbered from zero, and in the case of the bit offset, bit 0 is the least significant bit of the little-endian 4-byte word. (This column is really informational; if you use the BTK to generate your BIB, or you write your own code using the C structure definitions, this will be taken care of automatically.)

  • Len is the length, in bits, of the member. For arrays, this will be given as the length of one array element, an asterisk, and the number of array elements; if the latter is ?, the array is variable-length. Variable-length arrays must always be the last member of an item, so the number of elements in one can be deduced from the array's position and the total item length.

  • Name is the name of the member. As previously noted, these names are used to set the member

  • Type is the type of data contained by the member. These are typically standard C data types, but may be names of BIB structures; for arrays, the type name is followed by a length in brackets, which may be ? for variable-length arrays.

  • Description describes the meaning and use of the member. In some cases, a description may outline how the current software stack uses the member's value. This is purely advisory, and may change in the future.

Item MAX_MEM_SPEED (code 8)

This entry limits the speed at which a memory shim's memory is run. The entries in the speed array are indexed by the number of DIMMs installed; if one is installed, the first array entry is used, if two are installed, the second array entry is used, etc. If there are fewer array entries than installed DIMMs, the last entry in the array will be used.

Note that this is a board property. The actual speed will be the lowest of this value; the maximum speed supported by the processor; and the maximum speed supported by the installed memory, as described by the SPD for a DIMM or a virtual SPD for an onboard memory bank. Other factors evaluated by the memory controller configuration software may also result in a lower memory speed.

The instance number is numeric. Its value is the mshim instance.

Trailer Format

The trailer consists of three 4-byte words:

  • The first word contains a 32-bit Cyclic Redundancy Check of the block's contents, not including the header and trailer words. The polynomial used is the standard IEEE 802.3 version, as implemented by the Tile Processor's crc32_8 and crc32_32 instructions. The following C code could be used to calculate this value: 

      #include <stdint.h>

  uint32_t
  crc(uint8_t* ptr, int len)
  {
    uint32_t crc = ~0;

    while (len--)
    {
      uint8_t input = *ptr++;

      for (int i = 0; i < 8; i++)
      {
        crc = (crc >> 1) ^ (((input ^ crc) & 1) ? 0xEDB88320 : 0);
        input >>= 1;
      }
    }
    return ~crc;
  }

    Other, more efficient algorithms, using lookup tables, also exist and are freely available.

  • The second word contains the number of words in the block, not counting the header or trailer. So, for a block which contained no items, this word would contain zero.

  • The third word contains the trailer magic number. For a BIB, the magic number is 0x31426942 (BiB1 if interpreted as ASCII text).

Extensions

The BIB format is still evolving. The items currently included within it have been designed to handle variations among Tilera's current board products. To some extent we have tried to anticipate some other variations, which we expected would be of interest to customers. However, the current set of items is almost certainly not sufficient to handle every possible board configuration that a customer may require. If Tilera's current software stack is unable to cope with some specific property of your board application, you have two options.

One option is to customize the Tilera runtime software stack for your board. Tilera provides the source code for our low-level booter and hypervisor as part of the MDE, which you can modify to support your board (see the System Programmer's Guide for details). You could do this in a non-configurable fashion (resulting in a binary which would only run on your board), or you could develop a new BIB item, which triggers the new behavior (resulting in a binary that could run on your board, or on one of Tilera's standard boards). If you do choose to add your own data types or codes to the BIB, Tilera suggests you use values with the high bit set (i.e., types of 0x8000 or greater, codes of 0x80000000 or greater). This will prevent conflicts with future items added by Tilera.

The other, often better option involves asking if Tilera could provide support for a new BIB parameter, which would allow the software stack to invoke whatever board-specific behavior you need. This has a distinct advantage — Tilera would integrate support for the new configuration option into the standard software stack, and maintain it in the future. In this case, you would not need to merge your custom changes into future versions of the MDE. While Tilera may not be able to accommodate every such request, we will give each request very serious consideration, since in addition to helping you, it may help future customers with similar needs.

Either way, we urge you to discuss your plans with Tilera customer support or your Field Applications Engineer. We may be planning development of, or may have already developed, features which could be useful for your board application.

Additional Information Blocks

Note: this section describes an advanced topic which is not relevant to most board designs.

The BIB is designed to describe characteristics of a board or system which are expected to be stable. For instance, a board's serial number, or an Ethernet interface's MAC address, does not normally change after the board is manufactured. However, some systems are designed with removable components. If those components have attributes which should be described in the BIB, that creates a problem, since the BIB is not intended to be modified.

For instance, some models of the TILEncore card support an optional mezzanine card which, when installed, provides additional SFP and SFP+ module sockets. When that mezzanine is installed, we would like for certain entries to be added to the board's BIB; for instance, type SFP entries describing the wiring for those sockets. Also, that mezzanine has its own attributes (part number, board description) which we would like to make available through the BIB.

The solution to this problem is the additional information block (AIB). An AIB looks very much like a BIB, and contains the same item types as a BIB. It is added to an I2C EEPROM on the optional module. The BIB on the main board contains an item of type AIB which points to the additional EEPROM. If that EEPROM is present at boot time, then entries from the AIB in that EEPROM are effectively incorporated, by reference, into the BIB. Note that the BIB on the main board is not actually modified; this incorporation is done virtually, via software. In the current implementation, the BIB is copied into memory at boot time, then items from any AIBs are inserted into that in-memory copy, which is then used to satisfy any requests for BIB information.

BIB items are searched in the order that they occur in the coalesced BIB; for most item types, once a matching entry is found, the search stops. Entries incorporated by reference from an AIB replace the original AIB pointer item from the main board's BIB. Thus, an entry from an AIB may override an entry from the BIB on the main board, or not, depending upon whether the AIB pointer item occurs before, or after, the main board version of the BIB entry.

AIB Format

An AIB has almost exactly the same format as a BIB. It has a header, a set of items and a trailer, just like the BIB. It's also found in the same location (the very end of an I2C EEPROM). The only difference is in the magic numbers in the header and trailer. For an AIB, the header magic number is 0x30624961 (aIb0 in ASCII), and the trailer magic number is 0x31624961 (aIb1 in ASCII).

AIB Instance Numbers

As noted above, items in an AIB can potentially override BIB items. In some cases, this is desirable, but in other cases, it's not. For instance, the BIB might have an entry of type BOARD_SERIAL_NUM; an AIB in an optional board might also have such an entry. We'd really like both of those to show up in the final, coalesced BIB. One easy solution is to give the BIB entry instance number 0, and the AIB entry instance number 1. That seems workable; but what if you have a main board with two slots, each of which can take the same type of optional board? We can ensure that the I2C EEPROMs on each board get different addresses by supplying different signals to the plugin boards depending on which slot they're in; we can thus include two AIB pointer entries in the main board's BIB, incorporating both AIBs into the final BIB. However, the instance numbers present a problem: if we manufacture the optional board so that its entries have instance number 1, and then we install two of them, the two items from the AIBs will collide.

To avoid this situation, a special instance number, 0xFF, has been defined for AIB entries. When entries from an AIB are incorporated into the final, coalesced BIB, any entries from the AIB with instance number 0xFF are given the instance number from the AIB item, instead. Thus, the main BIB gets to control the instance numbers used by the AIB items. Here's an example:
