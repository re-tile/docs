Name

tile-mkrom — Bootstream Formatter

Synopsis

tile-mkrom [ -b | --bootfile ] boot-file [ -o | --output ] output-file [ { -f | --filesystem } append-file ] [ { -c | --config } width x height ] [ { -s | --start } x , y ] [ { -u | --ulhc } x , y ] [ -d | --dynamic ] [ -n | --no-snake ]
[ -H | --hypervisor ] [ -S | --secondary ] [ -C | --crc ]
[ -t | --stripe ] [ { -m | --memory } speed-list ] [ { -D | --dev } device[:speed0[,speed1]] ] [ { -p | --post } [ Q ] { t | q } ]
[ { -x | --extra } { x , y | value } ]

Description

tile-mkrom extracts the executable code and data from a Tile ELF format executable file and produces an output boot stream which can be interpreted by the built-in booter within a Tile Processor, or by one of the standard Tilera boot programs.

Note

Most users should never need to use tile-mkrom directly; it is run automatically by tile-mkboot. tile-mkrom will likely have a different interface, and may even disappear entirely, in a future release of the MDE.

Options

-b , --bootfile bootfile

Specify the source executable file. This must be a Tile ELF statically-linked executable (not an object module or a dynamically-linked program).

-o , --output output-file

Specify the output file.

-f , --filesystem append-file

Specify a second input file, which will not be reformatted, but which will be emitted unchanged to the output file after the formatted boot stream. This option was historically used to add the hypervisor filesystem to the output boot image (hence the option name). Nowadays it is generally used only to build a multi-segment boot image in several steps; see Example.

-c , --config widthxheight

Specify the size of the chip to be booted; width and height are measured in tiles. This option is almost universally omitted in favor of dynamic discovery; see the --dynamic option. Only relevant when the --no-snake option is not specified.

-s , --start x,y

Specify the tile coordinates of the tile into which the boot stream will be injected. This option is almost universally omitted in favor of dynamic discovery; see the --dynamic option. Only relevant when the --no-snake option is not specified.

-u , --ulhc x,y

Specify the tile coordinates to be assigned to the tile in the upper left-hand corner of the tile grid. The default for this option is 1,1; the Tilera hypervisor is not tested with other values, and they will almost certainly produce unexpected behavior.

-d , --dynamic

Use the L1 booter's dynamic discovery algorithm to determine the size of the tile grid. Only relevant when the --no-snake option is not specified.

-n , --no-snake

Omit the extra packet of information which describes the tile grid and other boot options from the output boot stream. This option must be used when formatting anything other than the level-1 boot executable (hv_l1boot).

-h , --hypervisor

Format the output stream in hypervisor boot format. This format should be used when formatting anything but the level-0.5 booter (hv_lhboot).

-S , --secondary

Format the output stream in hypervisor boot format, but do not include the normal CRC. This format is used internally by the Tilera Board Test Kit (tile-btk).

-C , --crc

Add a CRC to an output stream formatted in level-0 boot format. This should only be used when formatting the level-0.5 booter (hv_lhboot).

-t , --stripe

Add a flag to the output boot stream which requests that the L1 booter enable memory striping. This option must not be used when formatting anything other than the level-1 boot executable (hv_l1boot). This option is only effective on the TILEPro64 processor.

-m , --memory speed-list

Add data to the output boot stream which requests that the L1 booter set the speed of the memory interfaces. By default, all memory interfaces run at the maximum speed supported by the system; this option allows the user to decrease, but not increase, that speed.

The format of speed-list is a comma-separated list of decimal values, each representing the speed of a memory controller in millions of transactions per second. The first value in the list sets the speed of memory controller (mshim) 0, the second sets the speed of memory controller 1, and so on. If there are more memory controllers than items in the list, the last item is applied to all subsequent controllers; this means that specifying only one value sets the speed of all controllers. Specifying a value of zero for a controller leaves its speed unchanged from the default.

-D , --dev device[:speed0[,speed1]]

Add data to the output boot stream which requests that the L1 booter enable a particular I/O shim, and optionally set its clock speed or speeds, specified in MHz. If no -D options are present, all shims will be enabled, but if any are present, only the specified shims will be enabled. The pseudo-device core sets the core processor frequency.

-p , --post [ Q ] { t | q }

Add data to the output boot stream which requests that the L1 booter change its power-on self-test (POST) behavior. If t is specified, thorough POST will be run; if q is specified, quick POST will be run. If Q is added to one of those option letters, then the L1 booter will prompt the user, asking which sort of POST to run; in that case, the t or q specifies which sort of POST will be run if the user types nothing at the prompt. By default, the booter behaves as if Qt were specified.

-x , --extra { x,y | value }

Add a tile coordinate, or a 32-bit numeric value, to the packet of information which describes the tile grid and other boot options in the output boot stream. A numeric value with a leading 0x will be interpreted as hexadecimal. This option is used internally by the Tilera Board Test Kit (tile-btk). This option may appear more than once.

Example

As noted above, we do not recommend running tile-mkrom directly; tile-mkboot should be used instead. However, to provide some insight into how the bootrom assembly process works, here is an example of how tile-mkboot does it.

First, we format the hypervisor binary. We specify -n to omit the L1 booter configuration data, and -H to get the hypervisor boot format. 

$ tile-mkrom -n -H -b my_custom_hv -o temp_1

Next, we format the L1 booter. We omit -n so that we get the L1 booter configuration data, and specify -H to get the hypervisor boot format. We specify the formatted hypervisor as the filesystem so that it is appended to the output file. 

$ tile-mkrom -d -H -b hv_l1boot -o temp_2 -f temp_1

Then, we format the L0.5 booter. We specify -n to omit the L1 booter configuration data, -C to get a CRC, and omit -H to get the hardware (L0) boot format. We specify the output file from the previous step as the filesystem to append the formatted L1 booter and hypervisor to our output file. 

$ tile-mkrom -C -n -b hv_lhboot -o temp_3 -f temp_2

Finally, the hypervisor file system is constructed and appended to the previous output file. tile-mkboot does this internally, but it's equivalent to doing: 

$ cat temp_3 hvfs_file >output.bootrom
