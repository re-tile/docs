Name

tile-perf2xml — converts perf events data files into profile.xml file

Synopsis

tile-perf2xml [-h | --help] [-v | --verbose] [-c | --cd directory] [[-i | --input] perf.data ...] [-o | --output profile.xml] [-k | --split-cpus] [-p | --statistic-xml ProfileMetadata.xml] [-m | --monitor-xml monitor.xml] [-l | --vmlinux /path/to/vmlinux] [-s | --symbol-map host_path tile_path] [perf.data ...]

Description

The tile-perf2xml tool is used to convert the raw profile statistic data files output by the perf command into profile.xml files that can be viewed in the tile-eclipse IDE.

This tool is needed because by design, the perf command treats all hardware platforms generically, as collections of cpus. The profile.xml file output by tile-perf2xml incorporates some Tilera-specific metadata about both hardware profiling events and the TILE platform itself, and also reorganizes the profiling output in a form that facilitates "top-down" browsing and searching.

Usage Note: This command is available in both an x86 version and a TILE-native version, both named tile-perf2xml. The x86 version of the command is normally intended to be run internally by the tile-monitor monitoring utility. The native version of the command is intended for when you are running the perf commands directly on the TILE platform, and not making use of tile-monitor.

Command-Line Options

-h | --help

Optional. Displays usage information and list of command options, then exits.

-v | --verbose

Optional. Displays progress output messages and other useful information. By default, this command operates silently except for usage/execution error messages.

-c | --cd directory

Optional. Sets the default directory for subsequent pathname options. Can appear more than once. Before the first instance of this option, the default directory is the directory in which tile-perf2xml was invoked.

[-i | --input] perf.data

Required. Specifies a perf data file to process. You can specify multiple input data files in a single tile-perf2xml call. Each data file is treated as a separate "session" in the resulting profile.xml file. (The -i | --input marker is optional; any non-option arguments on the command line are assumed to be input data file pathnames.)

-o | --output profile.xml

Optional. Specifies the output pathname of the resulting profile.xml file. If not specified, defaults to profile.xml in the current directory.

-k | --split-cpus

Optional. Requests that profile data for a process/thread seen on multiple CPUs be listed separately for each CPU, rather than being aggregated across CPUs. (Note that this will produce multiple entries in the profile for the process, one for each tile it is seen on.)

-p | --statistic-xml ProfileMetadata.xml

Optional. Specifies the ProfileMetadata.xml file to use for statistic metadata. When running natively, this file needs to be copied/uploaded from your MDE directory, and can be found in ${TILERA_ROOT}/etc/ProfileMetadata.xml. If not specified, tile-perf2xml looks for this file first in the current directory and then in the MDE /etc directory. (Note: make sure you use the correct metadata file for your hardware; when profiling on a Gx chip, use the metadata file from a Gx MDE installation.)

-m | --monitor-xml monitor.xml

Optional. Specifies the monitor.xml file to use for hardware metadata. This file can be generated using tile-monitor's --generate-monitor-xml option (this is discussed below). If not specified, tile-perf2xml looks for this file in the same directory as the profile data input file.

-l | --vmlinux /path/to/vmlinux

Optional. Specifies vmlinux binary to use for OS/kernel symbol information. If not specified, defaults to /boot/vmlinux. If the vmlinux file is not found or does not contain symbol information, the profile data can still be processed, but there will be more unresolved OS/kernel addresses in the profile.xml output.

-s | --symbol-map host_path tile_path

Optional. Specifies a mapping between a "host" binary and a corresponding "tile" binary. This is used when the "tile" binary (the actual profiled binary) has a different pathname than the host file, or is a "stripped" version of the host file, lacking symbol information. Whenever the "tile" file is encountered in the profile data, the "host" file will be used to obtain symbol information. This option can be specified multiple times, to specify multiple mappings.

Note that the monitor.xml metadata file also may specify host/tile pathname mappings (see the discussion of generating this file below), so if this file already defines the mappings you need, you do not need to repeat them on the tile-perf2xml command line. If you do specify a mapping that conflicts with a mapping in the monitor.xml file, the tile-perf2xml command-line mapping overrides the one in the monitor.xml file.

perf.data

A perf.data file to process. (See description below under Usage.)

Usage

When you run the perf record command, it generates a raw profile data file (by default named perf.data). You can then run the tile-perf2xml command on this output file to produce a profile.xml file. This can be done either on x86, or natively.

The two use cases are discussed separately below.

The resulting profile.xml file is self-contained, and can be copied/archived wherever you like. The original sample capture directory can then be deleted if you no longer need it and want to reclaim the space.

Using tile-perf2xml on an x86 host

If you have a perf.data file, representing the profiling output of a single run, you can apply tile-perf2xml directly to it, to generate a profile.xml file for that run: 

    tile-monitor ... \
      --profile-tool perf \
      ...
      --profile-capture samples

    tile-perf2xml -o samples/profile.xml \
      -m samples/monitor.xml \
      samples/perf.data

This takes the perf data file(s) perf.data, and combines it with the other metadata files (described below) to produce a profile.xml file.

Note that this is essentially what tile-monitor's profile-analyze command does for you, so in general it's simpler to let tile-monitor do all the work: 

    tile-monitor ... \
      --profile-capture samples \
      --profile-analyze samples

You can also use tile-op2xml to combine the data from multiple profiling runs into a "multi-session" profile.xml file. The profiling data from each run is given a unique "session ID" in the profile, so you can view the data from multiple runs in a single instance of the IDE's Profile View.

To do this, you'll need to capture the data for each profiling run as a subdirectory of a parent capture directory, and then run tile-op2xml on the parent directory -- the tool will recognize there are multiple capture subdirectories, and automatically process them in turn, producing a single profile.xml file. For example: 

    mkdir multiple_samples

    tile-monitor ... \
      --profile-capture multiple_samples/run1

    tile-monitor ... \
      --profile-capture multiple_samples/run2

    ...

    tile-monitor ... \
      --profile-capture multiple_samples/runN

    tile-perf2xml -o multiple_samples/profile.xml \
      -m multiple_samples/run1/monitor.xml \
      samples/run1/perf.data
      samples/run2/perf.data
      ...
      samples/runN/perf.data

Again, you can do the same thing via tile-monitor's profile-analyze command, so in general it's simpler to let tile-monitor do the work: 

    mkdir multiple_samples

    tile-monitor ... \
      --profile-capture multiple_samples/run1

    tile-monitor ... \
      --profile-capture multiple_samples/run2

    ...

    tile-monitor ... \
      --profile-capture multiple_samples/runN

    tile-monitor --resume \
      --profile-analyze multiple_samples

As shown above, you use tile-monitor to perform multiple profiling captures to subdirectories of multiple_samples, and then run the profile-analyze command on the parent directory multiple_samples.

Regardless of how you invoke tile-perf2xml, you can then view the resulting profile.xml file in the IDE as usual: 

    tile-eclipse --profile profile.xml

Using tile-perf2xml "natively" on TILE

You can run the native tile-perf2xml command on the perf.data file to produce a profile.xml file: 

    perf record -g -c 1000000 \
      -e ONE,INSTRUCTION_STALL,INSTRUCTION_BUNDLE,LOAD_STALL \
      -o perf.data \
      -- \
      your_app arg1 ... argn

    tile-perf2xml -o profile.xml \
      -m monitor.xml \
      perf.data

This takes the perf data file perf.data, and combines it with other metadata files (described below) to produce a profile.xml file.

Regardless of how you invoke op2xml, you can then view the resulting profile.xml file in the IDE as usual: 

    tile-eclipse --profile profile.xml

Metadata files needed by tile-perf2xml

The tile-perf2xml command requires a couple of metadata files in order to properly process your data file. If you're running tile-perf2xml on x86, it will automatically locate these files either in your MDE installation directory or in the profile capture directory itself. You can optionally override the default locations for these files, but when running the tool on the host you will rarely need to do this. If you are running the tile-perf2xml command natively, you'll need to obtain/generate these metadata files yourself and upload them to the TILE platform. This is not as bad as it sounds: these files are easily obtainable from your MDE installation, and they can also be uploaded once and reused as long as your program build does not substantially change.

First, the ProfileMetadata.xml file provides Tilera-specific metadata about the hardware events for the TILE platform. This file can be found in your MDE installation: 

    ${TILERA_ROOT}/etc/ProfileMetadata.xml

or 

    ${TILERA_ROOT}/tile/etc/ProfileMetadata.xml

Note: Make sure that you use the correct metadata file for your architecture; in other words, upload the metadata file from a Gx MDE installation when profiling on Gx hardware.

Second, the monitor.xml file provides Tilera-specific metadata about the specific hardware you're running on. This is automatically generated by tile-monitor's profile-capture command when tile-monitor creates a capture directory.

However, the tile-monitor command does include an option to generate this file on demand. It can be invoked as follows: 

    tile-monitor --dev your_tile_host --resume \
       --symbol-map host_path tile_path \
       ... any other symbol-file path mappings ... \
       --generate-monitor-xml monitor.xml \
       --quit

This connects to your hardware, without rebooting, generates the monitor.xml file, including the set of remote-to-local symbol file mappings you specify, and then exits.

Note that the tile-perf2xml command provides a --symbol-map command-line option, which has the same meaning as the one for tile-monitor, and takes precedence if there is a conflict between the two. You can specify symbol mappings using either the monitor.xml file, the tile-perf2xml command line, or both.
