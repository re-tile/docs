Name

tile-op2xml — converts OProfile raw data into profile.xml file

Synopsis

tile-op2xml [-h | --help] [-c | --cd /base/path] [-o | --output profile.xml] [-x | --max-depth frames] [-k | --split-cpus] [-p | --statistic-xml ProfileMetadata.xml] [-m | --monitor-xml monitor.xml] [-D | --debuginfo-dir debuginfo_pathname] [-R | --debuginfo-root debuginfo_root_pathname] [oprofile_capture_directory ...]

Description

The tile-op2xml tool is used to convert the raw profile statistic data generated by the OProfile utility into profile.xml files that can be viewed in the tile-eclipse IDE.

This tool is needed because by design, the OProfile utility treats all hardware platforms generically, as collections of cpus. The profile.xml file output by tile-op2xml incorporates some Tilera-specific metadata about both hardware profiling events and the TILE platform itself, and also reorganizes the profiling output in a form that facilitates "top-down" browsing and searching.

Usage Note: This command is available in both an x86 version and a TILE-native version, both named tile-op2xml. The x86 version of the command is normally intended to be run internally by the tile-monitor monitoring utility. The native version of the command is intended for when you are running the OProfile utility directly on the TILE platform, and not making use of tile-monitor.

Command-Line Options

-h | --help

Optional. Displays usage information and list of command options, then exits.

-c | --cd /base/path

Optional. Specifies a "current directory" for paths in subsequent arguments. May be specified multiple times.

-o | --output profile.xml

Optional. Specifies the output pathname of the resulting profile.xml file. If not specified, defaults to profile.xml in the current directory.

-x | --max-depth frames

Optional. Normally not needed. Sets an upper bound on the depth of stack traces in the resulting profile output file. This can be useful in rare cases where recursive code or complex code flows results in excessively deep stack traces.

-k | --split-cpus

Optional. Requests that profile data for a process/thread seen on multiple CPUs be listed separately for each CPU, rather than being aggregated across CPUs. (Note that this will produce multiple entries in the profile for the process, one for each tile it is seen on.)

-p | --statistic-xml ProfileMetadata.xml

Optional. Specifies the ProfileMetadata.xml file to use for statistic metadata. When running natively, this file needs to be copied/uploaded from your MDE directory, and can be found in ${TILERA_ROOT}/etc/ProfileMetadata.xml. If not specified, tile-op2xml looks for this file first in the current directory and then in the MDE /etc directory. (Note: make sure you use the correct metadata file for your hardware; when profiling on a Gx chip, use the metadata file from a Gx MDE installation.)

-m | --monitor-xml monitor.xml

Optional. Specifies the monitor.xml file to use for hardware metadata. This file can be generated using tile-monitor's --generate-monitor-xml option (this is discussed below). If not specified, tile-op2xml looks for this file in the same directory as the profile data input file.

-D | --debuginfo-dir debuginfo_pathname

Optional. Specifies the pathname to use when looking up separate GDB debuginfo (.debug) files. Defaults to /usr/lib/debug. Note that when processing cross-platform profiling data (for example, using tile-op2xml on x86 to process a TILE profile capture directory), you should used the companion option --debuginfo-root to specify the root directory for the --debuginfo-dir pathname.

-R | --debuginfo-root debuginfo_root_pathname

Optional. Specifies an alternate root directory for the --debuginfo-dir pathname. When processing cross-platform profiling data (for example, using tile-op2xml on x86 to process a TILE profile capture directory), this option should be set to the /tile directory of the Tilera MDE installation corresponding to the data that you're processing.

On x86, this defaults to ${TILERA_ROOT}/tile, which is generally what you want. You would only need to change this if processing profiling data generated using a different MDE release, in which case you should specify the ${TILERA_ROOT}/tile pathname for that MDE installation.

On TILE, this option defaults to the empty string, so the --debuginfo-dir pathname is used as-is. You do not need to specify an alternate root when processing profiling data natively one TILE.

oprofile_capture_directory

A profile-capture data directory to process. (See description below under Usage.)

Usage

The OProfile utility stores its generated profiling data in the TILE file system, as a directory hierarchy found under: 

    /var/lib/oprofile/samples/current/...

When you invoke the profile-capture command of tile-monitor, it creates a host-side "capture" directory that contains:

  • a copy of the OProfile profiling tree
  • a Tilera-specific monitor.xml metadata file generated by tile-monitor, which describes the hardware on which the profiling run was made

You can then run the x86 tile-op2xml command on the capture directory to generate a profile.xml file. This file encapsulates all the profiling data in the capture directory, and can be viewed in the Profile View of the IDE.

You can also run the native tile-op2xml command on the original, TILE-side OProfile sample directory to generate a profile.xml file that encapsulates the data in the capture directory. (It's important to run this command on the original TILE-side profiling data, since profiling data copied to the x86 host directory is post-processed so the x86 versions of the OProfile utilities can operate on it.)

These two use cases are discussed separately below.

The resulting profile.xml file is self-contained, and can be copied/archived wherever you like. The original sample capture directory can then be deleted if you no longer need it and want to reclaim the space.

Using tile-op2xml on an x86 host

There are a couple ways to run the x86 tile-op2xml command on a sample capture directory.

If you have a single capture directory, representing the profiling output of a single run, you can apply tile-op2xml directly to it, to generate a profile.xml file for that run: 

    tile-monitor ... \
      --profile-capture samples

    tile-op2xml -o samples/profile.xml samples

Note that this is essentially what tile-monitor's profile-analyze command does for you, so in general it's simpler to let tile-monitor do all the work: 

    tile-monitor ... \
      --profile-capture samples \
      --profile-analyze samples

You can also use tile-op2xml to combine the data from multiple profiling runs into a "multi-session" profile.xml file. The profiling data from each run is given a unique "session ID" in the profile, so you can view the data from multiple runs in a single instance of the IDE's Profile View.

To do this, you'll need to capture the data for each profiling run as a subdirectory of a parent capture directory, and then run tile-op2xml on the parent directory -- the tool will recognize there are multiple capture subdirectories, and automatically process them in turn, producing a single profile.xml file. For example: 

    mkdir multiple_samples

    tile-monitor ... \
      --profile-capture multiple_samples/run1

    tile-monitor ... \
      --profile-capture multiple_samples/run2

    ...

    tile-monitor ... \
      --profile-capture multiple_samples/runN

    tile-op2xml -o multiple_samples/profile.xml multiple_samples

Again, you can do the same thing via tile-monitor's profile-analyze command, so in general it's simpler to let tile-monitor do the work: 

    mkdir multiple_samples

    tile-monitor ... \
      --profile-capture multiple_samples/run1

    tile-monitor ... \
      --profile-capture multiple_samples/run2

    ...

    tile-monitor ... \
      --profile-capture multiple_samples/runN

    tile-monitor --resume \
      --profile-analyze multiple_samples

As shown above, you use tile-monitor to perform multiple profiling captures to subdirectories of multiple_samples, and then run the profile-analyze command on the parent directory multiple_samples.

Regardless of how you invoke tile-op2xml, you can then view the resulting profile.xml file in the IDE as usual: 

    tile-eclipse --profile profile.xml

Using tile-op2xml natively on TILE

You need to run the native tile-op2xml on the current TILE-side OProfile output directory (that is, /var/lib/oprofile/samples/current/...), which is created either by using the profiling commands of tile-monitor, or by running the native OProfile utilities directly on TILE.

The simplest method is to use tile-monitor to drive most of the profiling process, up to the profile-capture stage. At this point, you'll have the OProfile capture directory on TILE (/var/lib/oprofile/samples/current/...), and the captured sample directory on the x86 host, which includes the monitor.xml file that is needed for processing the profiling data. (You can also generate the monitor.xml file separately. See the discussion of this file below.)

At this point, you can run the native tile-op2xml as follows: 

    tile-monitor ... \
      ... normal sequence of profiling commands ...
      --profile-capture samples

    tile-monitor --resume \
      --root \
      --here

    (in a tile-console shell, or other shell window on TILE:)

    # tile-op2xml -m samples/monitor.xml -o samples/profile.xml \
         /var/lib/oprofile/samples/current

This example uses the monitor.xml file that was generated by the profile-capture step, and writes the resulting profile.xml file to the samples capture directory. This produces the same result as if you had run profile-analyze from tile-monitor.

Note that you generally need to run tile-monitor in two separate sessions as shown above, since the profile-... commands assume that you are not mounting /usr/bin, while the native op2xml tool is found in /usr/bin in the standard MDE installation tree. So you must first do your profiling, then separately run the native tile-op2xml tool. (This is one reason why the native tile-op2xml command is only really needed/useful when you are using the native OProfile commands without tile-monitor.)

Also note that the above invocation of tile-op2xml is essentially what tile-monitor's profile-analyze command does for you, so unless you have a specific need to run op2xml natively, in general it's simpler to let tile-monitor do all the work: 

    tile-monitor ... \
      --profile-capture samples \
      --profile-analyze samples

Regardless of how you invoke op2xml, you can then view the resulting profile.xml file in the IDE as usual: 

    tile-eclipse --profile profile.xml

Metadata files needed by tile-op2xml

The tile-op2xml command requires a couple of metadata files in order to properly process your data file. If you're running tile-op2xml on x86, it will automatically locate these files either in your MDE installation directory or in the profile capture directory itself. You can optionally override the default locations for these files, but when running the tool on the host you will rarely need to do this. If you are running the tile-op2xml command natively, you'll need to obtain/generate these metadata files yourself and upload them to the TILE platform. This is not as bad as it sounds: these files are easily obtainable from your MDE installation, and they can also be uploaded once and reused as long as your program build does not substantially change.

First, the ProfileMetadata.xml file provides Tilera-specific metadata about the hardware events for the TILE platform. This file can be found in your MDE installation: 

    ${TILERA_ROOT}/etc/ProfileMetadata.xml

or 

    ${TILERA_ROOT}/tile/etc/ProfileMetadata.xml

Note: Make sure that you use the correct metadata file for your architecture; in other words, upload the metadata file from a Gx MDE installation when profiling on Gx hardware.

Second, the monitor.xml file provides Tilera-specific metadata about the specific hardware you're running on. This is automatically generated by tile-monitor's profile-capture command when tile-monitor creates a capture directory.

However, the tile-monitor command does include an option to generate this file on demand. It can be invoked as follows: 

    tile-monitor --dev your_tile_host --resume \
       --symbol-map host_path tile_path \
       ... any other symbol-file path mappings ... \
       --generate-monitor-xml monitor.xml \
       --quit

This connects to your hardware, without rebooting, generates the monitor.xml file, including the set of remote-to-local symbol file mappings you specify, and then exits.
