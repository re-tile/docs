Name

pgtable — /proc/PID/pgtable, /proc/kpgtable: page table special files for Tilera Linux

Description

The /proc/PID/pgtable family of special files allows users to see which virtual addresses are mapped to which physical addresses, and with what caching characteristics. This can be useful for diagnosing performance or correctness issues.

An application can open /proc/self/pgtable to view its own page table, or it can open /proc/513/pgtable (for example) to view process 513's page table. (Page tables for threads within a process all show the same information.) A separate /proc/kpgtable can also be examined by root to see the page table information that is specific to the kernel.

The pgtable files consist of one line per page; see the end of this man page for an example. The lines are broken down into fields as follows:

The virtual address (VA) of the start of the page.

The protection bits of the page. Normally these are like file protection bits, with "R" for read, "W" for write, and "X" for execute. The first two may appear in lowercase if a readable page has not yes been accessed or a writeable page has not yet been dirtied.

The physical address (PA) of the start of the page.

The Linux controller number (NUMA node) of the page. Note that this is also encoded in the high bits of the physical address.

The various page attributes of the page; these make up the rest of the line (see below).

There are many possible page attributes that may appear for a given page. They appear as a list of whitespace-separated tokens at the end of each line:

Huge - This attributes indicates that the page is a huge page (16MB) rather than a typical small page (64KB). Such pages allow for fewer TLB miss faults when accessing memory.

NoLocal - this page is not cached on the local cpu, but may be cached remotely; equivalent to "NoL1 NoL2" (see below).

NoL1 - this page is not cached in the local cpu's L1 data cache, but may be cached in the L2 or remotely.

NoL2 - this page is not cached in the local cpu's L2 cache, but may be cached in the L1 data cache or remotely.

CacheNone - this page is not cached in any cache. It is read and written directly to RAM.

Home=N - this page has its L3 (remote) cache on the specified cpu "N". This is typical for stack pages, for example, which are generally homed on the cpu where the task using that stack is executing.

HashHome - this page has its L3 (remote) cache distributed across the set of hash-for-home cpus. This is the default for most pages unless overridden.

NoHome - this page is not cached in the L3 (remote) cache, but brought in directly from memory. This is normal for a read-only page; for a read-write page it indicates an "incoherent" mapping that must be handled by explicit cache "flush" and "invalidate" directives.

NC - this page is non-coherent. This attribute is only used for pages that are mapped read-only or private, since they will not be modified and can therefore be cached more freely in all levels of the cache hierarchy without concern for coherence. Applies only to pages that are cached in the L3 (remote) cache.

Priority - this attribute indicates that the "priority" bit is set on a page table entry. These cache entries (also called "pinned") allow for partitioning of the cache.

Migrating - this attribute indicates that the "migrating" bit is set on a page table entry; that is, the page is being transitioned from one set of pages attributes to another. It is rarely seen.

NotPresent - this attribute indicates that the "present" bit is not set on a page table entry, and so is rarely seen.

Some page attributes are only seen in /proc/kpgtable:

Global - this attribute indicates that the page mapping is intended to be valid for all processes.

Kernel - this attribute indicates that the page mapping can only be accessed from kernel privilege level.

ZeroPage - this attribute marks the kernel's "zero page".

A page's attributes are normally set by default by the particular chip. For example, the TILEPro chip will normally use "hash for home" on all pages except the stack page. This can be overridden in various ways; see the Optimization Guide (UG105) for more information on memory allocation.
