Name

mcstat — Monitor Memory Controller Usage

Synopsis

mcstat --help

mcstat -t seconds

mcstat -i seconds

mcstat -w seconds

mcstat -c command [arg...]

Description

mcstat is a convenient way to monitor memory activity. Tile memory is accessed via up to 4 memory controllers. mcstat reports statistics for each memory controller over some period. The duration of this period is controlled by command line options. With -c the duration is the time it takes to run the given command. With the -i, -w, or -t options the duration is the given number of seconds. With -i or -w mcstat repeatedly monitors intervals of the given number of seconds. With -t mcstat measures just one such interval.

If an application's performance scales well and then flattens out, this is often reason to suspect that it has saturated one or more memory controllers. mcstat is useful for detecting situations where application performance is limited by memory bandwidth. On systems which are not configured to use memory striping, it's also useful for checking that memory controller usage is roughly evenly balanced. In the latter case, homing application data structures differently, or using different sets of tiles, may provide better performance; see numactl and taskset.

mcstat has two different output formats, depending on the command line options. With -i or -w it uses a condensed report format which allows it to fit all the information from a single interval onto a single line. In the other two modes, it uses a line per memory controller to make the output easier on the eyes.

Here is an example of the line-per-controller output format: 

    memory                               % read   % write
controller      % peak      % read         miss      miss    latency
--------------------------------------------------------------------
         0           3          66           36        78        115
         1           1          62            5        56        112
         2           2          18           22        81        115
         3           0          17           30        41         97

The meaning of the columns is:

memory controller

Index of the memory controller

% peak

Number of memory operations performed by the controller as a percentage of the theoretical peak. This theoretical peak is nearly impossible to achieve in practice. Even under the best conditions, the memory controller starts to be saturated (and thus a potential performance bottleneck) at 50%. Even with much lower numbers there may still be a memory bottleneck, depending on the pattern of references (see the descriptions of misses below.)

% read

Percentage of memory operations which are reads (as opposed to writes.)

% read miss

Percentage of read operations which page-missed in the controller as a percentage of all read operations. High miss rates result from memory reference patterns with low locality. The penalty for read misses is decreased memory bandwidth, but even with 100% miss rates, this will be about 80% of bandwidth with no misses at all. Often there is little one can do to improve miss rates.

% write miss

Percentage of write operations which page-missed in the controller as a percentage of all write operations. All the same considerations given above for read misses also apply to write misses, but the penalty is considerably higher.

latency

The average latency in core clock cycles of a load instruction that travels to the memory controller, as measured by the memprof driver. This measurement includes cycles spent traveling across the Tile Processor's mesh networks to and from the memory controller. As a result, controllers that are farther from the memprof dedicated or shared tile will have higher latency measurements, particularly in heavily loaded systems where memory network congestion can increase routing latency. Note that latency will appear more variable when using the shared tile configuration of the memprof driver because fewer samples are taken per measurement interval.

The condensed format used with the -i option looks like this: 

     mc 0      |     mc 1      |     mc 2      |      mc 3     |
 sat read  lat | sat read  lat | sat read  lat | sat read  lat |
   0   94  103 |   0   95  102 |   0   96   87 |   0   93   93 |
   0   98  105 |   0  100  105 |   0  100   87 |   0   97   88 |
   0   95  105 |   0   97  106 |   0   97   87 |   0   94   88 |
   0   98  105 |   0  100  105 |   0  100   87 |   0   97   88 |
   0   98  105 |   0  100  105 |   0  100   87 |   0   97   88 |

The semi-condensed format used with the -w option looks like this: 

         mc 0          |         mc 1          |         mc 2          |          mc 3         |
 sat read  rm  wm  lat | sat read  rm  wm  lat | sat read  rm  wm  lat | sat read  rm  wm  lat |
   0   97   9  71  103 |   0   99   1  47  112 |   0   93   1  85   88 |   0  100   0   0   97 |
   0   96   9  77  103 |   0   99   1  58  112 |   0   94   1  85   88 |   0  100   0   7   96 |
   0   97   9  75  103 |   0   99   1  13  111 |   0   98   1  64   88 |   0  100   0  13   96 |
   0   98   6  68  103 |   0  100   0   6  112 |   0   99   1  37   88 |   0  100   0   0   96 |
   0   97  11  72  103 |   0   99   1  26  112 |   0   97   1  40   88 |   0  100   0   0   96 |

In these formats there are up to 5 columns for each controller, and the correspondence to the less-compressed format is as follows:

sat
% peak (percent of theoretical peak throughput)
read
% read (percent of read operations, as opposed to write operations)
rm
% read miss (percent of read operations that miss an open bank)
wm
% write miss (percent of write operations that miss an open bank)
lat
latency (average load instruction latency)
