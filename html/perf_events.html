perf_events — Linux performance recording/reporting tool

Profiling applications with the Linux "perf" tool

Introduction

For profiling applications, the Tilera MDE provides a native port of the Linux perf profiling tool (aka "perf events" and formerly known as PCL, or Performance Counters for Linux). This is a Linux tool that collects data from performance counters, such as the platform's hardware counters, and provides report-generation and annotation support.

The perf tool

The perf tool is a native utility; it is run on the TILE host alongside the profiled application.

If you're used to using tile-monitor to run your application, you can view the "perf" tool as just another Linux command that is invoked using tile-monitor's run or launch commands: 

$ tile-monitor --dev wc-5 --resume
...
Command: run perf record fork 3 1000
...

If you're using tile-console or a similar shell environment, you can run perf commands directly: 

$ tile-console --dev wc-5
...
# perf record fork 3 1000
...

The perf Tool's Commands

The perf tool provides a suite of profiling commands, accessed by specifying a command name as the tool's first argument. These commands include:

  • perf list -- list the available performance counters that can be passed to perf record.
  • perf record -- runs a specified Linux command with performance data capturing enabled, generating a "perf.data" raw data file.
  • perf report -- reads a "perf.data" file and produces some basic textual reports, with a number of filtering and display options.
  • perf annotate -- reads a "perf.data" file and generates a copy of the program sources annotated with performance data.
  • perf diff -- reads two "perf.data" files and displays the differences between them.
  • perf help -- displays a summary of the available commands.

We'll describe some of the more useful perf commands below. For more detail, refer to the perf command's man page.

The perf list Command

perf list

The perf list command lists the available performance counters that can be used in the perf record command. These include a number of "generic" counters, with lower-case names, which are mapped internally to the corresponding hardware or software events. Also listed are hardware events which can be specified. These typically have upper-case names.

The perf record Command

perf record {options} {-e EVENT_NAME | --event=EVENTNAME} ... {--} command arguments...

    -o, --output <file>     output file name (optional, defaults to "perf.data")

    -e EVENT1,EVENT2,EVENT3,EVENT4
        --event=EVENT1,EVENT2,EVENT3,EVENT4
                            event(s) to sample

    -C, --cpu cpu_list      record data only for the specified list of cpu ID(s)
                            (the list is comma separated and can include ranges,
                             for example: --cpu 0,2-14 )

    -c, --count <n>         event interval to sample (for all events)

    -g, --call-graph        collect call-graph (stack chain/backtrace) information

    -a                      record data for all processes (a.k.a. system-wide profiling)

    --                      terminates option processing

The perf record command captures performance data during the execution of a specified command. This command may fork processes or spawn threads. The performance data of the initial command plus any processes/threads it creates are captured in the resulting "perf.data" file. (One can filter the results when generating a report to remove OS and other processes that are not of interest.)

For example, the following command: 

# perf record -o samples.data fork 3 1000

Runs the program fork passing it the arguments 3 and 1000, and recording performance data in a file named "samples.data" in the current directory.

If the -o option is not used to specify an output file, the default output location is the file "perf.data" in the current directory. If the output file already exists, it is renamed with an ".old" suffix.

One can specify event name(s) to profile as a comma-separated list, as shown below, up to the maximum supported by the hardware. (By default, the number of cycles, aka the "ONE" event, is sampled.) One can also specify an event interval, using the -c. Note that currently the event interval applies to all events. 

# perf record -o samples.data -e ONE,INSTRUCTION_STALL,LOAD_STALL,READ -c 12000000 -- ./fork 3 1000

One can also specify the -g option to request per-thread profiling data. It's necessary to specify this option to perf record if you want to view the results with the corresponding -g option to perf report. If the -g option is not specified, performance data is aggregated at the process level. 

# perf record -o samples.data -g -- ./fork 3 1000

For cases where one needs to profile a long-running executable, or one that has a period of "set up" work before it enters the steady-state that one wants to profile, or where it's simply desirable to profile everything in the system to see what's most active, one can use the -a option to request "system-wide" profiling. This will capture profiling data for all processes running while sampling is being done.

The best way to use this option is to profile the "sleep" command running for the desired duration. This will essentially ignore the sleeping process, and capture profiling data for other processes, including the one you're interested in. For example: 

# start_my_long_running_app
... wait for app to reach "steady state" ...
# perf record -o samples.data -a -g -- sleep 30
... wait for 30 seconds, until perf reports completion ...
... kill the long-running app ...

Note: there's no way to exclude OS processes, etc. that you're not interested in while recording, but you can always filter that information out when viewing the resulting profile data.

The perf report Command


perf report {options} {module_name}

    -i, --input <file>    input file name

    -s, --sort <key[,key2...]>
                          sort by key(s): pid, comm, dso, symbol

    -d, --dsos <dso[,dso...]>
                          only consider symbols in these dsos
    -C, --comms <comm[,comm...]>
                          only consider symbols in these comms
    -S, --symbols <symbol[,symbol...]>
                          only consider these symbols

    -n, --show-nr-samples Show a column with the number of samples

    -g, --call-graph
                          Display call-graph information.

The perf report command is used to view the raw data file generated by the perf record command. By default, it assumes the data is stored in a file named "perf.data" in the current directory, but one can use the -i to specify a different name/location.

By default, the output is aggregated by executable and symbol name: 

# perf record fork 3 1000

# perf report

# Overhead  Command      Shared Object              Symbol
# ........  .......  .................  ..................
#
    99.22%     fork  fork               [.] e
     0.78%     fork  [kernel.kallsyms]  [k] __vm_enough_memory

You can use the -sort to request that the output be broken out by Linux task (aka "pid"), executable (aka "dso"), command (aka "comm"), and/or function name (aka "symbol"). The -sort option also specifies the display order of the columns.

Note that adding the pid breaks out the data by process/thread ID. In other words, the -sort option does more than just specify the columns to display/sort by. Also, sorting by symbol may also cause the perf tool to look up symbols (such as OS symbols) it would otherwise display as hex values. For this reason, it's _strongly_ recommended that you explicitly specify at least -sort pid,dso,symbol when using perf report. Otherwise, you may not get all of the profiling information you expect in the output.) 

# perf report --sort pid,dso,comm,symbol

# Overhead  Command:  Pid      Shared Object  Command              Symbol
# ........  .............  .................  .......  ..................
#
    58.14%           fork:  828  fork                  fork  [.] e
    20.93%           fork:  830  fork                  fork  [.] e
    20.16%           fork:  829  fork                  fork  [.] e
     0.78%           fork:  827  [kernel.kallsyms]     fork  [k] __vm_enough_memory

You can then use options like -d, -C, and -S to filter the output to specific executables, commands, and/or symbols: 

# perf report --sort pid,dso,comm,symbol -d fork -S e

# dso: fork
# symbol: e
# Events: 128  cycles
#
# Overhead  Command:  Pid  Command
# ........  .............  .......
#
    58.59%           fork:  828     fork
    21.09%           fork:  830     fork
    20.31%           fork:  829     fork

There is no option in perf report to filter by CPU ID; however, you can use the -C / --cpu option of the perf record command to "pre-filter" the data, so to speak, to only include the CPUs you're interested in.

The -n option adds a column to the output showing the actual number of sample counts, in addition to the percentage value: 

# perf report --sort pid,dso,comm,symbol -n -d fork

# Overhead  Samples    Command:  Pid  Command  Symbol
# ........ ..........  .............  .......  ......
#
    58.59%         75           fork:  828     fork  [.] e
    21.09%         27           fork:  830     fork  [.] e
    20.31%         26           fork:  829     fork  [.] e

The -g displays call-graph information.

The perf annotate Command

perf annotate {options}

    -i, --input <file>    input file name

    -d, --dsos <dso[,dso...]>
                          only consider symbols in these dsos

    -P, --full-paths      Don't shorten the displayed pathnames

    -l, --print-line      print matching source lines (may be slow)

The perf annotate command is used to produce a disassembly for a specified command, annotated with source lines and performance data from a specified raw perf.data file. By default, the command assumes the data is stored in a file named "perf.data" in the current directory, but one can use the -i to specify a different name/location.

The -d option can be used to limit the output to a specific binary file. The -P option requests that full pathnames be displayed for the file(s) annotated.

The perf diff Command

perf diff {options} old.data new.data

    -s, --sort <key[,key2...]>
                          sort by key(s): pid, comm, dso, symbol

    -d, --dsos <dso[,dso...]>
                          only consider symbols in these dsos
    -C, --comms <comm[,comm...]>
                          only consider symbols in these comms
    -S, --symbols <symbol[,symbol...]>
                          only consider these symbols

The perf diff command compares two output files for the same command and reports on the difference in samples seen in the one versus the other.

The , , , and options select and filter the output in the same way as they do for perf report. 

# perf record -o sample1.data -c 100000 ./fork 3 1000

# perf report -n --sort pid,dso,comm,symbol -d fork -i sample1.data

# Overhead  Samples    Command:  Pid  Command  Symbol
# ........ ..........  .............  .......  ......
#
    59.52%         75           fork:  780     fork  [.] e
    20.63%         26           fork:  781     fork  [.] e
    19.84%         25           fork:  779     fork  [.] e


# perf record -o sample2.data -c 100000 ./fork 3 6000

# perf report -n --sort pid,dso,comm,symbol -d fork -i sample2.data

# Overhead  Samples    Command:  Pid  Command  Symbol
# ........ ..........  .............  .......  ......
#
    50.29%         86           fork:  807     fork  [.] e
    25.15%         43           fork:  806     fork  [.] e
    24.56%         42           fork:  808     fork  [.] e


# perf diff --sort pid,dso,comm,symbol -d fork sample1.data sample2.data

# Baseline  Delta      Command:  Pid  Command  Symbol
# ........ ..........  .............  .......  ......
#
     0.00%    +50.29%           fork:  807     fork  [.] e
     0.00%    +25.15%           fork:  806     fork  [.] e
     0.00%    +24.56%           fork:  808     fork  [.] e
