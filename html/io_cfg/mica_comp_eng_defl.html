    Register Definitions



    Deflate Engine Configure Control (MICA_COMP_ENG_DEFL_REG_DEF_CTL: 0x0000)

    Deflate Engine Configure Control.

    Note that the DEF_CTL register has same fields as the DM_SPECIFIC in the user space MiCA OPCODE (CTX_USER_OPCODE.DM_SPECIFIC), except the CONFIG field. Refer to the User Context registers for more details.




    Bits Name Type Reset Description
    9 CHINT RW 0 A compression hint indicating whether the input data is compressable. For highly compressable packets, compression engine can choose a different algorithm for pattern matching and for Huffman code selection. User application can choose, for example, 8:1 compression ratio, as the threshold for high compressability.
    ...0 = Normal compressability
    ...1 = High compressability
    If the hint is configured incorrectly, it will have an impact on the compression ratio/bit rate, but the functional correctness will not be compromised.
    8 SHINT RW 0 An input size hint indicating the size of the input file. For small packets, compression engine can choose a different algorithm for pattern matching and for Huffman code selection. User application can choose, for example, less than 2 k bytes, as the threshold for small packets.
    ...0 = Normal packet size
    ...1 = Small packet size
    If the hint is configured incorrectly, it will have an impact on the compression ratio/bit rate, but the functional correctness will not be compromised.
    7 BHINT RW 0 An input type hint. Setting this bit to 1 indicates that the hardware should treat the input as binary data rather than ASCII test. The compression engine may choose a different algorithm for pattern matching and for Huffman code selection.
    ...0 = ASCII input
    ...1 = Binary input
    If the hint is configured incorrectly, it will have an impact on the compression ratio/bit rate, but the functional correctness will not be compromised.
    6 DTREE RW 0 Which type of Huffman codes to use in compression. Dynamic Huffman code will typically improve the compression ratio. It has a minor impact on the compression speed and a minor to large impact on the decompression speed (depending on the implementation of the decompression engine).
    ...0 = Fixed Huffman codes
    ...1 = Dynamic Huffman codes
    5:2 MAX_CHAIN_LENGTH RW 3 The number of iterations that the hardware should search the history buffer. Setting this parameter higher will severely impact compression speed.
    ...0 = 1
    ...1 = 2
    ...2 = 3
    ...3 = 4
    ...4 = 6
    ...5 = 8
    ...6 = 16
    ...7 = 32
    ...others = reserved
    1 FORMAT RW 0 The compression format
    ...0 = compress data into raw deflate format
    ...1 = compress data into GZIP format




    Match control (MICA_COMP_ENG_DEFL_REG_MATCH_CTL: 0x0040)

    Match control.

    Bits Name Type Reset Description
    8:7 EFFORTS RW 3 Once the pattern matching logic finds a long string that is greater than NICE_MATCH, the compression engine will stop searching for additional pattern matches. Once the pattern matching logic finds a good string that is greater than GOOD_MATCH, the compression engine will reduce efforts to search for additional pattern matches.
    0: NICE_MATCH=24, GOOD_MATCH=10
    1: NICE_MATCH=24, GOOD_MATCH=16
    2: NICE_MATCH=32, GOOD_MATCH=24
    3: NICE_MATCH=32, GOOD_MATCH=32
    6:5 MAX_DIST RW 0 Maximum match distance. The compression engine will not search beyond the MAX_DIST of a sliding window of the history. With a larger MAX_DIST, the compression engine will search harder and typically results in a higher compression ratio and a slower compression speed.
    0: 8 k
    1: 4 k
    2: 2 k
    3: 1 k
    1 MATCH1_RATIO RW 0 The speculative match engine can be configured to optimized for speed or for compression ratio.
    0: for speed
    1: for compression ratio
    0 MATCH0_RATIO RW 0 The primary match engine can be configured to optimize for speed or for compression ratio.
    0: for speed
    1: for compression ratio




    DMA Engine Selection (MICA_COMP_ENG_DEFL_REG_DMA_SEL: 0x0080)

    Either Inflate Engine or Deflate Engine can be selected to support memory copy (ENGINE_TYPE = 0 or 1).
    Bits Name Type Reset Description
    0 MC RW 0 1: deflate engine supports memory copy (ENGINE_TYPE=0/1). 0: inflate engine supports memory copy (ENGINE_TYPE=0/1).




    Dynamic Huffman code customization control (MICA_COMP_ENG_DEFL_REG_CUST_CTL: 0x00c0)

    Dynamic Huffman code customization control.
    Bits Name Type Reset Description
    2:1 CUST_BTYPE RW 2 This field is valid only when CUST_CODE=1. This field specifies the BTYPE when custom code is implemented.
    0 CUST_CODE RW 0 This field is valid only if dynamic Huffman codes are used. If the data is compressed with dynamic Huffman codes, the code representation should be part of the a deflate-format block. Custom implementation could choose to skip the code representation.
    0: send code representation in a deflate block with dynamic Huffman codes. This is the typical configuration.
    1: do not send code representation in a deflate block with dynamic Huffman codes.




    Performance Counter Control 0 (MICA_COMP_ENG_DEFL_REG_PERF_CTL_0: 0x0800)

    Non-saturating performance counters are supported. Upon counter value wraps around, a corresponding interrupt can be generated. The performance counters are read to clear, and can be frozen by writing 0 to the ENA.

    Bits Name Type Reset Description
    1 RTC RW 0 1: performance counters are read to clear.
    0 ENA RW 1 1: enable all performance counters. when ENA=0, freeze. when ENA goes to 1 (from 0), reset and enabled. when ENA=1, enabled.




    Performance Counter Control 1 (MICA_COMP_ENG_DEFL_REG_PERF_CTL_1: 0x0840)

    Performance Counter Control 1

    MODEx: performance counter mode selection for PERF_CNT_x

    0: event count - number of memory copy (ENGINE_TYPE=0/1) started
    1: event count - number of transaction (ENGINE_TYPE=2) processed
    2: event count - number of transaction (ENGINE_TYPE=2) started
    3: event count - number of GZIP transaction started
    4: event count - number of match results processed
    5: event count - number of match results (that has one literal) processed
    6: event count - number of match results (that has two literals) processed
    7: event count - number of match results (that has match length = 3) processed
    8: event count - number of match results (that has match length = 4 to 7, inclusive) processed
    9: event count - number of match results (that has match length = 8 to 15, inclusive) processed
    10: event count - number of match results (that has match length = 16 or more) processed
    12: event count - number of cycles that the deflate engine is busy working on the transaction
    13: byte count - number of memory copy bytes (ENGINE_TYPE=0/1) started
    14: byte count - number of uncompressed input bytes (ENGINE_TYPE=2) started
    15: byte count - number of compressed output bytes (ENGINE_TYPE=2) processed

    If there is no in-flight transaction,
    counter(mode=1) = counter(mode=2)

    number of transactions in raw deflate format = counter(mode=2) - counter(mode=3)

    counter(mode=4) may not be equal to sum of counter(mode=5) + counter(mode=6) + counter(mode=7) + counter(mode=8) + counter(mode=9) + counter(mode=10) in that one match result may have both literal and match length

    51:48 MODE6 RW 14 Performance counter mode selection for PERF_CNT_6 (same encoding as MODE0).

    Interrupt vector, write-one-to-clear (MICA_COMP_ENG_DEFL_REG_INT_VEC_W1TC: 0x1000)
