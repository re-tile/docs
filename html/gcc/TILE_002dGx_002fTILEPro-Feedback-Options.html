3.17.38 TILE-Gx/TILEPro Feedback Options

Feedback-based optimization (also known as profile-guided optimizations) means using run-time data from a "training run" to make better optimization decisions. With this data, the compiler will know how often functions are called to guide inlining, how often loops are executed to guide unrolling and other loop optimizations, and which blocks of code are executed more often to guide code scheduling, layout, and register allocation decisions. Performance gains can be quite large, 20% or more, at the expense of a more complex build process.

Gcc provides feedback-based optimizations through the -fprofile-generate and -fprofile-use flags. Both TILE-Gx and TILEPro ports of gcc support these flags, but they also provides similar functionality through the -ffeedback-prefixed group of flags. There are two differences between the two. First, the fprofile options use multiple files, one per object file, to collect run-time data, while the ffeedback options uses a single file. Second, the fprofile options only use the data for compile-time optimization, while the ffeedback options use the data for both compile-time optimization and cache packing optimization at link-time. Note that the optimizations done at compile-time, as well as the raw data collected for compile-time use, are exactly the same between the two sets of options.

This section describes how to use the ffeedback-prefixed options. The ffeedback workflow supports two sets of feedback optimizations, one for the compiler, and one for the linker. The compiler based feedback uses the feedback data to guide compile-time decisions on inlining, code layout, unrolling, and creating hot/cold sections. The linker based feedback optimization is Tilera-specific. It uses feedback data to better assign functions to physical addresses to reduce instruction cache conflict misses.

Feedback-based optimization is a multi-step process. In general, one must:

  1. Compile using -ffeedback-generate, -ffeedback-generate-compiler, or -ffeedback-generate-linker, which will create an instrumented application that will create a feedback file.
  2. Link with -ffeedback-emit to indicate where to write the linker feedback data.
  3. Run the instrumented application with one or more representative test inputs.
  4. Convert and combine the raw feedback data emitted above using the tile-convert-feedback program.
  5. Recompile with -ffeedback-use, -ffeedback-use-compiler, or -ffeedback-use-linker, which will optimize using the feedback file(s) generated in the previous step. It's also helpful to specify -freorder-blocks-and-partition, which will separate each function into "hot" and "cold" portions. By collecting all the "hot" portions together, instruction memory locality is improved.

    When -ffeedback-use or -ffeedback-use-linker is passed to the link step, the compiler uses the feedback data to select function addresses that improve cache locality. For this to work effectively, each function must be in its own section, which the compiler will do implicitly by enabling -ffunction-sections. The -static flag also gives the link optimizer freedom to choose optimized addresses for system library code, which can further improve performance.

In summary, the full feedback-based optimization workflow using the cross-tools is:

  • Compile instrumented object files: 
              tile-gcc -O2 -ffeedback-generate -c *.c
  • Link instrumented object files: 
              tile-gcc -O2 -ffeedback-generate -ffeedback-emit=/tmp/raw_feedback \
            -static *.o -o my_app
  • Discard old feedback data and run instrumented application: 
              rm -rf *.gcda raw_feedback
          tile-monitor --pci --batch-mode --here --upload my_app /my_app \
            --run - /my_app my_arg1 my_arg2 - --download /tmp/raw_feedback raw_feedback
  • Convert and combine raw feedback data into a single feedback file: 
              tile-convert-feedback -o my_app.feedback raw_feedback
  • Compile feedback-optimized objects: 
              tile-gcc -O2 -ffeedback-use=my_app.feedback -freorder-blocks-and-partition -c *.c
  • Link feedback-optimized objects: 
              tile-gcc -O2 -ffeedback-use=my_app.feedback -static *.o -o my_optimized_app

Note that the profile data degrades gracefully, remaining mostly valid if only small application changes are made. As individual functions are modified they lose the benefits of feedback optimization but others do not. Thus not all the above steps need to be executed for every build during development to realize most of the benefits.

Here is a list of the feedback-related options:

-fcoverage-permissive
Inform but don't fail on missing or mismatch coverage counters.
-ffeedback-generate
Inserts extra instrumentation code that collects information for later feedback-directed optimization via the -ffeedback-use option. -ffeedback-generate is used when compiling individual source files. It must be combined with the -ffeedback-emit option at the link step to specify where the raw feedback should be written.
-ffeedback-generate-compiler
Similar to -ffeedback-generate, but only instruments code to collect data for compiler feedback-directed optimization via -ffeedback-use-compiler.
-ffeedback-generate-linker
Similar to -ffeedback-generate, but only instruments code to collect data for linker feedback-directed optimization via -ffeedback-use-linker.
-ffeedback-emit=directory
Specifies a directory name into which the raw feedback data will be emitted. Each instrumented process will create a unique file in the specified directory. The instrumented binary will create this directory at runtime if it does not already exist.

The tile-convert-feedback program converts the contents of a raw feedback directory, which may contain data from many different processes, into a single feedback file suitable for use with -ffeedback-use.

-ffeedback-use=file
Perform both compiler and linker feedback-based optimization using the runtime profile data in the specified file produced by tile-convert-feedback.
-ffeedback-use-compiler=file
Perform compiler feedback-based optimization using the runtime profile data in the specified file produced by tile-convert-feedback.
-ffeedback-use-linker=file
Perform linker feedback-based optimization using the runtime profile data in the specified file produced by tile-convert-feedback.
-fprofile-correction
Enable correction of flow inconsistent profile data input. This option exists on all gcc ports, but it's enabled by default on TILE-Gx/TILEPro.
-fstrip-file-prefix=path
Strip the prefix path off of file names for feedback use.
-Wcoverage
Report any inconsistency involving coverage counters.
