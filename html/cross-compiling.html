Compiling for the TILE Architecture

This application note describes the processing of building applications for the TILE architecture, especially open source packages using the standard configure mechanism.

Overview

Conventional native compilation means building an application on the same platform on which it will run. For example, in the usual process of building and running a program on Linux, the compiler and the program both run on the same Linux system: 

$ cc -o hello hello.c
$ ./hello
"Hello, world!"

Cross compilation, in comparison, means building on one platform (called the build platform), and running the resulting program on another platform (usually called the target, though configure confusingly calls it the host platform). The appropriate compiler must be used: one that runs on the build platform and is configured to produce code for the target platform. And the resulting program must be transferred to the target platform and executed there. For instance: 

$ tile-cc -o hello hello.c
$ tile-monitor --pci --upload hello /tmp/hello -- /tmp/hello
"Hello, world!"

Native compilation is the norm on "full" distributions such as a typical Linux workstation, while cross compilation is the norm for embedded systems. The Tilera platform has elements of both, and supports both modes. Some pros and cons:

  • Native compilation is typically more straightforward, and some packages are difficult to cross-compile, since their build systems are not set up for it.
  • Native compilation requires all the build tools to be ported to the build system, so build systems requiring non-standard build tools may not work out of the box.
  • Cross compilation allows running the tools on the familiar development host platform, with all the resources of the host, including more direct access to the file systems.

Native Compilation

In general, we recommend trying native compilation first. One general technique is to exploit tile-monitor's ability to mount host filesystems or copy files to and from the TILE target, for two purposes:

  • To make available the extended tools (such as native compilers and full set of libraries) which are not part of the standard RAM-based distribution. Use the --root command to mount the necessary directories from the the MDE's tile directory (/usr, /lib, etc.).
  • To make available the host working directories. A very convenient technique is to start tile-monitor in the package build directory, and use the --here command to mount it on the TILE processor. One can also use --mount-same to make other host directories available, such as explicit source and build directories. Or use --upload to copy your source files to the TILE processor, and --download to copy important files from the build directory back to the host. (The "upload" technique takes more time initially, but may be faster overall.)

With these mounts or copies in place, the build environment is essentially identical to building on the host, and one can run commands like configure and make with build results being written under the current working directory on the host.

Another general technique is to build a disk-based server, unpacking the "tile" tarball to create persistent disk image (see the Getting Started guide, UG204.) This server can be used just like any general-purpose Linux server, using ssh to log in, scp or NFS to transfer files, etc.

Example: MySQL

MySQL is a good example of a package that is quite difficult to cross compile, since its build system is not set up for it. See the following sections for more information on make and configure.

For instance, assuming you have extracted the mysql package into /u/user/mysql-5.0.45, and wish to build it in place: 

$ cd /u/user/mysql-5.0.45
$ tile-monitor --pci --root --here
Command: run configure --prefix=/u/user/mysql-5.0.45/install --build=tilegx-unknown-linux-gnu
Command: run make
Command: run make install

In a host makefile or script, one could replace the interactive tile-monitor commands above with tile-monitor command-line options. One can use build parallelism options such as make -j n to speed up the build, but watch out for two caveats:

  • If the TILE system is just running with a RAM filesystem, there is in general no swapping support, so excess consumption of memory from many parallel build processes may crash the system.
  • If file systems are mounted, build performance is often gated by host file access, so parallel speedup may be limited.

Makefiles

Of course in general applications are not simply built on the command-line; the build process is automated using the make command, with the build rules specified in one or more makefiles. A trivial example is: 

# Sample hello Makefile

hello: hello.c
	$(CC) -o $@ hello.c

clean:
	rm hello

Note that the compiler is specified with the variable $(CC). By default, $(CC) will evaluate to the native compiler cc. But we can override it with a cross-compiler as follows: 

$ make clean hello CC=tile-gcc
rm hello
tile-gcc -o hello hello.c

Makefiles may also invoke other tools, such as $(LD) (the linker) and $(AS) (the assembler) which can be overridden with tile-ld and tile-as, respectively.

For cross compilation, if a Makefile invokes tools such as cc or gcc directly, such references should be replaced with variables as above.

Configure

More complex open-source packages typically use the tool configure to create a custom Makefile on a particular system. By default configure assumes native compilation, and will automatically probe the build system to determine the tools and libraries available, among other properties, in order to create the appropriate Makefile.

Run configure --help to see all the options. For native builds, it may be necessary to specify --build=tilegx-unknown-linux-gnu.

For cross compilation, the most important option is --host=HOST. The HOST string is looked up in the config.sub file to set some other parameters, such as the host operating system and machine architecture.

We wish to use HOST=tilegx-unknown-linux-gnu, but the tile architecture is not yet standard, so may not exist in the config.sub file. If necessary, copy the customized config.sub and config.guess files from the the MDE's distribution's etc/ directory, for instance with: 

    for name in config.guess config.sub; do
        for path in `find . -name $name`; do
            cp -f $TILERA_ROOT/etc/$name $path
        done
    done

Then we can run ./configure --host=tilegx-unknown-linux-gnu. If the MDE tools are in the executable search path, then configure will find tile-gcc, and incorporate it as the value for $(CC) in the generated Makefile.

While make will create the package, in general one wants to run make install to actually create a distribution. By default, the generated Makefile will attempt to install on the build system, so use the configure --prefix option to specify a path prefix for the distribution to be built.

Example 1: PCRE

A simple example is PCRE, version 7.4. For simplicity, we also specify --disable-cpp to omit the C++ wrappers. (PCRE comes with a good README file that explains options such as this one.) 

$ ./configure --host=tilegx-unknown-linux-gnu --disable-cpp
$ make

Example 2: libpcap

In general, the above steps are not quite sufficient.

For example, let's consider a fairly simple package, the packet sniffer library libpcap, version 0.9.8. if we go through the above steps, configure will find tile-gcc and so on, and then fail with: 

configure: error: pcap type not determined when cross-compiling; use --with-pcap=...

The --with-pcap parameter is specific to libpcap, and specifies the type of network interface. The problem is that by default, libpcap's configure will probe the build system for the type of network interface, but it cannot do that for cross-compilation, since it does not have access to the host system, and thus requires this parameter to be set by the user. Let's use linux to specify using the socket interface: 

$ ./configure --host=tilegx-unknown-linux-gnu --with-pcap=linux
        :
checking Linux kernel version... unknown
configure: error: cannot determine linux version when cross-compiling

Now what? This is a similar problem: normally configure gets the Linux version from the build system, but with cross-compilation, does not have access to the host system. Again, the solution is to supply the parameter explicitly. In this case, by inspection of the configure script, we need to set a variable called ac_cv_linux_vers: 

$ ./configure --host=tilegx-unknown-linux-gnu --with-pcap=linux ac_cv_linux_vers=2.6.15
        :
config.status: creating Makefile
        :

This time it worked, and we can build: 

$ make clean
$ make
tile-gcc -O -fPIC -I.  -DHAVE_CONFIG_H  -D_U_="__attribute__((unused))" -c ./pcap-linux.c
        :
tile-ranlib libpcap.a

Example 3: openssh

Another complication is configure scripts that attempt to run programs on the host system as part of the build process. With native compilation, this is not a problem, since the host and build systems are the same. But with cross-compilation, we must run those programs on the host system.

For instance, with openssh, version 4.4p1, we first need to have an openssl build (see below), and then we can run configure with say: 

./configure --host=tilegx-unknown-linux-gnu --with-ssl-dir=/u/user/openssl-0.9.8g/install --prefix=/u/user/openssh-4.4p1/install

Building, we get: 

$ make install
        :
/bin/sh: line 4: ./ssh-keygen: cannot execute binary file
/bin/sh: line 9: ./ssh-keygen: cannot execute binary file
/bin/sh: line 14: ./ssh-keygen: cannot execute binary file
make: *** [host-key] Error 126

Examining the Makefile, particularly the host-key rule, we see it is running the built ssh_host_key program. So we need to run this command on the host. The rule immediately below, host-key-force is a simpler version that when run will conveniently print out the commands, e.g.: 

$ make host-key-force
./ssh-keygen -t rsa1 -f /u/user/openssh-4.4p1/install/etc/ssh_host_key -N ""

We can run this on the host via tile-monitor as follows: 

~/p4/build/install/bin/tile-monitor --pci \
  --upload ./ssh-keygen /tmp/ssh-keygen \
  --when-idle --download /tmp/ssh-host-key \
    /u/user/openssh-4.4p1/install/etc/ssh_host_key \
  --download /tmp/ssh-host-key.pub \
    /u/user/openssh-4.4p1/install/etc/ssh_host_key.pub \
  -- /tmp/ssh-keygen -t rsa1 -f /tmp/ssh-host-key -N ""

This somewhat complex command will upload the executable, run it, and then download the two generated files to the appropriate destination. The output is: 

Generating public/private rsa1 key pair.
Your identification has been saved in /tmp/ssh-host-key.
Your public key has been saved in /tmp/ssh-host-key.pub.
The key fingerprint is:
58:99:b4:5f:0a:19:ca:fa:8d:46:bd:73:ae:cc:e5:91 root@localhost

We can modify the two other commands in the host-key-force rule similarly.

Now make install will succeed, since the generated key files have been created. There is one additional problem: 

$ make install
/u/user/openssh-4.4p1/install/sbin/sshd -t -f /u/user/openssh-4.4p1/install/etc/sshd_config
/u/user/openssh-4.4p1/install/sbin/sshd: /u/user/openssh-4.4p1/install/sbin/sshd: cannot execute binary file
make: [check-config] Error 126 (ignored)

The check_config rule does not produce any files, it merely runs sshd to check the sshd_configfile. We can verify this manually by running another tile-monitor command to run sshd on the Tile host, and then comment out the check_config rule to allow the make to succeed.

Example 4: texinfo

Some packages attempt to resolve the problem of building build-time tools by creating a parallel directory for building native build tools. An example is texinfo, version 4.11. Indeed it gets it almost right, with just one apparent bug in a Makefile.

A specific wrinkle with texinfo is that the config.sub file is not in the usual place, but rather in build-aux/config.sub; this is pointed out by running configure as usual: 

$ ./configure --host=tilegx-unknown-linux-gnu --prefix=/u/user/texinfo-4.11/install
         :
checking host system type... Invalid configuration `tile': machine `tile' not recognized
configure: error: /bin/sh build-aux/config.sub tile failed

Once we edit the build-aux/config.sub file to add the tile block as usual, configure succeeds.

Unfortunately, building fails: 

$ make
        :
make[4]: *** No rule to make target `../gnulib/lib/libgnu.a', needed by `makedoc'.  Stop.
make[4]: Leaving directory `/u/user/texinfo-4.11/tools/info'

The problem is that we need to build in the tools/gnulib/lib directory before building in the tools/info directory. tools/Makefile specifies: 

#SUBDIRS = $(native_tools) gnulib/lib lib info makeinfo po util doc
SUBDIRS = lib info makeinfo util

So it seems reasonable to just reverse the commenting. In this makefile, $(native_tools) expands to blank, and this will recurse into gnulib/lib before lib. (It's quite possible we don't really need the additional directories such as makeinfo.)

So trying again, we get a lot further, building the native tools, and invoking tile-gcc to build host tools. Finally there is an error due to a dependence on the curses library: 

$ make
        :
terminal.o: In function `terminal_initialize_terminal':
/u/user/texinfo-4.11/info/terminal.c:617: undefined reference to `tgetflag'
make[3]: *** [ginfo] Error 2
make[3]: Leaving directory `/u/user/texinfo-4.11/info'

The solution is to build curses (in this case using ncurses-5.6) and then re-configuring to pick up that dependence. There do not appear to be specific options to specify locations for libraries, so the generic variables are used to specify include and link paths: 

./configure --host=tilegx-unknown-linux-gnu --prefix=/u/user/texinfo-4.11/install \
  LDFLAGS=-L/u/user/ncurses-5.6/install/lib/ \
  CPPFLAGS=-I/u/user/ncurses-5.6/install/include/ncurses/

At this point, the build succeeds: 

$ make
$ make install

Other Build Systems

Not all packages use configure, or at least the standard version generated by Autoconf. For instance, openssl (at least version 0.9.8g) uses a script called Configure (note capital 'C') that does not use a config.sub file, but rather has a set of host configurations embedded in it.

The workflow for native builds is to use the ./config script to identify the build system, and run ./Configure using that as the host system. That will not work for cross-compiling.

A simple solution is to use the generic cc configuration, and then override the compiler explicitly: 

$ ./Configure cc --prefix=/u/user/openssl-0.9.8g/install
$ make CC=tile-gcc
$ make CC=tile-gcc install

For this simple package, this appears to be sufficient. In practice, one should probably add a tile platform line to Configure, specifying better optimization options and more specific pre-processor symbols.
