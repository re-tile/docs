Name

tile-mpipe-cc — C Compiler for TILE-Gx Classifier Processor

Synopsis

tile-mpipe-cc [--help] [-S] [-E] [-s] [-Idir...] [-Dmacro[=defn]...] [-Umacro...] -o outfile input_file

Description

tile-mpipe-cc compiles programs written in a small subset of the C language to run on the classifier hardware.

Options

--help

Print usage message and exit.

-o output_file

output_file is required. It is the filename of the output written by tile-mpipe-cc.

input_file

input_file is required. It is the filename of the classifier program source.

-E

Pass input_file through the C preprocessor and exit without compiling.

-S

Compile input_file into assembler source code but exit without invoking tile-mpipe-as.

-Idir

Add the directory dir to the list of directories to be searched for header files.

-Dmacro=definition

The contents of definition are processed as if they appeared in a #define directive.

-Umacro

Cancel any previous definition of name, either built in or provided with a -D option.

-s

Enable "small switch" mode. This mode implements switch statements by iterating through a table in table memory. This saves instruction memory at the expensive of speed and data memory. Use this if you are running out of instruction memory but have cycles to burn.

Writing Classifier Programs

The classifier program reads from an input stream of bytes providing the packet's header, analyzes that header, and writes a "packet descriptor" to an output stream. The packet descriptor controls how later hardware stages process the packet.

[ Note: for complete details of the Classifier, please see the Tile Processor I/O Device Guide for the TILE-Gx Family of Processors. ]

tile-mpipe-cc supports:

  • C preprocessor (#define, #if, #include, // and /**/ comments, etc.)

  • Standard C control flow: while, for, do/while, if, else, switch, break, continue, and return.

  • Local scalar variables. They must be of type uint8_t, int16_t or uint16_t. All arithmetic operations are 16 bit.

  • C99-style declarations. Variables can be declared anywhere in a program block, not just at the beginning. They can also be declared inside "for" loop initializers.

  • Numeric constants. Constants can be written in decimal, or in hexadecimal with a leading "0x". An optional "U" suffix indicates that the value should be uint16_t, not int16_t.

  • Casting. You can cast to any of int16_t, uint16_t, or void.

  • Arithmetic. All of the standard C operators are supported, except division and remainder.

  • Comparison.

  • Short-circuiting logical operators.

  • gcc-style "statement expressions", such as ({ int x = 3; x + 2; })

  • Assorted intrinsics for accessing hardware resources (see below).

  • Globals arrays of integer types. Since there is no writable memory, all arrays are implicitly 'const'; however, global variable values, including array elements, can be externally supplied when the program is loaded, by a link-like step.

  • Labels and goto.

  • likely() / unlikely() hints for optimizing if statements.

  • Exception handling (described below).

tile-mpipe-cc does not support:

  • Subroutines. You can write only one function, which is invoked for each incoming packet. The hardware provides no writable memory, so without a stack, true subroutines are difficult to implement. You can, however, use gcc-style ({...}) statement expressions to create macros that are nearly as powerful as inline functions.

  • Division or remainder. Not supported by the hardware. Would be too slow to implement in software.

  • Fast multiplication. Not supported by hardware, so multiplies are implemented as a sequence of shift-and-add operations. If you multiply by a constant it can be relatively efficient, depending on the constant. Multiplying two arbitrary variables together can consume a large fraction of the packet processing budget.

  • Non-scalar datatypes other than global arrays. There are no structs, unions, etc., and no pointer variables.

  • Enums. There are no enums, although you can use #defines instead.

  • Static variables. The hardware runs multiple independent instances of each classifier program. The instances cannot communicate, and are stateless. There are therefore no static local variables. Global variables can be "static", which means the compiler assumes they are not modified by any link-time tools. All globals are implicitly "const".

  • typedef, const, and volatile are not supported.

  • floating point. Only 16-bit integer operations are supported.

  • Inline assembly is not supported. However most hardware resources are accessible through intrinsics. (See below.)

  • String or character constants. Except that the magic "printf" intrinsic does accept a string constant for the format string.

  • The "?:" operator.

Intrinsics:

[ Note: for details on all these intrinsics, please see the Tile Processor I/O Device Guide for the TILE-Gx Family of Processors. ]

  • uint16_t below_hwm(void);

    Returns 1 if the classifier header queue is below its high water mark, which generally means the classifiers are keeping up with incoming packets. Otherwise this returns 0. When this flag is 0 the system is falling behind and will start dropping packets that exceed their processing cycle count budget. See the CLASSIFIER_BUDGET special-purpose register.

  • uint16_t byteswap(uint16_t n);

    Exchanges the high and low bytes of n.

  • uint16_t checksum(uint16_t a, uint16_t b);

    Computes a 16-bit checksum of a and b.

  • uint16_t ctz(uint16_t n);

    "Count trailing zeroes". Returns the index of the lowest 1 bit, starting from 0, or 16 if n == 0.

  • uint16_t cycles_remaining(void);

    Returns the number of cycles remaining in the processing budget. If this hits zero, and below_hwm() returns false (meaning the system is falling behind), the packet will be forcibly dropped. As long as below_hwm() returns true, the system tolerates slow classifiers. See the CLASSIFIER_BUDGET special-purpose register.

  • uint16_t get2(void);

    Returns the next 16 bits from the packet header input buffer, advancing the buffer pointer. There is no get1().

  • uint8_t mem1(void);

    uint16_t mem2(void);

    Returns the next byte/word from table memory addressed by the tPtr (mempos) register. tPtr will be post-incremented by 1 or 2, appropriately. Note that if you mix mem1 / mem2 / mseek with C-style table memory access (e.g., "n = table[3]"), tile-mpipe-cc will assume you are smarter than it is and bypass some of its usual optimization attempts.

  • void mseek(&global_table_entry);

    Sets the tPtr (mempos) register to the location of the specified global table entry for subsequent mem1() or mem2() reads. Note that the "&" sign is optional -- mseek(&table[3]) and mseek(table[3]) are equivalent.

  • uint16_t peek2(void);

    Like get2, but does not advance the buffer pointer.

  • void put1(uint16_t n);

    Writes a single byte to the output buffer, advancing the output buffer pointer by 1.

  • void put2(uint16_t n);

    Writes two bytes to the output buffer, advancing the output buffer pointer by 2.

  • void iseek(uint16_t n);

    Seeks the input buffer pointer to the designated index, so that future get2 and peek2 calls return values from the given index.

  • void initial_iseek(uint16_t index);

    Sets the initial input buffer pointer register to the designated index, so that the first get2 or peek2 calls return values from the given location. Note that index must be a constant.

  • uint16_t itell(void);

    Returns the current position of the input buffer pointer.

  • void oseek(uint16_t n);

    Seeks the output buffer to the designated index, so that future put1 and put2 calls start writing at the given location. There is no 'otell'.

  • void initial_oseek(uint16_t index);

    Sets the initial output buffer pointer register to the designated index, so that the first put1 or put2 calls start writing at the given location. Note that index must be a constant.

  • uint16_t hash0_lo(void);

    uint16_t hash0_hi(void);

    uint16_t hash1_lo(void);

    uint16_t hash1_hi(void);

    Returns the low or high 16 bits of the hash0 or hash1 register.

  • void set_hash0_lo(uint16_t val);

    void set_hash0_hi(uint16_t val);

    void set_hash1_lo(uint16_t val);

    void set_hash1_hi(uint16_t val);

    Sets the low or high 16 bits of the hash0 or hash1 register.

  • void hash0_seed1(uint16_t n);

    void hash0_seed2(uint16_t n);

    void hash1_seed1(uint16_t n);

    void hash1_seed2(uint16_t n);

    Seeds the initial contents of the hash0 or hash1 register to the hash of the given 1 or 2 byte input value. The _seed1 functions ignore the high byte of their input.

  • void hash0_acc1(uint16_t n);

    void hash0_acc2(uint16_t n);

    void hash1_acc1(uint16_t n);

    void hash1_acc2(uint16_t n);

    Accumulates 1 or 2 bytes of data into the 32-bit hash0 or hash1 registers. The _acc1 functions ignore the high byte of their input.

  • uint16_t checksum_and_hash0_acc1(uint16_t a, uint16_t b);

    uint16_t checksum_and_hash0_acc2(uint16_t a, uint16_t b);

    uint16_t checksum_and_hash1_acc1(uint16_t a, uint16_t b);

    uint16_t checksum_and_hash1_acc2(uint16_t a, uint16_t b);

    Computes checksum and accumulates 1 or 2 bytes of data into the 32-bit hash0 or hash1 registers. The _acc1 functions ignore the high byte of their input.

  • uint16_t checksum_and_hash0_seed1(uint16_t a, uint16_t b);

    uint16_t checksum_and_hash0_seed2(uint16_t a, uint16_t b);

    uint16_t checksum_and_hash1_seed1(uint16_t a, uint16_t b);

    uint16_t checksum_and_hash1_seed2(uint16_t a, uint16_t b);

    Computes checksum of two words and CRC of one of the words into the accumulator.

  • uint16_t random(void);

    Produces a pseudorandom 16-bit value, based on a free-running LFSR. Not cryptographically secure.

  • void set_random(uint16_t n);

    Sets the low 16 bits of the CLASSIFIER_RAND register. The upper 6 bits are set to zero. Setting to zero stops the LFSR from changing.

  • void set_ctl(uint16_t n);

    Sets the CLASSIFIER_CTL register. Bit 0 (the low bit) freezes the classifier when set. Bit 1 will send an interrupt to the tile when written with a 1.

  • void set_pass(uint16_t n);

    Sets the CLASSIFIER_PASS register. Used in an implementation specific way for communication between the simulator and the classifier. It shares storage with the CLASSIFIER_FAIL and CLASSIFIER_DONE registers, which have no separate intrinsics. This data is also visible to the Tile software.

  • uint16_t packet_is_cut_through(void);

    uint16_t packet_has_error(void);

    uint16_t packet_has_mac_error(void);

    uint16_t packet_has_crc_error(void);

    uint16_t packet_is_truncated(void);

    uint16_t packet_channel(void);

    Provides access to assorted packet flags.

  • uint16_t packet_size(void);

    Returns the size of the packet in bytes.

  • printf(format, ...);

    In the simulator this acts like the normal C printf, but it does nothing on hardware. Under simulation this intrinsic is handled magically and is very efficient, using 0 or 1 cycles. Of course, computing the values being printed may consume more cycles.

    The format string must be a string literal. Because the other parameters are necessarily integers, only the various integer format strings like "%d" and "%04x" are allowed.

Exception handling:

tile-mpipe-cc allows creation of an optional exception handling routine. Control will be transferred to this routine when the classifier attempts to access data beyond the end of the iHdr buffer or beyond the L2 size of the packet. For example: 

    shared uint16_t foo;
    void classify()
    {
      // main classifier code ...
      foo = 42;
    }

    void ex()
    { 
      // exception handling code ...
      if (foo == 42)
        return; 
    }

Things to note:

  • The second routine defined (by position in the code) will be the exception handler. Its address will be automatically set in the appropriate register.

  • Variables defined by the "shared" keyword (before the main classifier routine) will be visible to both routines. The current value of a shared variable is preserved when control is transferred to the exception handler.

  • Shared variables each reserve an internal register. As such, they are a limited resource and should be used sparingly. Furthermore, their use is not optimized as well as normal variables; they are less efficient and should not be used for general computation.
