Name

tile-thd — Tilera HTTP Daemon

Synopsis

tile-thd [-h|--help] [-c|--config path] [-a|--address address] [-p|--port port] [-d|--docroot path] [--debug]

Description

The tile-thd HTTP daemon is a minimalist HTTP-protocol document server. It is intentionally designed to be small and simple to configure, and intended as a platform for small, ad-hoc debugging displays and other development tools. You can run tile-thd on a tile of the Tilera chip and use it to serve up information on the current state of the chip and of applications being run on it.

The tile-thd HTTP daemon ships with a sample configuration file and a small document tree containing a number of such displays, which can be used out of the box. However, one can also use the tile-thd daemon with a custom configuation and doc tree, to display diagnostic pages for your own purposes.

Important: The tile-thd HTTP daemon is not a full standards-compliant web server and should not be used for industrial-grade web server applications. Instead, use a standard web server such as Apache or Jetty for such purposes. The tile-thd HTTP daemon is meant for those times when a full-blown web server is overkill. It is intentionally designed to require a minimum of configuration and setup, and to take up less space and processing power than a full-service web server.

Command-Line Options

-h | --help

Optional. Displays usage information and list of command options, then exits.

-c|--config path

Optional (but recommended). Specifies the location of the tile-thd configuration file. Can be absolute or relative to the directory from which the HTTP daemon is invoked. Defaults to /etc/tile-thd/conf.

-a|--address address

Optional. Specifies hostname/address of network interface that the daemon should use. If specified, overrides the address, if any, specified in the configuration file.

-p|--port port

Optional. Specifies port number that the daemon should use. If specified, overrides the port number option specified in the configuration file.

-d|--docroot path

Optional. Specifies root of HTML document directory tree. Can be absolute or relative to the directory from which the HTTP daemon is invoked. If specified, overrides the docroot option specified in the configuration file.

--debug

Optional. Runs the daemon in debug mode. In this mode, the daemon does not run as a background process, does not fork when handling requests, and its debug log output is automatically directed to standard out. This mode is intended for use in debugging issues with configuration and page processing.

Usage

The tile-thd HTTP daemon is similar in configuration and usage to most modern web servers, so if you're familiar with how to run a web server such as Apache, Tomcat, or Jetty, you should have no issues with setting up tile-thd.

The tile-thd daemon is provided in both an x86 version and a TILE-native version. In normal use you'll run the TILE version on the Tilera processor. However, there might be cases where you'd want a small host-side server as well, so both are provided.

To run the TILE version of the tile-thd daemon, one can invoke it using tile-monitor as shown below: 

    tile-monitor \
      --root \
      --run -+- /usr/bin/tile-thd -p 8080 \
                -c /etc/tile-thd/conf \
                -d /etc/tile-thd/html -+- \
      --quit

This uploads the necessary MDE directories to the Tilera processor, starts the tile-thd daemon, pointing it to the default configuration and document tree, and overrides the port number to 8080. (If port 8080 is in use on your system, you can choose any other available port.)

Note that it's important to use --root, which uploads files to the Tilera chip, rather than --mount, which only mounts files temporarily. When tile-monitor exits, such temporary directory mounts would go away, and tile-thd would not be able to find its docroot directory.

Once the server is up and running, you can connect to it from your favorite web browser via its URL: 

    http://tilera_hostname:8080/

(Note that you can substitute the IP address of the Tilera host for the hostname, if hostname lookup is not available for your Tilera host.)

Configuration

The tile-thd configuration file specifies the configuration options for the daemon. The available options are listed below. (See the comments in the sample configuration file itself for additional detail.)

debug_log: pathname

Optional. Specifies pathname to which debug log information is written.

Can be a valid absolute path. The file is created if it does not exist. Can also be "stdout" and "stderr" for standard IO streams.

access_log: pathname

Optional. Specifies pathname to which access (i.e. incoming request) log information is written.

Must be a valid absolute path. The file is created if it does not exist.

address: hostname/IP-address

Optional. Specific interface (hostname or IP address) to which daemon should bind. This is only needed if the server host has multiple network address bindings.

port: port-number

Optional. Specifies the port number to which the daemon should bind. Defaults to port 80.

Note: the daemon must be run under an administrator username to bind to port 80. If you don't have this level of access, a common alternative is to use port 8080.

pid_file: pathname

Optional. File to which the daemon writes its process ID.

daemonize: { true | false }

Optional. (Debugging option.) Normally the daemon runs as a background daemon process. If this option is set false, the server runs as a foreground process.

Note: in most cases, it's simpler to just use the --debug command-line option instead.

fork: { true | false }

Optional. (Debugging option.) Normally the daemon forks a child process for each request, and hence requests can be handled in parallel. If this option is set false, the daemon handles all requests sequentially in the main daemon process.

Note: in most cases, it's simpler to just use the --debug command-line option instead.

docroot: directory-pathname

Required. Specifies the root of the directory tree from which HTTP requests can retrieve documents. Can be absolute or relative to the directory from which the HTTP daemon is invoked.

However, note that by using the path_mapping option, you can map URLs to paths outside of this directory tree.

index_file: index-filename.html

Optional. Default filename to look for when a URL requests a directory. If found, the index file is returned. Otherwise, a directory listing is provided. Defaults to index.html.

mime_type: extension, mime/type

Optional. Associates a file suffix (for example, html) with a MIME type that tells the browser how the file is meant to be displayed.

Multiple mime_type entries are allowed, but each file extension should be mapped to only one MIME type.

The default tile-thd configuration file includes a number of MIME type definitions for common file types like HTML files, text files, CSS files, graphic files, and video files.

path_mapping: /url/path => /host/path

Optional. Defines an association between a URL path and an absolute path in the file system. Allows a given location in the HTTP daemon's docroot tree to be mapped to an arbitrary location outside of the docroot tree.

Multiple path_mapping entries are allowed.

For each path mapping, the first pathname is the path that appears in the URL. For example, in the URL http://hostname:8080/this/is/the/path?arg1&arg2, the URL path is /this/is/the/path.

The second pathname in the path mapping is the absolute path of a file-system directory to which the above URL path should be mapped.

WARNING: be careful when using this option, since by design the HTTP daemon provides no security beyond that provided by ordinary Linux file/directory access permissions.

exe_path /pathname

Optional. Specifies a file (or directory of files) that are allowed to be executed, in a manner similar to CGI-BIN scripts.

Multiple exe_path entries are allowed.

When a URL requests a file that is either (a) declared in an exe_path entry, or (b) in a directory declared in an exe_path entry, and the file is an executable binary or script file, the file is executed and its output is displayed in the browser.

WARNING: be careful when using this option, since by design the HTTP daemon provides no security beyond that provided by ordinary Linux file/directory access permissions.

server_stylesheet /url/path

Optional. Specifies an HTML fragment to be included in the <head> section of any pages generated by the HTTP daemon (such as directory listings, 404 errors, etc.).

The pathname should be a URL path, that is, the path one would use in a URL to access the file from a browser.

server_header /url/path

Optional. Specifies an HTML fragment to be included before any other content in the <body> section of any pages generated by the HTTP daemon (such as directory listings, 404 errors, etc.).

The pathname should be a URL path, that is, the path one would use in a URL to access the file from a browser.

server_footer /url/path

Optional. Specifies an HTML fragment to be included after any other content in the <body> section of any pages generated by the HTTP daemon (such as directory listings, 404 errors, etc.).

The pathname should be a URL path, that is, the path one would use in a URL to access the file from a browser.

Server-Side Scripting

The tile-thd HTTP daemon supports a minimal form of server-side scripting via the exe_path configuration option. You can use this option to declare a file, or a directory of files, as enabled for execution. The file(s) themselves must also have the "execute" permission flag on, of course; the exe_path option just declares that the server is allowed to invoke them.

If a URL request references a file declared executable by exe_path, or a file in a directory declared executable by exe_path, instead of returning the file itself the server executes the file and returns its output to the requesting browser.

Executable files are invoked with the current directory set to the docroot directory, and run using whatever username the server was started with. (Hence, you should exercise caution when using this option, as there is no security aside from ordinary Linux file permissions.)

The output of the executable file must be in CGI-BIN format. This means at a minimum it must begin with a "Content-Type:" declaration giving the MIME type of the subsequent content, followed by a blank line, which indicates the end of the CGI-BIN headers. (The blank line is required; don't forget to add it!)

As an example, here's the how you'd generate an appropriate header using a shell script: 

test.sh:

#!/bin/sh
# CGI-BIN Headers (must end with blank line)
echo 'Content-Type: text/plain'
echo ''
...

If you have PHP installed on your system (and in particular if the installation includes the pgp-cgi command), you can use PHP as your scripting language as shown below: 

test.pgp:

#!/usr/bin/php-cgi -n
<html><head>
<title>PHP Example</title>
</head><body>
...
<?php phpinfo() ?>
...
</body></html>

You could also use a compiled C/C++ program, so long as it outputs an appropriate CGI-BIN header: 

test.c:

#include <stdio.h>

int main(int argc, char** argv)
{
  // CGI-BIN Headers (must end with blank line)
  printf("Content-Type: text/plain\n");
  printf("\n");
  ...
}

If the requesting URL includes query parameters (for example, 

    http://hostname:8080/path/to/script_name?param1=a&param2=b

then these are passed as-is to the script as arguments, e.g.: 

    script_name param1=a param2=b

Note: when using forms with exe_path scripts, remember to use "GET" requests, which are passed as command-line arguments, rather than "POST" requests, which currently are not supported.

For example: 

<form action="formhandlerscript.php" method="get">
...
</form>

Browser Compatibility

The web server is very minimalistic, but should support any HTTP/1.1 compliant browser.

The example HTML pages are written using the "strict" HTML DOCTYPE, and have been tested on the following browsers (minimum versions): 

  Firefox            2.x (Windows) 3.x (Linux)
  Internet Explorer  6.x (Windows)
  Chrome             16.0.912.77 (Windows)

The charting pages use the HTML 5 Chart component. This is supported natively by Firefox 2.x or better.

In Internet Explorer, the Google "Explorer Canvas" emulator is used, so you may notice that charts take more time to update when viewing in IE.

In general, browser protocol options that are not implemented, such as persistent connections, are simply ignored, but any decent browser should tolerate this.

Debugging

The tile-thd HTTP daemon is intended to be small and simple to upload and configure, but there's always some networking/configuration issues, etc. to be worked out, so if you can't connect to the server or an error page is displayed in your browser, be patient and recheck both your configuration and the tile-monitor command line used to upload and run the server.

The issue could be as simple as a misspecified port number, or you might be attempting to access the web server over a Tilera network interface that's not currently configured for networking (or perhaps not even physically connected to the network). Also, try using the IP address instead of the Tilera hostname in the URL, just in case hostname mapping is not set up.

If the server fails to start up for some reason, such as being unable to connect to the specified port, it should display a message to this effect.

If the server does start normally (i.e. it displays the "Server listening..." message), try running the server in --debug mode, which displays the debug log output to standard output.

You can also configure the server to generate access and debug log files, and look at these to help diagnose the problem (for example, if it's something that only happens once in a while).

To enable logging, in the configuration file uncomment the following two options, and make sure each one points to a valid pathname on the TILE processor: 

  access_log: /tmp/tile-thd-access.log

  debug_log: /tmp/tile-thd-debug.log

The access log contains typical information about incoming requests and the server's response to them. If you're not sure whether you're connecting to the server, check this file.

The debug log contains internal logging information, including any errors or unexpected conditions. If the server isn't displaying pages properly, check here to see how it's handling incoming connections, and to what server-side pathnames it is mapping the requests.

If all else fails and you still can't connect to the server, consult your local networking expert, because it's possible there may be a problem with how your hardware is connected and/or networking is configured.

Running Without the MDE

The tile-thd HTTP daemon does not depend on any Tilera-specific libraries, so it is possible to run the daemon itself outside of a Tilera MDE installation. (Having, of course, copied the daemon's binary file, configuration file, and document tree from an MDE.)

The tile-thd HTTP daemon does currently depend on a few shared libraries: 

  /usr/lib/libstdc++.so
  /usr/lib/libstdc++.so.6
  /lib/libm.so.6
  /lib/libm-2.12.so
  /usr/lib/libstdc++.so.6.0.13
