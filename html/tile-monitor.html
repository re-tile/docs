Name

tile-monitor — Support command line interaction with a Tile Processor from the host

Synopsis

tile-monitor [option...] [command...] [-- exe [arg...]]

Description

tile-monitor is a command line program which is very useful when developing applications for the Tile Processor. It allows you to interact in various ways with a Tile Processor, including booting the Tile Processor, transferring files or accessing the host filesystem via FUSE, and running, debugging, and profiling applications, without ever actually logging into that Tile Processor. Some simple examples can be found at the end of this document.

Note that tile-monitor is available both when running on a host system, and also when running natively on a Tile Processor. However, when running on a Tile Processor, the set of available options is different. When running natively, options related to PCIe cards, or to the simulator, are unavailable; this includes --config, --create-image, --functional, --image, --image-file, --sim-arg, --sim-args, and --sim-prefix. When running on a host system, the --self option is unavailable. Otherwise, the two commands have identical options and behavior.

Basic Operation

tile-monitor processes all of its command line options, then handles any requested booting and connects to the shepherd process running on the Tile Processor, then processes any command line commands, and finally enters an interactive mode in which more commands can be processed.

In this document, ${TILERA_ROOT} refers to the root of the Tilera MDE install directory. For example, ${TILERA_ROOT}/bin/tile-monitor is the actual tile-monitor executable on a host system.

Related Programs

Many of the interesting things done by tile-monitor are actually done by other processes. Booting requires a bootrom file, which is often created by tile-mkboot. Rebooting a networked Tile Processor is done by tile-reboot. A simulated Tile Processor is actually simulated by tile-sim. On the Tile Processor, a special shepherd process does most of the actual work involved in file transfers and running and debugging applications, and the oprofile and perf utilities do most of the actual work involved in hardware profiling.

Use tile-gcc or tile-g++ to compile executables for use with the Tile Processor.

The tile-eclipse IDE uses tile-monitor, as well as other tools, like tile-gcc and tile-gdb, to support its graphical interface.

Booting (and Connecting)

Before you can interact with a Tile Processor, it must be booted. Then, tile-monitor can connect to an appropriate shepherd process on the Tile Processor.

All simulated platforms, and most hardware platforms, can be booted by tile-monitor using a dynamically generated bootrom. Some hardware platforms require special installation of an appropriate bootrom, which tile-monitor can help create.

tile-monitor uses tile-mkboot to actually create bootrom files, passing it the default hypervisor config file, the default hypervisor binary files directory, and the default vmlinux file.

Use --hvc to specify the hypervisor config file. The default is vmlinux.hvc, or vmlinux-sim.hvc if you are using the simulator. These default config files can be found in the directory ${TILERA_ROOT}/tile/etc/hvc on the host, or /etc/hvc natively.

Use --hv-bin-dir to specify the hypervisor binary files directory. The default is ${TILERA_ROOT}/tile/boot/ on the host, or /boot/ natively.

Use --vmlinux to specify the vmlinux file. The default is ${TILERA_ROOT}/tile/boot/vmlinux on the host, or /boot/vmlinux natively.

Use --initramfs to specify the initramfs.cpio.gz file. The default is ${TILERA_ROOT}/tile/boot/initramfs.cpio.gz on the host, or /boot/initramfs.cpio.gz natively. Note that by default no initramfs.cpio.gz file is included if you use the --rootfs option.

Use --boot-dir dir both to imply --hv-bin-dir dir, and also to set the default directory that will be used for the classifier, vmlinux, and initramfs.cpio.gz files.

Use --bootrom-file to specify a specific (pre-generated) bootrom file.

Various other options will be passed through to tile-mkboot, including --hvd, --hvi, and --hvx.

To boot and connect to a networked Tile Processor over an IP connection, use --dev hostname. If the networked Tile Processor was previously booted with a different bootrom file, you must specify --reflash to allow tile-monitor to reflash the SROM on the device to allow rebooting into the new bootrom file. If the Tile Processor was already previously booted, use --resume.

When dynamically creating a bootrom for a networked Tile Processor, a set of --hvd options will be automatically supplied. These definitions will include CHIP_WIDTH (the width of the chip, in tiles), and CHIP_HEIGHT (the height of the chip, in tiles).

To boot and connect to a Tile Processor PCIe card installed in the current host, use --dev pciN. If the Tile Processor was already previously booted, use --resume.

When dynamically creating a bootrom for a Tile Processor PCIe card, a set of --hvd options based on the contents of the appropriate /dev/tilepciN/info file will be automatically supplied. These definitions will include CHIP_VERSION (1 for TILEPro, or 10 for TILEGx), CHIP_WIDTH (the width of the chip, in tiles), CHIP_HEIGHT (the height of the chip, in tiles), and BOARD_HAS_PLX (1 if the board has a PLX, else 0).

To boot and connect to a Tile Processor USB card installed in the current host, use --dev usbN. If the Tile Processor was already previously booted, use --resume.

When dynamically creating a bootrom for a Tile Processor USB card, a set of --hvd options based on the contents of the appropriate /dev/tileusbN/info file will be automatically supplied. These definitions will include CHIP_VERSION (1 for TILEPro, or 10 for TILEGx), CHIP_WIDTH (the width of the chip, in tiles), and CHIP_HEIGHT (the height of the chip, in tiles).

To use the simulator (aka tile-sim), use the --config or --image option. Two important tile-sim options, namely --functional and --config, are directly supported by tile-monitor, and most of the others can be specified using --sim-arg or --sim-args. For more information on legal simulator options, see the documentation for tile-sim.

A boot image, which captures the state of the simulator with a particular chip configuration, after booting Linux and starting the shepherd, can be used to save time when using the simulator. Use --image or --image-file to request the use of a particular boot image. Use --create-image to create a new boot image.

By default, tile-monitor includes a complete Linux filesystem in the bootrom, intended to be used as an initial "RAM" filesystem; this is known as the "initramfs" filesystem, and the files that make up that filesystem are stored in a "cpio" format file called an "initramfs.cpio" file. Typically the file is compressed with gzip and stored as an "initramfs.cpio.gz" file. The default file that tile-monitor uses is in /boot/initramfs.cpio.gz in the Tile filesystem hierarchy (under $TILERA_ROOT/tile in the cross-build environment).

However, tile-monitor can also create a bootrom that boots onto a root partition stored on disk. if you use the --rootfs option to specify a partition holding the root filesystem, tile-monitor will by default add a root= option to the kernel boot command line, telling the kernel what partition to mount as "root", and it will omit the initramfs file from the bootrom.

The partition argument can also be one of the two special forms "LABEL=mylabel" or "UUID=myuuid", where "mylabel" might be, for example, "root" or "home" or some other filesystem label, and "myuuid" might be a full filesystem UUID. In that case tile-monitor will add a TLR_ROOT= option to the kernel boot command line and include the initramfs file in the bootrom so that the initramfs "/init" script can process the TLR_ROOT directive.

If you specify an --initramfs option explicitly, it will always be included; by contrast, if you specify the --no-initramfs option explicitly, no initramfs.cpio.gz file will be included. The latter can be useful if you have made a custom Linux build with an embedded initramfs in the vmlinux file itself.

Transferring Files

To run an application, its executable (and any other executables and/or data files it needs) must be available on the target. This can be done by building a custom vmlinux which includes the files inside its file system. But it is usually simpler to use --upload, --upload-same, --upload-tile, or --upload-tile-libs to transfer files/directories from the host to the target.

After running an application, you can use --download or --download-same to transfer files/directories from the target back to the host.

Or, just use the various FUSE commands to remotely mount host directories via special mount points on the target.

FUSE

tile-monitor can remotely mount directories on the host filesystem onto the target filesystem using special mount points managed by the FUSE (file system in user space) library.

The --mount, --mount-same, and --mount-tile, commands perform the actual mounting operations.

The --here command is shorthand for a very common pair of operations, namely, mounting the current working directory as itself, and then entering that directory on the target, allowing many programs to act as if they were actually running on the host. Note that the --here command uses the actual current directory path (similar to "pwd -P") to choose the name for the remote mount.

The --root command is shorthand for uploading the /etc, /sbin, and /var directories, and mounting the /bin, /boot, /lib, and /usr directories. This is very useful when booting with a minimal initramfs, as it allows most programs to act as if a much more complete root filesystem was available.

Various operations on files accessed via FUSE mount points may not always work exactly as one might expect. Some important issues are mentioned below.

The actual file operations are done by tile-monitor, and so they have the same permissions as the current user. Since TILE processes usually run as root, this can cause some programs to break.

All file operations are done via host paths, not via persistent handles to host files, which can cause surprising results when open files are deleted or renamed, or when mmap() is used in certain ways.

All timestamp information comes from the host, which may cause clock drift issues, even if --rtc has been used to synchronize the clocks.

File locking (via fcntl()) is purely local, and does not affect host processes.

Running Applications

One of the most common uses of tile-monitor is to launch applications on a remote Tile Processor. An "application" is just a collection of processes, defined by a launched process, plus any "interesting" processes spawned by a process in that application. For more information about applications, see "tmc/task.h".

The --run command will launch an application, and wait for it to exit. The --launch command will launch an application without waiting for it to exit. The --wait command can then be used later to wait for all launched applications to exit.

To run an application and then exit, you can use the special -- exe [arg ...] syntax, which is just shorthand for --run -+- exe [arg ...] -+- --wait --quit.

When asked to launch an executable, if the executable path does not seem to exist, the shepherd will look for it in the directories listed in its PATH environment variable, much like a standard shell.

Every Linux process has an environment, which is a set of key/value string pairs accessible via getenv(). By default, a process inherits its environment from its parent, and for launched processes, this parent is the shepherd. The --env command can be used to modify the initial environment of the next launched process. Note that this does not modify the shepherd's own environment, and thus cannot be used to change the PATH used to find the requested executable.

Every Linux process (or thread) has an affinity, which is the set of tiles on which it is allowed to run. The affinity of a process can have significant effects on how the process runs, especially when dataplane tiles may be involved. The --tile command can be used to modify the set of tiles in the affinity of the next launched process.

Debugging

tile-monitor makes it easy to debug user-level processes, using tile-gdb.

First, you must request that a given process be debugged, using --attach, --debug, --debug-exe, --debug-exes, --debug-next, --debug-on-crash, and/or --debug-tile. For example: 

$ tile-monitor --dev gxpci0 --debug-on-crash --here -- myexe

Second, once a process is ready to be debugged, tile-monitor will print out some instructions. For example: 

[monitor] Process PID would like to be debugged.
Paste the following lines into another window:

${TILERA_ROOT}/bin/tile-gdb -q
set height -1
file myexe
set sysroot ${TILERA_ROOT}/tile
target remote host:port

Finally, in a separate shell window, paste the specified lines. For example: 

$ ${TILERA_ROOT}/bin/tile-gdb -q
(gdb) set height -1
(gdb) file myexe
(gdb) set sysroot ${TILERA_ROOT}/tile
(gdb) target remote host:port
Remote debugging using host:port

Note that tile-monitor will suggest commands that use local binary files from which it thinks that it can get useful debug information. This is derived from various sources, including the --mount (etc), --upload (etc), and --symbol-map commands. If tile-monitor is not able to identify an appropriate local binary file, it will use a remote file path, and will warn you to modify the suggested commands in an appropriate manner: 

[monitor] WARNING: Replace any remote binaries with local binaries!

You can then use any of the standard tile-gdb commands to debug your program, including bt, break, and continue. (Note that when one first connects with tile-gdb, the application has already been started, so one must continue, not run.) When you have finished debugging, you can use the detach and then quit commands to quit after allowing the process to continue running, or just quit to quit after killing the process.

Whole Application Debugging

Traditionally, a debugger is associated with a single process, and does not interact in any way with any other process, or any other debugger. Various events, such as user interrupts (and other signals) and hitting breakpoints, will cause the process to stop, allowing the user to debug the process. The continue command, or a related command such as finish, next, or step, will cause the process to resume execution.

When a process consists of multiple threads, and one of the threads stops, all the other threads in the process are frozen by the debugger. This prevents them from modifying shared state, and thus complicating debugging, while the user debugs the stopped thread. The user can easily switch to another thread, causing the stopped thread to become frozen, and the new thread to become stopped. The continue command will cause all of the threads to resume execution.

Tilera supports a similar model, known as whole-application debugging, for applications that consist of multiple processes. In this model, each debugger is still associated with a single process, and does not interact directly with any other process or any other debugger. But whenever any process stops, all the other processes in the same application are frozen by the shepherd process. This prevents them from modifying global state, and thus complicating debugging, while you debug the stopped process. Note that freezing a process has no effect on any associated debugger, which will not even realize that the process has been frozen. The continue command will cause all of the processes to resume execution.

It takes a small amount of time (known as skid) to freeze a process. During this time, before it is frozen, a process might actually stop for various reasons. Also, a frozen process will stop if you interrupt it. If there are multiple stopped processes in the same application, the continue command, rather than causing the process to continue, will freeze the process until all the processes are ready to resume execution.

An interesting side effect of the whole-application debugging model is that, when you are stepping through more than one process in the same application, each process might be frozen several times during each step. This will appear to you, and to the debuggers, as if each process was simply running slowly.

Bare Metal Debugging

Normal process-level debugging is restricted to examining the state in a user-level process. As an advanced feature, when running under the simulator, tile-monitor also supports "bare metal" debugging, which is basically low level debugging of an individual tile. This is especially useful when debugging operating system code.

Although bare metal debugging uses the familiar tile-gdb interface, it has some important limitations; in particular, since it makes no assumptions about page tables it can only examine memory which is already mapped in that tile's DTLB.

To use bare metal debugging you must first request it using either --bm-debug-attach-tile or --bm-debug-on-panic. When a tile is ready to be debugged, tile-monitor will prompt you to run tile-gdb by pasting some lines, in a manner similar to that described above.

Profiling

Profiling of applications is handled by (a) selecting a profiling utility and (b) using a number of standard commands to initialize, start, and stop profiling, and to collect and process the results for display.

The profiling tools currently supported are:

  • the OProfile open-source profiling utility
  • the Linux perf utility (a.k.a. "Perf Events")

The profiling commands are described in more detail below in the section Profiling Commands.

For more detail on the oprofile utility, see Profiling from the Command Line with OProfile.

For more detail on the Linux perf command, see Profiling applications with the Linux "perf" tool.

Interactive Mode

Interactive mode provides emacs-style editing and history recall using the libedit library. Some important keys include up/down to navigate the history, left/right and Ctrl+A/Ctrl+E to move the cursor, Ctrl+D and backspace to delete characters, tab to induce filename completion, and Ctrl+R and Ctrl+L to redraw.

When used on the command line, all commands have a leading -- prefix, just like all options. In interactive mode, this prefix is not needed (and is not allowed).

When used on the command line, all commands which take a variable number of arguments must wrap their arguments in a pair of matching delimiters, represented as -+-, but which technically can be any string starting with a dash that is unlikely to match any actual argument, including a single dash. In interactive mode, these delimiters are not needed (and are not allowed).

Use the --quit command, after the --wait command if needed, or better yet, the special -- syntax, to quit before entering interactive mode.

Peers

Each monitor can optionally be associated with one 'peer' monitor, via the --peer-connect and --peer-listen options.

If both monitors are using the simulator, this allows various simulated I/O devices to be hooked up to each other. For example, this can be used to simulate linux networking between the two simulations.

Having a 'peer' monitor also enables the use of the --peer-barrier command.

Basic Options

--dev name

Specify the hardware device (by name or label) to connect to, after first booting it.

--hvc file

Request the use of the given hypervisor configuration file.

This option is passed through to tile-mkboot.

--hvd var[=value]

Provide a variable definition to be used while processing hypervisor configuration files. If value is entirely missing (no equal sign, e.g. --hvd FOO), the value "1" is used. To assign an empty string as value, use --hvd FOO=.

This option is passed through to tile-mkboot.

--hvh file

Request the pre-processing of the given hypervisor configuration file. Unlike "--hvc", this does NOT replace the default HVC file.

This option is passed through to tile-mkboot, changing "--hvh" to "--hvc".

--hvi dir

Append a directory to the search path used to find hypervisor configuration files.

This option is passed through to tile-mkboot.

--hvx value

Append a single space plus value to the definition of the special XARGS variable, which starts out defined, but empty. This variable is used by some hypervisor configuration files, usually to extend the primary client args.

This option is passed through to tile-mkboot.

--net addr

Connect via direct network connection to a networked Tile Processor after first booting it.

The address addr may be a numeric address or a hostname.

Booting will be done using tile-reboot, which will use ssh with the given address, using a freshly created bootrom. Note that if the bootrom differs from the one currently installed on the remote host, you will need to include the --reflash option on the command line as well.

The bootrom is expected to run the shepherd with --net, which is done by default, to listen for network connections on port 963.

This option is no longer generally useful, as by default typical bootroms for networked Tile processors do not enable the shepherd --net option, which is required for the tile-monitor --net option to work. Instead, see the tile-monitor --ssh option.

--reflash

Allow updating the SROM on a networked Tile Processor with a new bootrom file image. Without this argument, it is only possible to reboot a networked Tile Processor with the same bootrom file that it used the previous time it booted.

--resume

Skip the normal booting step.

--rootfs block-device

Specify a boot argument to Linux of root=block-device, and suppress the default inclusion of an initramfs.cpio.gz file. For example, "--rootfs /dev/sda1" will cause a "root=/dev/sda1" Linux boot argument to be included, making Linux use /dev/sda1 as the root filesystem partition when booting. If the block-device starts with either "LABEL=" or "UUID=", then a "TLR_ROOT=" Linux boot argument will be included instead, and the initramfs.cpio.gz file will be included to handle it.

--self

Install a new boot image on the local Tile Processor, and then reboot it.

Booting will be done using tile-reboot with the --self option, using a freshly created bootrom.

--ssh addr

Connect via ssh to a networked Tile Processor after first booting it.

The address addr may be a numeric address or a hostname.

Booting will be done using tile-reboot, which will use ssh with the given address, using a freshly created bootrom. Note that if the bootrom differs from the one currently installed on the remote host, you will need to include the --reflash option on the command line as well.

tile-monitor will then connect to the remote host using ssh. If a password is required, it can be entered interactively (as is also true for the tile-reboot invocation). To avoid having to use a password when connecting to a networked Tile Processor booted from the default disk-based environment, you may use the default Tilera ssh key (see /root/.ssh/tilera_root_rsa.README for details).

Simulator Options

--config size

Passed through to the simulator to specify a chip size (which can be of the form "WxH", or a special string like "tile64").

--create-image path

Create a boot image for later use with --image-file.

--functional

Passed through to the simulator to induce "functional" mode.

--image name

Specify a boot image file by name, which denotes ${TILERA_ROOT}/lib/boot/name.image. Standard images are provided for common chip configurations, including 1x1, 2x2, 4x4, and tile64. A boot image file allows the simulator to start up as if booting had already taken place. Use the --create-image option to create new boot images. The --image option implies using the simulator.

--image-file path

Specify a boot image file by path. See the --image option for more information on boot image files. The --image-file option implies using the simulator.

--sim-arg arg

Specify an additional simulator argument, e.g. --sim-arg --grind-coherence.

--sim-args -+- arg ... -+-

Specify multiple additional simulator arguments, e.g. --sim-args -+- --profile-interval 1000 -+-.

Advanced Options

--batch-mode

Add final --wait and --quit commands.

--bme name[=path]

Specify a BME executable.

The argument to this option is passed through to tile-mkboot as a final positional argument, and is also used to identify where debugging information for the given BME executable can be found.

--boot-dir dir

Equivalent to specifying both --hv-bin-dir dir, and setting the default directory to use for the classifier, vmlinux, and initramfs.cpio.gz files.

--boot-only

Exit immediately, after booting.

--boot-timeout

Change how long (in seconds) the monitor will wait for the booted system to consume the bootstream. By default, this is 200 seconds, which is normally more than long enough. In cases where a longer wait is necessary, you may specify a longer time with this option, or completely disable the timeout by specifying a value of 0.

--bootrom-file path

Specify a bootrom file by path.

--classifier path

Use the specified classifier file, instead of ${TILERA_ROOT}/tile/boot/classifier (on the host) or /boot/classifier (natively).

--console

Redirect console output from the standard console device to stdout. If the console device is locked or can't be opened, tile-monitor will exit with an error.

The standard console device is normally encoded by the "serial=" line in the appropriate config file (/etc/tilenet/ADDR.conf, /etc/tilepciN.conf, or /etc/tileusbN.conf).

When using the simulator, the standard console device is a magic device.

--try-console

Try to redirect console output from the standard console device to stdout. If the console device is locked, tile-monitor will print a warning message and not attempt to open it. See --console for how the standard console device is encoded.

--console-device device

Like --console, but using the specified console device (e.g. /dev/ttyS0), instead of the standard console device.

--console-out path

Like --console, but redirect console output to the specified file, instead of to stdout.

--try-console-out path

Like --try-console, but redirect console output to the specified file, instead of to stdout.

--create-bootrom path

Create a bootrom file and exit. This file can be used later with --bootom-file, or can be used directly with sbim.

--create-hvc path

Create a preprocessed hypervisor config file and exit.

--gdb-port port

Assign gdb ports starting from the specified port.

--greet-timeout

Change how long (in seconds) the monitor will wait for the shepherd on the booted system to come up and greet it. By default, this is 200 seconds, which is long enough for normal boot scenarios. For scenarios with potentially longer, valid, boot delays (for example, mboot), you may wish to specify a longer time here.

--hv-bin-dir dir

Use the hypervisor binary files in the given directory, instead of the ones in ${TILERA_ROOT}/tile/boot (on the host) or /boot (natively).

--initramfs path

Add "initramfs.cpio.gz=path" to the tile-mkboot arguments, thus adding an optional user cpio archive to the boot image.

--local-gdb

Create local gdb sessions as needed. Use the "#g" idiom described by --help-stdin to interact with these local sessions.

--mkboot-args -+- arg ... -+-

Specify extra arguments to be passed through to tile-mkboot. Only use this option when no other option seems appropriate.

--no-classifier

Do not use any classfier file.

--no-dev

Do not use any device whatsoever. Only meaningful with --create-bootom and --create-hvc.

--no-hv-bin-dir

Do not use any hypervisor binary files.

--no-initramfs

Do not use any initramfs.cpio.gz file.

--no-readline

Disable use of "readline" in interactive mode.

--no-rtc

Do not specify --rtc during PCI or USB boot.

--no-vmlinux

Do not use any vmlinux file.

--peer-connect what

Connect to a 'peer' monitor. The what can be a standard port number, optionally preceded by a host and a colon, to establish an internet connection, or a special string starting with '@', to establish a unix connection.

--peer-listen what

Listen for a 'peer' monitor. The what can be a standard port number, to listen for an internet connection, or a special string starting with '@', to listen for a unix connection.

--readline

Force use of "readline" in interactive mode.

--shepherd-port port

Connect to the shepherd on the given port (defaults to port 963).

--sim-prefix prefix

Specify a prefix to the simulator command-line. For example, by specifying --sim-prefix nice, tile-monitor would run nice tile-sim ....

--tag-console

Tag lines of stdout/stderr from remote processes with a special prefix indicating which remote process produced them.

--verbose

Increase verbosity.

--vmlinux path

Use the specified vmlinux file, instead of ${TILERA_ROOT}/tile/boot/vmlinux (on the host) or /boot/vmlinux (natively).

Basic Commands

--cd dir

Specify the working directory of the shepherd, which will affect commands which take file paths, such as --upload and --run, and which will be used as the initial working directory for new processes.

--debug-on-crash

Debug all processes which experience a signal which normally induces a core dump, such as SIGSEGV, SIGBUS, SIGILL, or SIGABRT.

--env key=value

Modify the (initially empty) environment used by all future launched processes. The syntax is the same as putenv(), and in particular, you can use --env key to remove an entry from the environment.

--launch -+- [executable] [argument...] -+-

Launch the specified executable with the specified arguments. The resulting process, and all of its interesting descendants, are known as an "application".

--quit

Tell the shepherd to kill all processes which are still running, and then exit. Use --wait first, if needed, to instead wait for processes to exit on their own.

--run -+- [executable] [argument...] -+-

Like --launch, but also wait for the application to exit. In interactive mode, you can hit Ctrl+C to stop waiting.

--tile tile

Modify the set of tiles to be used as the affinity of (only) the next launched process. This can be used to restrict the process to a specific set of tiles, or to allow the use of tiles which would otherwise be avoided, such as dataplane tiles.

The tile argument can be a cpu number (e.g. 63), a range of cpu numbers (e.g. 1-63), a coordinate (e.g. 7,0), a rectangle (e.g. 8x8), a rectangle with a top left coordinate (e.g. 4x4@2,2), dataplane (for all the dataplane tiles), standard (for all the non-dataplane tiles), or all (for all the tiles).

All of the tile arguments above can be inverted by adding a leading caret (e.g. --tile ^0), in which case, the specified tiles will be removed, instead of added.

Multiple --tile options can be combined to describe a more complex set of tiles (e.g. --tile 0 --tile 4-7). Alternatively, multiple tile arguments can be combined using a plus sign (e.g. --tile 0+4-7). These two syntaxes can even be combined (e.g. --tile 8x2+2x8 --tile ^dataplane).

There are often multiple ways to specify the same set of tiles. For example, --tile 8x2@0,0+2x8@0,0+8x2@0,6+2x8@6,0 could be also written as --tile 8x8@0,0+^4x4@2,2.

If there are no --tile commands, or the argument to the very first --tile command starts with a caret, then an initial --tile standard will be assumed. A very common example is --tile ^0, which means all non-dataplane tiles, except for tile zero.

When the launch is actually attempted, any offline tiles, including dedicated tiles, will be implicitly dropped from the set of requested tiles. If the resulting set is empty, the shepherd will complain, and the launch will fail.

--tiles -+- [tile...] -+-

Specify multiple tile arguments, as if multiple --tile options had been used.

--wait

Wait until all processes have exited. This is normally only needed after using --launch to launch applications asynchronously, because --run always waits for its launched application to exit. See also --quit.

File Commands

--delete path

Delete the specified path on the target.

--download tile-path host-path

Copy a file (or directory) from the target to the host.

--download-same path

Shorthand for --download path path. Note that path may be a relative path.

--here

Shorthand for --mount CWD CWD --cd CWD, where CWD is the current working directory (on the host).

--mkdir dir

Create the specified directory on the target, including all ancestors, if necessary.

--mount host-dir tile-dir

Request that the given host directory be mounted remotely, using FUSE, onto the given target directory.

--mount-same dir

Shorthand for --mount dir dir. Note that dir may be a relative path.

--mount-tile dir

Shorthand for --mount ${TILERA_ROOT}/tiledir dir. Note that dir must be an absolute path.

--rename path newpath

Rename a directory or file or symlink.

--rmdir dir

Remove an empty directory.

--root

Basically shorthand for --mount-tile /bin --mount-tile /boot --mount-tile /lib --mount-tile /usr --upload-tile /var --upload-tile /etc --upload-tile /sbin.

Note that /etc is uploaded because, for example, /etc/mtab is modifiable. Note that /var is uploaded because, for example, /var/lib/oprofile/ is modifiable. Note that /sbin gets uploaded because mounting it is (currently) buggy, but its contents are restored when the monitor exits.

--unlink path

Delete a file (or symlink).

--upload host-path tile-path

Copy a file (or directory) from the host to the target.

--upload-same path

Shorthand for --upload path path. Note that path may be a relative path.

--upload-tile path

Shorthand for --upload ${TILERA_ROOT}/tilepath path. Note that path must be an absolute path.

For example, --upload-tile /usr/lib would upload the entire standard /usr/lib directory to the target.

--upload-tile-libs name

Shorthand for --upload-tile path for every appropriate /lib/libname.* and /usr/lib/libname.*.

For example, --upload-tile-libs stdc++ would upload the C++ support libraries, and --upload-tile-libs ncurses would upload the ncurses support libraries.

Advanced Commands

--attach pid

Debug the process with process identifier pid.

--bm-debug-attach-tile x,y

Initiate bare metal debugging on the tile with the given coordinates.

This is completely unrelated to the --debug-tile option.

--bm-debug-on-panic

Request bare metal debugging for any tile that encounters an operating system panic.

Code running on a tile-sim tile can write the value SIM_CONTROL_PANIC to the SPR_SIM_CONTROL special-purpose register, indicating that an operating system "panic" has occured.

If the --bm-debug-on-panic option was supplied, tile-sim reacts to SIM_CONTROL_PANIC by initiating bare metal debugging for that tile.

This option is related to the --debug-on-crash option, in that both start a debugging session if something bad happens. However both the type of debugging and the circumstances under which it is initiated differ.

--debug -+- executable [argument...] -+-

Like --launch, but also debug the resulting process (and all of its interesting descendants).

--debug-exe executable

Add another executable to the list of debuggable executables. When a process is created, or calls "exec", and its executable is in this list, then it will be debugged.

--debug-exes -+- executables -+-

Set the list of debuggable executables. For more information, see --debug-exe.

--debug-next

Debug the next launched process, and all of its interesting descendants.

--debug-tile tile

Modify the set of debuggable tiles for the next launched process, and all of its decendants. When a process calls "exec" while running on such a tile, it will be debugged.

Currently, for historical reasons, when a process is launched, if the first tile in its affinity set is in the set of debuggable tiles, then it will be debugged, regardless of what tile it is actually running on at the time.

The semantics of --debug-tile, and the syntax for the tile arguments, are similar to that used by the --tile command, except that if there are no --debug-tile commands, then no tiles will be debuggable, and if the argument to the very first --debug-tile command starts with a caret, then an initial --debug-tile all will be assumed.

--dump

Dump out some shepherd state.

--forbid-tiles tiles

Forbid the use of the given tiles.

--forward tile-port host-port

Forward all connections to a given target port to a given host port, in a manner similar to "ssh -R". This can be useful if you have not actually given the target network connectivity. For instance, one can run X11 clients on the target against the X11 server on the host by specifying something like --forward 6000 6000.

Compare to --tunnel.

--gdb-command command

Add an extra command to be run by all new 'gdb' sessions. This can be used, for example, to add the command "set solib-search-path PATH", which can help the debugger locate the local copy of shared libraries which have been uploaded.

--huge-pages pages

Request that the given number(s) of huge pages be made available for allocation. Note that pages can be a single number, to be spread across all memory controllers, in which case the number is written to the special file "/proc/sys/vm/nr_hugepages"; or, it can be a list of numbers (separated by commas or spaces), one for each memory controller, in which case the numbers are parsed as integers and used to set the appropriate default hugepage count for each memory controller node on the system in order; for example, with 16MB default huge pages a list of four numbers would set /sys/devices/system/node/node[0-3]/hugepages/hugepages-16384/nr_hugepages.

--local-system command-string

Perform the requested command(s) locally, that is, on the monitor host, using system(). The command string can be a single shell command, or multiple commands separated by ';'.

--peer-barrier

Synchronize with the peer monitor, by waiting for it to do a peer-barrier.

--ping

Ping the shepherd, to make sure it is still alive.

--rtc

Set the clock on the Tile Processor to match the clock on the host. Note that this is done automatically whenever a PCI card is booted.

--shepherd-tiles tiles

Set the shepherd affinity to the given tiles.

--shepherd-verbosity verbosity

Specify the shepherd verbosity level (higher numbers yield more output). The default level is zero.

--sim -+- [command] [argument...] -+-

Send a command to the simulator. Send the "help" command to see a list of all the simulator commands that can be sent.

--spread-threads

Spread out the threads of the nextlaunched process. As each thread is created, including the first one, it will be locked to the next available cpu in the initial affinity set.

This command can yield performance improvements for certain executables which do not attempt to manage their own affinity in any interesting way. This command should NOT be used with any process which expects to use its initial affinity in any interesting way, since each thread will have an affinity containing exactly one tile before it can run any such code.

--symbol-map host-file tile-file

Specify a symbol mapping from a host file to a tile file, as if the file had just been uploaded via --upload.

--system command-string

Perform the requested command(s) on the target (i.e. the shepherd). The command string can be a single shell command, or multiple commands separated by ';'.

--tunnel host-port tile-port

Tunnel all connections to a given host port to a given target port, in a manner similar to "ssh -L". This can be useful if you have not actually given the target network connectivity. For instance, one can telnet to the target by running telnetd on the target, then specifying --tunnel 10023 23, to allow running telnet localhost 10023 (for instance) on the host.

Compare to --forward.

--verbosity verbosity

Specify the monitor verbosity level (higher numbers yield more output). The default level is zero. See also the documentation for the --verbose option.

Profiling Commands

The available profiling commands, in the order they are typically used, are:

  • --profile-tool

  • --profile-init

  • --profile-kernel

  • --profile-vmlinux

  • --profile-events

  • --profile-flags

  • --profile-reset

  • --profile-start-daemon

  • --profile-start

  • --profile-stop

  • --profile-dump

  • --profile-capture

  • --profile-analyze

  • --profile-shutdown

The profiling commands are described in more detail below.

Note that the usage and/or the effect of these commands generally depends on the currently selected profiling tool. (See --profile-tool) The usage/effect differences between the tools, where they exist, are noted below for each command.

For more detail on the oprofile tool, see Profiling from the Command Line with OProfile.

For more detail on the Linux perf tool, see Profiling applications with the Linux "perf" tool.

--profile-analyze dir

Analyzes the profiling data previously captured via --profile-capture, so that it can be viewed in the tile-eclipse IDE's Profile View.

The usage of this command is the same for all profiling tools. (See --profile-tool).

The effect of this command depends on the currently selected profiling tool.

  • oprofile: Runs the tile-op2xml utility on the capture directory to process the profiling data.
  • perf: Runs the tile-perf2xml utility on the capture directory to process the profiling data.

--profile-capture dir

Downloads the current profiling data into the specified host-side directory, so that it can then be analyzed by the --profile-analyze command.

The usage/effect of this command is the same for all profiling tools. (See --profile-tool)

--profile-dump

Performs any necessary flushing of profiling state to ensure it will be available to --profile-capture. (Note: This command is implicitly invoked by --profile-stop.)

The usage of this command is the same for all profiling tools. (See --profile-tool)

The effect of this command depends on the currently selected profiling tool.

  • oprofile: Ensures any extra debugging information has been captured. (Generally not needed unless you have enabled the OProfile utility's debugging features.)
  • perf: Currently has no effect.

--profile-events flags

Specifies event-related flags to be used by --profile-start.

This command is optional. If not specified, a default set of commonly-used profiling events is automatically specified.

The usage of this command depends on the currently selected profiling tool. (See --profile-tool)

  • oprofile: Event flags are specified in the form: "--event=EVENT_NAME:interval ...", that is, you specify an interval for each event.
  • perf: Event flags are specified in the form: "-c interval -e EVENT1,EVENT2,...", that is, you specify one interval for all events.

--profile-flags flags

Specifies any additional tool-specific flags to be used by --profile-start.

The usage of this command depends on the currently selected profiling tool. (See --profile-tool)

  • oprofile: The --callgraph=n flag can be used to specify the callgraph depth.
  • perf: Currently there are no additional flags to specify.

--profile-init

Performs any initialization (uploading files, initializing flags state, etc.) required by the currently selected profiling tool.

NOTE: Whenever you change profiling tools with --profile-tool, you must reinvoke --profile-init to initialize the newly-selected profiling tool.

The usage/effect of this command is the same for all profiling tools. (See --profile-tool)

--profile-kernel

Enables profiling of kernel (OS) code, and performs any additional initialization (file uploading, etc.) required to profile using the default kernel (/boot/vmlinux).

This command is optional; if specified, it should follow --profile-init.

(See the equivalent command --profile-vmlinux to enable profiling with a kernel binary other than the default.)

The usage of this command is the same for all profiling tools. (See --profile-tool).

The effect of this command depends on the currently selected profiling tool.

  • oprofile: Enables kernel profiling as described above.
  • perf: Kernel profiling is always enabled; this command merely selects the default kernel for profiling.

--profile-reset

Clears any previously collected profiling data from the TILE file system. Note that this command does NOT delete any host-side data you may have previously captured with --profile-capture.

The usage of this command is the same for all profiling tools. (See --profile-tool).

The effect of this command depends on the currently selected profiling tool.

  • oprofile: Deletes the /var/lib/oprofile/samples/current sample tree.
  • perf: Deletes the /tmp/perf.data capture file.

--profile-shutdown

Shuts down any supporting process (profiling daemon, etc.) to reduce system load.

If profiling is currently running (see --profile-start), this command also does an implicit --profile-stop to cleanly stop profiling.

The usage of this command is the same for all profiling tools. (See --profile-tool).

The effect of this command depends on the currently selected profiling tool.

  • oprofile: Shuts down the profiling daemon process.
  • perf: Has no effect; there is no profiling daemon process.

--profile-start

Starts profile data collection.

This command implicitly uses the state, if any, previously specified via the commands --profile-kernel, --profile-vmlinux, --profile-events, and --profile-flags for the current profiling run.

If the current profiling tool has a daemon process, etc. and it has not yet been started, this command does an implicit --profile-start-daemon to start this daemon process. (You can use the --profile-start-daemon command to separately start the daemon process, so that --profile-start executes faster.)

NOTE: For the oprofile tool, profiling flags, etc. are actually used when the profiling daemon is restarted. You must fully stop and restart the profiling daemon in order to change profiling flags; if you do not, you will be profiling with the same settings as the previous profiling run.

Otherwise, the usage/effect of this command is the same for all profiling tools (see --profile-tool).

--profile-start-daemon

Pre-starts any supporting process (profiling daemon), etc. so that the --profile-start command can enable profiling more quickly.

The usage of this command is the same for all profiling tools. (See --profile-tool).

The effect of this command depends on the currently selected profiling tool.

  • oprofile: Starts the profiling daemon process.
  • perf: Has no effect; there is no daemon process.

--profile-stop

Stops profile data collection, and implicitly invokes --profile-dump.

The usage/effect of this command is the same for all profiling tools. (See --profile-tool).

--profile-tool { oprofile | perf }

Selects the profiling tool to be used by the rest of the profiling commands.

The profiling tools currently supported are:

  • oprofile -- the OProfile open-source profiling utility
  • perf -- the perf utility (a.k.a. "Perf Events")

For more detail on the oprofile tool, see Profiling from the Command Line with OProfile.

For more detail on the Linux perf tool, see Profiling applications with the Linux "perf" tool.

Currently, the default tool is oprofile. However, for forward-compatibility it is recommended that in new code you specify the --profile-tool option to explicitly define the tool you are using, since the flags, etc. accepted by the two tools do differ significantly.

You should use --profile-tool at the very start of a profiling session, not change tools it until you have captured and analyzed the data you've collected from that session.

Whenever you change profiling tools, you must then invoke --profile-init to initialize the newly-selected profiling tool.

If you find you didn't really want to switch tools, you can re-invoke --profile-tool to switch back. However, once you've invoked --profile-init, any uncaptured state collected by the previous tool is effectively lost.)

The behavior of the other profiling commands is determined by the currently selected tool. In general the purpose of each command remains the same. However, some commands may have no effect for the current profiling tool, and for others the argument(s) to use will differ. These differences are noted for each profiling command.

--profile-vmlinux vmlinux

Enables profiling of kernel (OS) code, and performs any additional initialization (file uploading, etc.) required to profile using the specified vmlinux kernel.

This command is optional; if specified, it should follow --profile-init. Also, this command is otherwise equivalent to --profile-kernel, so at most one of the two should be specified.

The usage of this command is the same for all profiling tools. (See --profile-tool).

The effect of this command depends on the currently selected profiling tool.

  • oprofile: Enables kernel profiling as described above.
  • perf: Kernel profiling is always enabled; this command merely selects the specified vmlinux kernel for profiling.

--generate-monitor-xml path

Generates a monitor.xml metadata file to the specified directory path. For use with tile-op2xml and tile-perf2xml to generate profile.xml files by hand.

Examples of tile-monitor Usage

To boot the default device, run a standard command, and then exit, try:

tile-monitor -- ls /usr/bin

To boot the default device, upload an executable, run it, and then exit, try:

tile-monitor --resume --upload myapp /bin/myexe -- /bin/myexe

To connect to the default device, mount and cd into the current working directory, run an application, and then exit, try:

tile-monitor --resume --here -- myapp myarg

To boot the default device, mount and cd into the current working directory, launch, in parallel, a server application on tile 3,3, and a client application on tiles 1,1 and 5,5, wait for them to exit, and then exit, try:

tile-monitor --here --tile 3,3 --launch - myserver - --tile 1,1 --tile 5,5 --launch - myclient - --wait --quit

To boot the simulator, using a standard image file, and then enter interactive mode, try:

tile-monitor --image tile64
