Name

tile-sim — Simulate a Tile-Gx Processor

Synopsis

tile-sim [option...]

Description

Normally, tile-sim is not run directly by users, and instead, is invoked indirectly by tile-monitor, which itself may be invoked indirectly by tile-eclipse.

The various options documented below may be passed to the simulator via the --sim-arg option of tile-monitor.

tile-sim simulates a Tile Processor, which is a single chip that contains an array of tiles and associated I/O devices. Each tile contains a processing engine, a cache engine, and a switch engine that connects a tile to its neighbors. I/O devices include memory controllers and external communications interfaces. tile-sim is a system simulator that provides both timing-accurate and functional modes, as well as support for detailed debugging and performance analysis.

By default, tile-sim does timing-accurate simulation. The --functional option makes tile-sim run faster, at the expense of timing accuracy.

Performance Analysis

Various tools are provided by tile-sim to aid performance analysis. Profiling produces aggregate statistics over a program run. Tracing produces detailed cycle-by-cycle results.

Use the --xml-profile options to produce detailed profiling data that can be viewed in the tile-eclipse Profile View.

Trace files contain very detailed information, down to the machine cycle level. These files can be viewed as text, or by using tile-trace-view which provides a graphical user interface (GUI) to display trace data.

See the Profiling Options and Tracing Options for more detail.

To focus profiling and/or tracing on specific program segments, use the Profiler Control API and Simulator Control API, which enabled controlling profiling and tracing options respectively under program control. See the Optimization Guide for more information.

For instance, to turn tracing on and off around an interesting section of code, use: 

#include <arch/sim.h>

sim_set_tracing(SIM_TRACE_CYCLES | SIM_TRACE_DISASM);
        :
// Interesting code.
        :
sim_set_tracing(SIM_TRACE_NONE);

Likewise, to turn profiling on and off around an interesting section of code, use: 

#include <arch/sim.h>

sim_profiler_enable();
        :
// Interesting code.
        :
sim_profiler_disable();

Command-Line Options

Basic Options

-h [--help]

Print (on the standard output) the help message, containing the list of command-line options understood by tile-sim and other usage information.

--config { XxY | chip }

Set the tile configuration to be simulated, where X is the number of tiles horizontally (the number of columns), and Y is the number of tiles vertically (the number of rows).

As a shorthand, a chip name can also be specified, which will configure the tile array, memory controllers, and other values (such as core clock speed) appropriately. Accepted chip arguments are:

gx8016

A TILE-Gx8016 processor composed of a 4x4 tile array and I/O shims per the product definition.

gx8036 or gx36

A TILE-Gx8036 processor composed of a 6x6 tile array and I/O shims per the product definition.

gx8064 or tile64

A TILE-Gx8064 processor composed of an 8x8 tile array and I/O shims per the product definition.

gx8100

A TILE-Gx8100 processor composed of a 10x10 tile array and I/O shims per the product definition.

--cycle-limit cycles

Run the simulator for no more than the specified number of cycles.

--force-functional

Run the entire simulation in functional mode, ignoring programmatic attempts to disable functional mode.

--functional

Reduce simulation time at the cost of lower timing accuracy. (Default is off, so tile-sim is timing-accurate by default.)

--options-file file

The file parameter specifies the name of a configuration file holding addition tile-sim options. A standard configuration file tile-sim.cfg is always read if it exists. The priority of options is first options given on the command line, then an options file specified by --options-file, then options in tile-sim.cfg.

Configuration files contain lines of the form 'name = value' and can contain comments, e.g.: 

# Options file for AES
xml-profile = aes.xml
output-file = aes
trace-cycles = true
trace-stall-info = true
trace-disasm = true
--output-file prefix

Prepend the specified prefix to names of output files. Without --output-file, all output will appear on the standard output. With --output-file, output will be divided into files per type, per tile, using the specified prefix.

--clock-speed speed

Sets the core clock speed in cycles per second. Accepts an optional modifier G (giga), M (mega), or K (kilo). The default is 1.2 GHz on TILE-Gx.

--symbols file

Adds the symbols from the specified file to those searched during symbol lookups, for instance for stack backtraces.

--print-final-cycle-count

Print out the final cycle count for the simulated and host machines (always to stdout).

-v [--version]

Print (on the standard output) the version of the invoked tile-sim.

--backtrace-limit limit

Stop the simulation if the simulator's backtrace history ever contains more than limit stack frames. A very large number of stack frames is often an indication of a serious bug in the simulated program. If this option is not specified, or if limit is set to 0, the simulator will not limit the number of backtrace stack frames.

Profiling Options

--xml-profile filename

Emit performance statistics, including function profiles, into the specified XML file, which can be viewed in the Profile View of tile-eclipse.

--xml-stats filename

Emit summary performance statistics into the specified XML file, which can be viewed in the Profile View of tile-eclipse.

--profile-interval interval

Used with one of the profiling options, produce the specified statistics every interval cycles.

--profile-functions

Profile code at per-function granularity.

--profile-lines

Profile code at source line granularity.

--profile-pc

Profile code at per-PC granularity.

--profile-detailed-stalls

Collect microarchitectural stall reasons. Using this option will consume more memory and produce larger profile files.

--profile-idle

Profile even tiles not running any process.

--diff-friendly-stats

Produce a profile file that may be more easily compared with previously generated profiles. For example, this disables printing of host cycles.

--max-profile-pl level

Only perform detailed profiling at the maximum privilege level specified or below. 0 = USER, 1 = USER + SUPERVISOR, 2 = USER + SUPERVISOR + HYPERVISOR.

--profile-disable-at-startup

Disable profiling at startup but allow it to be enabled programmatically later.

--profile-boot

Profile boot code and dedicated tiles.

Tracing Options

--trace-tile-view prefix

Produce trace files for tile-trace-view, the Tilera Trace Viewer. Trace file names are prepended with the specified prefix. The --trace-tile-view option is a shorthand for specifying several of the individual tracing options together with --output-file.

--trace-cycles

Add the current cycle to the information produced by the other tracing options. This means prepending each output line with the tile processor cycle number pertaining to that line.

--trace-l2

Produce a trace of L2 Cache behavior, meaning the actions of the level-two unified cache on each tile. The output is printed to files or standard output as specified by --output-file.

--trace-register-writes

Produce a trace of register updates for the processor on each tile. The output is printed to files or standard output as specified by --output-file.

--trace-disasm

Produce a trace of instruction execution for the processor on each tile, including the instruction address and disassembled instruction bundle. The output is printed to files or standard output as specified by --output-file.

--trace-lines

Interleave the current function name, source file path, and source line number into the trace output. The information is emitted into the file every time the function, path, or line number changes. The output is printed to files or standard output as specified by --output-file.

--trace-stall-info

Produce a trace of stall information for the processor on each tile, including stalls on instruction fetch, data access, network reads and writes, etc. The output is printed to files or standard output as specified by --output-file.

--trace-memory-controller

Produce a trace of memory controller behavior. The output is printed to files or standard output as specified by --output-file.

--trace-syscalls

Produce a trace of system calls on each tile. The output is printed to files or standard output as specified by --output-file.

--trace-start-cycle cycle

Start tracing at a specified cycle.

Checkpointing Options

--checkpoint-file file

Specify the name of the checkpoint file to create when a checkpoint is created, e.g. when "sim_checkpoint" is called. Note that tile-monitor's --create-image option corresponds to this option.

--load-from-checkpoint file

Start simulation from specified checkpoint file. Note that not all options can be set or changed after a checkpoint is loaded. For example, new shims cannot be created when a checkpoint is loaded; shims must be created when the checkpoint is created. Note that tile-monitor's --image option corresponds to this option.

--checkpoint-interval cycles

Create a checkpoint every given number of cycles. The default checkpoint file is named "tmp". You may specify a checkpoint file with the --checkpoint-file option.

Advanced Options

--shim X:Y:direction:name:args

Specify an I/O shim at coordinates X,Y, cardinal direction (n, e, w, or s), with name, and optional arguments. Each element in the specification must be separated by colons or commas.

The shim name must consist of a prefix that is a valid shim type (e.g. mpipe is a valid shim type; and mpipe, mpipe0, mpipe1 are all valid shim names).

The shim name specified by this option is used only by the simulator (to determine the shim type, annotate simulator output, etc.); it is not related to the shim names in the hypervisor device table or its associated APIs.

Note that the direction can be prepended by r, which reverses the usual clock-wise direction of connecting multiple ports to the mesh interconnect.

Valid shim types are:

crypto

A MiCA crypto accelerator. Most users will use the crypto shim created by --config, but it is possible to add a crypto shim to custom configurations via --shim X:Y:dir:crypto.

memory

A DDR3 memory controller. For instance, the --config gx36 option implies two memory controllers as follows:

  • --shim 0:1:rw:memory:ports=2:banks=32:ranks=4
  • --shim 5:1:e:memory:ports=2:banks=32:ranks=4

A memory controller's ports are not on adjacent tiles; they are implicitly spaced apart by one tile. tile-sim specifies a single memory controller at 0:0:w by default.

mpipe

An mPIPE controller. Most users will use the mPIPE created by --config, but it is possible to add an mPIPE shim to custom configurations via --shim X:Y:dir:mpipe.

Once an mPIPE shim is created, the --extra-shim-args flag can be used to configure the virtual ethernet devices attached to the shim. This allows the user to control the link bitrate, source and destination pcap files, and traffic shaping parameters. For example, passing --extra-shim-args "mpipe0:mac=0:bps=10G:input=input0.pcap:output=output0.pcap" would configure link 0 on mpipe0 to run at 10 gbps and take its input and output from the indicated .pcap files. Appending ":mac=1:bps=1G:input=input1.pcap:link_enable=1" would configure link 1 on mpipe 0 to input from a pcap file, discard egress packets, and start generating packets immediately. By default, links are disabled at startup and need to be enabled either by passing the link_enable=1 flag or by calling sim_enable_mpipe_links() from within the simulated application.

mPIPE shim arguments can include the following:

bt-on-drop

Print backtraces when a packet is dropped.

panic-on-drop

Print backtraces and exit when a packet is dropped.

clock-speed=speed

Simulated clock speed of mPIPE shim.

classifier-clock-speed=speed

Simulated clock speed of classifier engines.

trace_mask=0xfff

Specifies a bitmask that enables tracing for various mPIPE components.

mac=index

Specifies the MAC to be configured by MAC-specific options.

link_enable

Brings up the current MAC at the start of the simulation, rather than waiting for a call to sim_enable_mpipe_links().

bps=bitrate

Specifies the maximum link bitrate for the current MAC.

input=file

Specifies an input pcap file for the current MAC. This option cannot be used when creating a checkpoint; it can be used when loading a checkpoint or when not using a checkpoint.

output=file

Specifies an output pcap file for the current MAC.

peer=shim/mac

Connects the current MAC to a particular shim and MAC on a peered simulator. For example, ":peer=mpipe0/3" would connect to mpipe0, MAC 3 on the remote simulator. See the tile-monitor documentation for more information.

repeat=count

Repeat the current MAC's input pcap file 'count' times.

stall

Controls the controller behavior when the input FIFO fills, i.e. consumers are not keeping up with input. With the stall option specified, the controller will stall and not consume further input, as opposed to the default behavior of dropping input packets, which is what the hardware does.

silent-stall

Like the stall option, except that no report is printed.

classifier_program=name

Specifies a classifier program name.

trio

A TRIO controller. Most users will use the TRIO created by --config, but it is possible to add a TRIO shim to custom configurations via --shim X:Y:dir:trio.

Once a TRIO shim is created, the --extra-shim-args flag can be used to configure the three StreamIO links attached to the shim. This allows the user to control the link bitrate and simulator peering parameters. For example, passing --extra-shim-args "trio0:mac=0:bps=32G:peer=trio0/1" would configure link 0 on trio0 to run at 32 gbps (x8 gen2) and to connect to link 1 on a peered simulator.

TRIO shim arguments can include the following:

clock-speed=speed

Simulated clock speed of TRIO shim.

trace_mask=0xfff

Specifies a bitmask that enables tracing for various TRIO components.

mac=index

Specifies the MAC to be configured by MAC-specific options.

bps=bitrate

Specifies the maximum link bitrate for the current MAC.

peer=shim/mac

Connects the current MAC to a particular shim and MAC on a peered simulator. For example, ":peer=trio0/2" would connect to trio0, MAC 2 on the remote simulator. See the tile-monitor documentation for more information.

--uart-output file

Output file for the UART. Use "-" for stdout. If ommitted, the output for the Rshim will be used.

--uart-input file

Input file for the UART. Use "-" for stdin. If ommitted, reads from the uart will block.

--srom-boot

Boot the chip from the contents of the SPI Flash ROM.

--ram-mb-per-controller megabytes

Amount of RAM per controller, in megabytes.

--grind-coherence

Enables additional checks to help find shared memory cache coherence bugs in the simulated program(s). (This option slows down simulation considerably.)

--grind-warnings-not-fatal

With --grind-coherence, don't exit immediately on the first grind-coherence warning.

--raw

Use tile-sim directly, in "raw" mode.

--watch address

Specify a (one-byte) virtual address to watch for writes.

--rwatch address

Specify a (one-byte) virtual address to watch for reads.

--pa-watch address

Specify a (one-byte) physical address to watch for writes.

--rwatch address

Specify a (one-byte) physical address to watch for reads.

Caveats

tile-sim models all major features of the TILE Processor, including the tile core processor, cache sub-system, on-chip interconnect, and I/O devices including memory. But not every single feature of all devices is modeled. Missing or partially implemented features include:

  • Tile core CPU performance counters are not modeled.

  • Memory shim performance counters are not modeled.

  • Various I/O device configuration and status registers not used by the standard MDE runtime stack (Hypervisor, Linux, and user-space libraries) are not modeled.

  • The RShim UART is partially modeled: output goes to stdout.

  • Certain features of the processor front-end are not modeled, notably the hardware instruction prefetch engine, branch prediction engine, and L1I way prediction.

  • The mPIPE model does not model several less-frequently used features, including per-bucket sequence numbers, timestamps, and IOTLB faults.

  • The MiCA model includes only a limited set of the cryptography and compression algorithms that are available in the full TILE-Gx processor. It does support memory copy (DMA) operations from one application memory address to another. (See the MDE README for more details.)
